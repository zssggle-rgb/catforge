# 第三方竞争分析智能体 MVP 需求说明

## 一、文档定位

本文基于《基于可观测市场数据的第三方竞争分析智能体讨论稿》和 2026 年 6 月 8 日项目数据交互安全事项沟通会整理，用于定义 6 月内可演示、可汇报的 MVP 范围。

MVP 的核心目标是：基于奥维可观测市场数据，先跑通第三方竞争分析智能体的最小闭环，为海信定价智能体提供外部市场证据视角下的竞品、卖点价值、价值战场和品牌溢价输入。

## 二、产品定位

第三方竞争分析智能体不是通用问答工具，也不是海信内部定价模型本身。它是海信定价智能体体系中的第三方市场分析能力模块。

其职责是：

- 从市场可观测数据出发，反向分析某个型号的用户任务、目标客群和价值战场；
- 在价格、渠道、参数、卖点、销量/榜单、用户反馈和目标人群维度上识别客观竞品；
- 输出可解释、可追溯、可复核的第三方市场分析结论；
- 为后续定价智能体提供竞品、品牌溢价、卖点价值和用户价值输入。

## 三、MVP 建设原则

| 原则 | 要求 |
| --- | --- |
| 方法先行 | MVP 先验证分析方法、呈现形态和业务认可度，不以全量系统集成为目标 |
| 可观测数据优先 | 首版以奥维可采集的外部市场数据为基础，海信内部数据接入不作为前置条件 |
| 结论先行 | 面向领导和业务用户先给结论，再展开证据、表格和推导过程 |
| 用户语言 | 从参数语言转为用户任务、目标客群、使用场景和价值战场语言 |
| 可解释 | 每个竞品、卖点价值和溢价判断都要给出依据、置信度和限制条件 |
| 权限分级 | 定价、价格支撑、溢价等敏感指标按权限展示，不默认开放给所有用户 |
| 可迭代 | 语义库、规则、样本池和数据源允许后续按品类和海信数据接入情况迭代 |

## 四、MVP 范围

### 4.1 首版品类范围

讨论稿中的首期范围是电视、厨电、洗衣机三个品类。会议中结合交付节奏进一步收敛为：

| 品类 | MVP 建议 | 原因 |
| --- | --- | --- |
| 彩电 | 优先样板 | 数据和分析口径较成熟，适合证明方法正确 |
| 洗衣机 | 优先候选 | 更贴近当前业务汇报需求，可作为第二个样板 |
| 厨卫 | 暂缓 | SKU、套装、烟灶热等组合复杂，第一版风险较高 |
| 空调 | 暂不纳入 | 本次会议未作为优先范围讨论 |

MVP 建议先做 1-2 个品类。若时间不足，至少做出一个品类和一两个型号的完整样例。

### 4.2 数据时间范围与样本池

| 项目 | 建议口径 |
| --- | --- |
| 时间窗口 | 2026 年 1-5 月电商数据 |
| 渠道 | 京东、天猫等主要电商渠道 |
| 样本选择 | 销量靠前、评论充足、具备代表性的重点 SKU |
| 样本规模 | 可用 Top SKU 或覆盖目标品类 80%-85% 销量的 SKU 池 |
| 型号样例 | 由海信项目组确认 1-2 个待分析型号，录音中型号识别不稳定，不在本文固化 |

## 五、目标用户与权限

| 用户角色 | 主要使用内容 | 权限建议 |
| --- | --- | --- |
| 经营/业务领导 | 竞品结论、品牌溢价趋势、AI 解释、关键建议 | 看结论和趋势，不展示敏感明细 |
| 产品/品类人员 | 产品百科、竞品关系、价值战场、卖点价值 | 可看较完整分析过程 |
| 营销人员 | 用户任务、目标客群、用户感知卖点、品牌溢价解释 | 不展示定价敏感字段 |
| 电商/渠道人员 | 竞品、价格压力、渠道表现、用户评价 | 可看渠道和竞品证据 |
| 定价/算法人员 | 价格、溢价、价值支撑、数据口径、模型输入 | 需较高权限 |
| 数据/IT/安全人员 | 数据来源、接口、部署、审计、脱敏 | 管理和审计权限 |

## 六、核心用户流程

1. 用户进入第三方竞争分析智能体 MVP 页面。
2. 用户输入或选择一个产品型号，选择品类、渠道和时间窗口。
3. 系统展示产品百科，说明该型号的基本信息、价格/渠道/销量排序、参数、卖点和用户评价。
4. 系统反推用户任务、目标客群和价值战场。
5. 系统给出 2-3 个核心竞品结论，区分主战场、次战场或机会战场。
6. 用户展开分析过程，查看竞品选择理由、参数/卖点/用户任务/价格/渠道/证据对比。
7. 用户查看品牌溢价指数和趋势，理解当前品牌相对行业和竞品的价格位置。
8. 系统输出 AI 解释、限制条件和后续建议。

## 七、功能需求

### FR-1 型号输入与查询

| 项目 | 说明 |
| --- | --- |
| 输入 | 品类、型号、渠道、时间窗口，可选价格带 |
| 输出 | 查询到的产品基础信息和可分析状态 |
| 要求 | 支持搜索框式输入；型号不存在或数据不足时提示原因 |

### FR-2 产品百科

产品百科用于形成单个型号的市场可观测画像。

应展示：

- 品牌、型号、系列、品类、价格带；
- 主要渠道和榜单/销量排序信号；
- 核心参数，如尺寸、规格、容量、能效、刷新率等；
- 页面宣传卖点和原始卖点；
- 标准卖点映射；
- 用户评价中的满意点、吐槽点和高频感知要素；
- 数据来源、采集时间和置信度。

### FR-3 品类语义字典与卖点标准化

系统需要按品类建设语义资产，包括：

- 标准参数；
- 标准卖点；
- 用户任务；
- 目标客群；
- 使用场景；
- 价值维度；
- 价值战场；
- 竞品识别规则。

MVP 可先覆盖样板品类，不要求一次性完成全品类语义库。

### FR-4 用户任务与目标客群反推

系统应基于参数、卖点、价格、渠道、榜单/销量信号和用户评论，反推：

- 用户为什么买；
- 谁可能买；
- 主目标客群、次目标客群和弱覆盖客群；
- 证据来源和置信度。

示例：护眼卖点、儿童观看评论和相关参数可反推出“有孩家庭护眼用户”和“儿童安心观看任务”。

### FR-5 价值战场定义

系统应识别型号进入的价值战场，并区分主战场、次战场、机会战场或弱战场。

彩电可参考的价值战场包括：

- 家庭观影改善；
- 游戏电视；
- 大屏换新；
- 高端画质；
- 家庭护眼。

每个价值战场应展示：

- 进入理由；
- 战场中的主要竞品；
- 战场规模或样本池；
- 本品优势和短板；
- 证据和置信度。

### FR-6 竞品识别与排序

竞品识别应结论先行，先给出 2-3 个核心竞品，再展开原因。

识别维度包括：

| 维度 | 说明 |
| --- | --- |
| 销量/榜单表现 | 优先选择市场上卖得好的真实竞争对象 |
| 价格与渠道 | 同价格带、同渠道或存在价格压力 |
| 参数配置 | 核心规格、配置和功能相近 |
| 标准卖点 | 核心卖点组合相似或形成替代 |
| 用户任务 | 服务同一用户任务 |
| 目标客群 | 目标用户存在重叠 |
| 价值战场 | 在同一价值战场中竞争 |
| 品牌位置 | 品牌势能、溢价或市场地位影响竞争关系 |

输出应包括：

- 第一竞品、第二竞品、第三竞品；
- 主战场竞品和次战场竞品；
- 直接竞品、替代竞品、标杆竞品、潜在竞品；
- 竞品选择理由；
- 证据链和限制条件。

### FR-7 卖点价值分层

系统应将卖点按价值强弱分层：

| 类型 | 含义 | MVP 处理 |
| --- | --- | --- |
| 基础门槛卖点 | 缺失会影响竞争，但不一定形成溢价 | 必须识别 |
| 竞争绩效卖点 | 做得越好越提升产品竞争力 | 必须识别 |
| 溢价倾向卖点 | 可能支撑更高价格或更好销售表现 | MVP 可探索，不作为强承诺 |
| 弱感知卖点 | 用户反馈少或难转译为用户价值 | 必须提示 |

会议确认“卖点价值指数/溢价卖点”有价值，但样本量和主观推断会影响准确性。MVP 可先输出分层和解释，不强制输出精确价格贡献。

### FR-8 品牌溢价指数

品牌溢价指数需要从“单张表”升级为可解释模块。

基础展示：

- 各品牌溢价指数；
- 周期趋势图；
- 与竞品品牌对比；
- 与自身历史对比；
- AI 解释和建议。

会议讨论的计算逻辑：

- 按品类核心规格分组；
- 彩电可按尺寸，冰箱可按门体/容积，空调可按匹数；
- 比较行业均价与品牌均价；
- 按销量权重汇总形成品牌溢价指数；
- 分析指数上升或下降原因，如高端尺寸占比变化、低价型号占比过高、核心规格价格位置变化等。

### FR-9 AI 解释与证据链

每个关键结论都应配套：

- 结论摘要；
- 分析口径；
- 数据范围；
- 证据来源；
- 置信度；
- 限制条件；
- 后续建议。

系统不得把无证据的模型推断包装成确定事实。

### FR-10 权限与敏感信息控制

MVP 前台应至少标识哪些字段属于敏感字段，并支持隐藏或分级展示。

建议规则：

- 产品百科、公开参数、公开卖点、公开评论可较广泛展示；
- 用户任务、目标客群、价值战场可对业务角色展示；
- 定价支撑、溢价指数细节、价格弹性、内部经营判断应限制权限；
- 对外汇报或导出材料需明确数据来源和授权边界。

## 八、数据需求

| 数据项 | MVP 必需 | 建议来源 | 说明 |
| --- | --- | --- | --- |
| 品类、品牌、型号、系列 | 是 | 奥维可观测市场数据 | SKU 主数据基础 |
| 产品参数 | 是 | 电商页面、品牌公开资料、奥维采集 | 用于功能归因和参数对比 |
| 宣传内容/页面卖点 | 是 | 电商详情页、品牌公开资料 | 用于原始卖点抽取 |
| 标准卖点映射 | 是 | 奥维语义库 + 专家校准 | MVP 先覆盖样板品类 |
| 价格数据 | 是 | 奥维采集、电商公开数据 | 会议明确价格是必要项 |
| 销量/榜单/销售排序 | 是 | 电商榜单、销量排序、区间值 | 可用作辅助证据，不要求精确日销量 |
| 用户评价 | 是 | 电商评论、问答等 | 用于满意点、吐槽点、用户任务验证 |
| 搜索词榜单 | 建议 | 电商搜索结果/销量排序 | 用于竞品池构建 |
| 商智热销商品榜 | 待确认 | 海信商智账号或本地部署 | 涉及账号和安全审批 |
| 海信店铺经营明细 | 待确认 | 海信京东自营旗舰店后台 | 涉及安全和权限 |
| 海信内部销售/成本/毛利 | 否 | 海信内部系统 | 不作为 MVP 前置条件 |
| 日级销量/价格弹性 | 否 | 后续评估 | 噪声高、误差影响大，暂不作为验收项 |

## 九、数据安全与交互要求

MVP 应遵守以下安全边界：

- 海信内部数据默认不出域；
- 如需海信内部数据，优先在海信环境中部署或调用；
- 奥维外部市场数据可按审批方式输入海信侧环境；
- 表格上传、接口调用、云端部署、本地服务器运行都需安全确认；
- 所有数据交互应记录来源、用途、权限、时间和负责人；
- 涉及海信内部敏感数据的输出必须脱敏或隐藏；
- MVP 不因内部数据审批未完成而阻塞外部数据样板开发。

## 十、非 MVP 范围

以下内容不纳入首版 MVP 硬性交付：

- 海信内部全量数据打通；
- 成本、毛利、库存、投放、转化链路接入；
- 日级/小时级销量价格弹性模型；
- 全三品类完整上线；
- 自动生成最终定价决策；
- 将分析结果写回海信核心业务系统；
- 对所有品类建立完整语义库；
- 精确计算每个卖点的价格贡献值。

## 十一、MVP 验收标准

| 验收项 | 标准 |
| --- | --- |
| 页面样板 | 至少包含产品百科、竞品分析、品牌溢价指数三类展示 |
| 型号样例 | 至少完成 1 个型号端到端分析，最好完成 2 个型号 |
| 结论先行 | 竞品分析先输出 2-3 个核心竞品和主/副战场结论 |
| 推导过程 | 能解释竞品为什么被选中，包含参数、卖点、价格、渠道、用户任务、价值战场证据 |
| 数据来源 | 每个关键结论说明来源、时间窗口和限制条件 |
| 品牌溢价 | 展示指数、趋势、竞品对比和 AI 解释 |
| 用户语言 | 输出不只停留在参数对比，必须转译为用户任务和用户价值 |
| 权限控制 | 敏感字段有标识，前台可做隐藏或分级展示 |
| 风险提示 | 对样本不足、销量噪声、评论不足、低置信度结论做提示 |
| 汇报可用 | 能支撑 6 月下旬前期汇报或联合汇报 |
