# M12C 卖点价值量化与贡献归因 SOP 需求

## 0. 定位

M12C 是新版分析链路中的“卖点价值量化与贡献归因”模块。

它承接已经形成的事实层和语义市场图谱：

- M03B SKU 参数事实画像。
- M04C SKU 卖点事实画像。
- M05C SKU 评论事实画像。
- M07 SKU 市场画像。
- M09C 用户任务画像。
- M10C 目标客群画像。
- M11C 价值战场画像。
- M11D 语义市场图谱与销量分配。

M12C 要回答的问题是：

1. 一个 SKU 卖得好，哪些卖点在可比市场中更可能贡献了价格溢价、销量提升或销额提升。
2. 某个标准卖点在不同尺寸价格池、价值战场、用户任务、目标客群中是否有可观测价值。
3. 一个卖点是溢价卖点、基础门槛、厂家主张、机会缺口，还是拖后腿因素。
4. 与竞品相比，本品有哪些卖点在市场上形成了价值优势，哪些卖点只是参数相似但没有用户或市场验证。
5. 如果补强某个卖点，可能进入哪些更大的价值战场或目标客群空间。
6. 同池高价 SKU 比本品多了哪些有价值卖点、卖点组合或更强用户感知。
7. 一个卖点是独立支撑溢价，还是只在卖点组合中参与解释高端价值。
8. 一个卖点明明存在，为什么没有形成溢价：是同池都具备、评论感知不足、竞品表达更强，还是样本不足。

M12C 的输出是“可观测市场贡献估计”，不是严格因果归因。它必须区分两层含义：

1. **可比池卖点组差异**：有某卖点且证据成立的一组 SKU，相比同池对照组，在价格、周均销量、周均销额上的可观测差异。
2. **SKU 超额表现解释份额**：本品相对同池基准的超额价格、销量、销额表现，按卖点证据权重做的解释性分摊。

自然语言回答必须使用“可比池差异、解释份额、估计、支撑、相关、可观测贡献”等口径，不能说成“该卖点必然导致多少销量”，也不能把“某卖点分摊到 X 元”写成“这个卖点单独让 SKU 多卖 X 元”。

## 1. 业务目标

M12C 必须支撑小奥家电市场分析专家回答以下业务问题：

1. 某个 SKU 的哪些卖点是真正的溢价卖点。
2. 某个 SKU 卖得好，主要由哪些卖点支撑。
3. 某个卖点在一个价值战场里值多少钱，带来多少周均销量优势。
4. 某个卖点在不同价格带、尺寸段中是否都有价值，还是只在某些池子里有效。
5. 同样有 MiniLED、高刷、护眼、智能语音等卖点，为什么一个 SKU 卖得更好。
6. 某个竞品靠哪些卖点拦截了本品的成交理由。
7. 本品要扩大市场份额，应补强哪些卖点或重新表达哪些卖点。
8. 某个卖点是进入战场的基础门槛，还是能支撑溢价。
9. 某个卖点虽然厂家宣传了，但评论或市场没有验证，应该如何降级解释。
10. 某个卖点被用户负向评价时，对主战场、主客群或主任务有什么拖累。
11. 如果本品在同池中没有价格优势，哪些高价竞品卖点或卖点组合值得作为上探机会。
12. 如果某卖点池级价格差异为负，判断它是销量卖点、基础门槛、组合价值的一部分，还是用户感知不足。

## 2. 模块边界

### 2.1 必须解决

1. 以标准卖点为对象，生成品类级、市场池级、语义维度级的卖点价值指标。
2. 在同尺寸、同价格带、同价值战场、同用户任务、同目标客群的可比池内估算卖点价值。
3. 区分价格溢价、销量提升、销额贡献、评论验证、参数支撑和语义支撑。
4. 对 SKU 的卖点贡献做解释性归因，输出每个卖点的贡献角色、可比池差异和 SKU 超额解释份额。
5. 对样本不足、缺少对照组、价格促销干扰、评论不足等情况输出降级和复核原因。
6. 输出可供 CLI、Skill、飞书报告和智能体回答使用的稳定结果。
7. 支持按品类扩展，首版优先 TV。
8. 识别同池高价竞品拦截卖点、价格上探机会卖点、组合型增值卖点和用户感知不足卖点。

### 2.2 不解决

| 不做事项 | 原因 |
| --- | --- |
| 不直接读取原始四张表做业务判断 | 必须消费 M00-M11D 已事实化和画像化结果 |
| 不把缺失当成否定 | 缺失是未知，除非上游 taxonomy 明确该字段缺失可判否 |
| 不做严格因果推断 | 当前缺曝光、点击、转化、促销资源、线下渠道等数据 |
| 不跨不可比市场池直接归因 | 卖点价值必须在可比池内估算 |
| 不把累计销量作为强弱比较唯一依据 | 强弱比较优先用周均销量、重叠周或窗口标准化指标 |
| 不把服务履约卖点纳入产品溢价卖点 | 安装、物流、售后等只作为服务语境或风险 |
| 不覆盖小样本硬判 | 样本不足必须输出 `sample_insufficient` |
| 不把一个 SKU 在多个视角中的贡献相加 | 价值战场、任务、客群是不同解释视角，不能相互求和 |

## 4. 分析对象和口径

### 4.1 默认分析 SKU 集

M12C 默认 SKU 集为：

```text
claim_value_ready
= current M04C 卖点事实画像
  ∩ current M07 市场画像
  ∩ current M11D 语义市场图谱覆盖 SKU
```

如果需要强用户验证，使用：

```text
claim_value_ready_with_comment
= claim_value_ready
  ∩ current M05C 评论事实画像
```

默认报告回答中应优先使用 `claim_value_ready_with_comment`。对没有评论事实但有参数和卖点事实的 SKU，可以输出“参数和厂家表达成立，用户验证不足”。

### 4.2 市场窗口

首版使用 M07/M11D 当前的 `full_observed_window`，但所有结果必须带：

- `market_window`
- `window_start_week`
- `window_end_week`
- `active_week_count`
- `batch_id`
- `analysis_population`

后续可以支持 `recent_12w` 和自定义周范围。

### 4.3 可比池优先级

卖点价值不能全局比较，必须在可比池内估算。

可比池构建优先级：

1. 同五档尺寸 `size_tier`。
2. 同尺寸内价格带 `price_band_in_size_tier`，必要时允许相邻价格带扩展。
3. 同主价值战场或同辅价值战场。
4. 同主用户任务或高重合用户任务。
5. 同主目标客群或高重合目标客群。
6. 同渠道或主平台可比。
7. 样本不足时按固定规则逐级放宽，并记录放宽原因。

不可比情况必须降级：

- 跨尺寸档直接比较。
- 低价刚需池和高端画质池直接比较。
- 有评论 SKU 与无评论 SKU 混合后直接判断用户验证。
- 服务类卖点和产品硬件卖点混合比较。

## 5. 卖点价值分类

M12C 输出的卖点价值分类必须是业务可解释的，不只输出分数。

| 分类 | 含义 | 典型判断 |
| --- | --- | --- |
| `premium_driver_estimated` | 强溢价卖点 | 支撑主战场/主任务/主客群，有参数和评论验证，且可比池卖点价格差异为正 |
| `sales_driver_estimated` | 强销量卖点 | 价格差异不显著或为负，但在可比池中对应更高周均销量、销额或份额 |
| `basic_threshold` | 基础门槛卖点 | 同池大多数 SKU 都具备，缺失会拖累，具备不一定加价 |
| `value_bundle_claim` | 组合型增值卖点 | 单卖点不一定独立溢价，但与一组高价值卖点同时出现时支撑高端价值 |
| `weak_user_perception_claim` | 用户感知不足卖点 | 参数或厂家卖点存在，但本品评论验证弱、负向明显，或弱于高价竞品 |
| `high_price_competitor_intercept` | 高价竞品拦截卖点 | 同池高价且有成交能力的竞品具备，本品缺失、表达弱或评论验证弱 |
| `price_up_opportunity` | 价格上探机会卖点 | 同池高价 SKU 反复具备并有市场价值，本品补强后可能提升高端解释空间 |
| `brand_claim_only` | 厂家主张卖点 | 卖点文本存在，但参数、评论或市场验证不足 |
| `user_validated_need` | 用户验证需求 | 评论强支持，但参数或卖点表达不足，代表用户需求存在 |
| `drag_factor` | 拖后腿卖点 | 用户负向集中、参数支撑不足，或在关键战场/任务/客群里削弱成交理由 |
| `opportunity_gap` | 机会缺口 | 同池强竞品具备且有市场价值，本品缺失或表达弱 |
| `sample_insufficient` | 样本不足 | 可比池、对照组或评论样本不足，不能判断 |

其中 `value_bundle_claim`、`weak_user_perception_claim`、`high_price_competitor_intercept`、`price_up_opportunity` 可以作为业务展示标签，也可以映射到 `opportunity_gap`、`drag_factor`、`brand_claim_only` 等底层角色。报告和智能体必须优先展示业务可读标签。

## 6. 量化指标

### 6.1 市场池级指标

每个 `claim_code + context` 至少输出：

| 指标 | 说明 |
| --- | --- |
| `with_claim_sku_count` | 可比池中具备该卖点的 SKU 数 |
| `without_claim_sku_count` | 可比池中不具备或弱具备该卖点的 SKU 数 |
| `with_claim_avg_price` | 有卖点组均价 |
| `without_claim_avg_price` | 对照组均价 |
| `price_premium_abs` | 可比池卖点价格差异：有卖点组价格中位数 - 对照组价格中位数 |
| `price_premium_rate` | 可比池卖点价格差异率 |
| `weekly_sales_lift_abs` | 可比池卖点周均销量差异：有卖点组周均销量中位数 - 对照组周均销量中位数 |
| `weekly_sales_lift_rate` | 可比池卖点周均销量差异率 |
| `weekly_sales_amount_lift_abs` | 可比池卖点周均销额差异 |
| `market_share_lift` | 可比池内销量份额优势 |
| `effect_confidence` | 效果置信度 |
| `sample_status` | 样本状态 |

市场池级指标只能说明“有该卖点的一组 SKU 与对照组之间存在可观测差异”。同池 SKU 往往同时具备多个卖点，因此该指标不能被解释为单个卖点的因果净贡献。

### 6.2 SKU 卖点贡献指标

每个 `sku_code + claim_code + context` 至少输出：

| 指标 | 说明 |
| --- | --- |
| `claim_value_role` | 卖点价值分类 |
| `claim_evidence_strength` | 卖点证据强度 |
| `semantic_support_strength` | 主战场/主任务/主客群支撑强度 |
| `comment_support_strength` | 评论正向/负向验证强度 |
| `param_support_strength` | 参数支撑强度 |
| `estimated_price_premium_abs` | 本品超额价格解释份额：本品高于同池基准的价格差中，由该卖点参与解释的部分 |
| `estimated_weekly_sales_lift_abs` | 本品超额周均销量解释份额：本品高于同池基准的周均销量差中，由该卖点参与解释的部分 |
| `estimated_weekly_sales_amount_lift_abs` | 本品超额周均销额解释份额：本品高于同池基准的周均销额差中，由该卖点参与解释的部分 |
| `contribution_share_in_sku` | 在 SKU 超额表现中的解释份额 |
| `attribution_confidence` | SKU 归因置信度 |
| `reason_cn` | 中文业务解释 |

SKU 卖点贡献指标不是“该卖点单独增加 X 元或 X 台/周”。它是在本品已经高于同池基准时，把超额表现按卖点证据强度进行解释性分摊。只有当可比池卖点价格差异为正、样本有效、且角色为 `premium_driver_estimated` 时，才能把该卖点称为“溢价卖点”。如果池级价格差异为负或不显著，只能称为“销量卖点”“基础门槛”“组合型增值卖点”或“用户感知不足卖点”。

### 6.3 不允许输出的指标口径

| 禁止口径 | 替代表达 |
| --- | --- |
| “卖点 A 导致销量增加 X 台” | “在可比池中，有卖点 A 且证据成立的一组 SKU，对应周均销量差异约 X 台” |
| “卖点 A 价值等于 X 元” | “卖点 A 在该池中的有卖点组相对对照组价格差异约 X 元” |
| “卖点 A 支撑本品多卖 X 元” | “本品高于同池基准的价格差中，约 X 元由卖点 A 参与解释” |
| “本品销量由这些卖点相加得到” | “这些卖点按证据权重解释了本品相对同池基准的超额表现，不能相加成因果贡献” |
| “全品类 MiniLED 平均溢价” | “在指定尺寸价格池和战场中的 MiniLED 可比池价格差异” |
| “池级差异为负的卖点也是价格支撑卖点” | “该卖点不构成独立价格支撑，应判断为销量卖点、基础门槛、组合型增值卖点或用户感知不足” |

表格备注必须说明：

1. `可比池卖点价格差异` 和 `可比池卖点销量差异` 是组间可观测差异，不是单卖点因果贡献。
2. `本品超额价格解释份额` 和 `本品超额销量解释份额` 是本品超额表现的解释性分摊，不表示“一个卖点单独带来多少钱或多少销量”。
3. 同池 SKU 往往同时具备多个卖点，M12C 的数值用于排序、解释和发现机会，不用于财务口径归因。

## 7. 输出结果

### 7.1 必须输出表

| 表 | 粒度 | 用途 |
| --- | --- | --- |
| `core3_claim_value_context_pool` | claim + context + pool | 可比池定义、样本、放宽路径 |
| `core3_claim_value_pool_metric` | claim + context + pool | 有卖点组/对照组的量价差异 |
| `core3_sku_claim_value_quantification` | SKU + claim + context | SKU 卖点价值角色和量化指标 |
| `core3_sku_claim_contribution_attribution` | SKU + context | SKU 超额表现如何分摊给多个卖点 |
| `core3_claim_value_dimension_summary` | claim + 语义维度 | 某卖点在战场/任务/客群中的市场表现 |
| `core3_claim_value_review_issue` | SKU 或 claim | 样本不足、对照不足、异常、需复核问题 |

### 7.2 CLI 输出能力要求

M12C 完成后必须提供一组可被 `catforge_analyst`、OpenClaw Skill、小奥智能体和飞书报告复用的原子能力。

| CLI | 主要回答 | 必须输出 |
| --- | --- | --- |
| `claim-value-space` | 某卖点在某市场池、价值战场、用户任务或目标客群里值不值钱 | 可比池样本、有卖点组/对照组、价格溢价、周均销量优势、周均销额优势、置信度、样本限制 |
| `sku-claim-value` | 某 SKU 的每个卖点分别是什么价值角色 | 溢价卖点、销量卖点、基础门槛、厂家主张、机会缺口、拖后腿、样本不足及证据 |
| `claim-contribution` | 某 SKU 卖得好主要由哪些卖点解释 | SKU 相对同池基准的价格/销量/销额超额表现、Top5 贡献卖点、贡献份额、限制说明 |
| `claim-opportunity-gaps` | 本品相对竞品缺哪些有市场价值的卖点 | 竞品有价值卖点、本品缺失或弱表达、对应市场空间、机会优先级 |
| `claim-value-compare` | 本品和竞品在核心卖点上的价值差异 | 横向对比矩阵、共同门槛、本品优势、竞品拦截、可解释价差/销量差 |

每个 CLI 都必须支持：

- `--project-id`
- `--category-code`
- `--product-category`
- `--batch-id`
- `--market-window`
- `--analysis-population`
- `--format json|text`
- `--top-n`
- `--debug`
- `--include-business-labels`
- `--include-intercept-opportunities`

其中 `--debug=false` 时不得暴露表名、字段名、rule version、JSON 原始字段和内部 code；`--debug=true` 可以返回技术排查信息。面向报告和智能体的 `--top-n` 默认应为 5，短答场景可以缩短为 3。

### 7.3 CLI 业务输出结构

所有 CLI 的 JSON 输出必须遵循统一外壳：

```json
{
  "status": "ok|not_found|ambiguous|insufficient_data|error",
```

### 7.4 自然语言路由要求

M12C 必须接入 `catforge_analyst ask`，支持自然语言问题路由。

| 用户问题类型 | 路由 CLI | 示例 |
| --- | --- | --- |
| 问某卖点值多少钱 | `claim-value-space` | “MiniLED 在 65 寸高端电视里值多少钱？” |
| 问某 SKU 哪些卖点溢价 | `sku-claim-value` | “海信 65E7Q 哪些卖点是溢价卖点？” |
| 问某 SKU 为什么卖得好 | `claim-contribution` | “这款为什么卖得好，靠哪些卖点？” |
| 问比竞品贵在哪里 | `claim-value-compare` | “海信 65E7Q 比创维 65A7H PRO 贵在哪里？” |
| 问怎么扩大销量 | `claim-opportunity-gaps` + `claim-contribution` | “这款要扩大销量应该补哪些卖点？” |
| 问竞品靠什么拦截 | `claim-value-compare` + `claim-opportunity-gaps` | “创维靠哪些卖点拦截海信？” |

自然语言路由必须先解析 SKU 和卖点。如果 SKU 或型号匹配到多个候选，返回 `ambiguous`，列出候选让用户二次选择；不得自动猜测。

### 7.5 Skill 要求

M12C 必须更新两类 Skill：

| Skill | 职责 |
| --- | --- |
| `catforge-insight` | 遇到卖点价值、卖点贡献、值多少钱等问题时转交 `catforge_analyst`，不虚构未实现的 insight 命令 |
| `xiaoao-home-appliance-market-analysis` | 面向业务用户调用 M12C 分析能力并组织回答，不暴露 CLI、表名、JSON 和模块编号 |

Skill 必须包含：

1. 固定问题到 CLI 的路由表。
2. 多 CLI 组合 SOP。
3. 回答结构模板。
4. 样本不足、无评论、无卖点、无竞品时的边界回答。
5. 飞书报告链接生成规则。
6. 禁止事项：不得把可观测贡献写成因果，不得把服务履约写成产品溢价，不得跨不可比池比较。

### 7.6 飞书报告集成要求

M12C 结果必须支持写入现有竞品分析飞书报告，并新增“卖点价值量化”章节。

报告至少包含：

1. 竞品横向对比章节中新增“卖点价值量化”小节，横轴为本品和前三竞品，纵轴为比较内容。
2. 每个产品画像中新增独立的“卖点价值量化”小节，不能混入“卖点画像”事实小节。
3. “卖点画像”继续展示事实：卖点是否存在、参数是否支撑、评论是否支持、属于哪个卖点维度。
4. “卖点价值量化”展示价值：Top5 卖点、业务类型、业务含义、可比池价格差异、可比池销量差异、本品超额解释份额、值钱的市场和业务解释。
5. 本品与前三竞品核心卖点横向对比表。
6. 卖点对应的价值战场、用户任务、目标客群证据。
7. 高价竞品拦截卖点、价格上探机会、组合型增值卖点、用户感知不足卖点和拖后腿卖点。
8. 样本限制和置信度说明。

飞书报告中的表达必须是业务语言，不得出现内部字段名或规则版本。表格不允许只写“卖点 + 指数”，必须让业务读者一眼看懂含义。

推荐表头：

| 排名 | 卖点 | 业务类型 | 业务含义 | 可比池卖点价格差异 | 可比池卖点销量差异 | 本品超额价格解释份额 | 本品超额销量解释份额 | 值钱的市场 | 业务解释 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |

业务类型示例：

| 业务类型 | 业务含义 |
| --- | --- |
| 强溢价卖点 | 在同尺寸、同价格带、同语义市场中，有该卖点且证据成立的一组 SKU 价格更高 |
| 强销量卖点 | 价格不一定更高，但更能解释同池周均销量或销额优势 |
| 基础门槛卖点 | 同池普遍具备，有了不加价，缺了会掉队 |
| 组合型增值卖点 | 单独不一定溢价，但与一组高价值卖点组合后支撑高端解释 |
| 用户感知不足卖点 | 参数或卖点存在，但评论验证弱、负向明显，或弱于高价竞品 |
| 高价竞品拦截卖点 | 同池高价竞品具备并能成交，本品缺失、表达弱或评论弱 |
| 价格上探机会卖点 | 高价 SKU 反复具备且有市场价值，本品补强后可能提升上探空间 |
| 拖后腿卖点 | 厂家主张、参数或评论之间不一致，削弱关键战场/任务/客群 |

表格下方必须放置口径备注：

```text
说明：可比池卖点价格差异/销量差异是有卖点组与对照组的可观测差异，不是单一卖点因果贡献。本品超额解释份额是把本品高于同池基准的价格、销量、销额表现按卖点证据权重做解释性分摊，不能理解为“一个卖点单独增加 X 元或 X 台/周”。同池 SKU 往往同时具备多个卖点，数值用于排序、解释和发现机会。
```

当某卖点的可比池价格差异为负或不显著时，报告不得把它列为“价格支撑卖点”。必须进一步判断：

1. 是否仍然是销量卖点。
2. 是否只是基础门槛。
3. 是否只在卖点组合中参与增值。
4. 是否本品评论感知不如高价竞品。
5. 是否应转为高价竞品拦截或价格上探机会分析。

## 8. 质量与校验

M12C 必须做以下校验：

1. 每个量化结果必须能追溯到 batch、market_window、context 和可比池。
2. 样本不足时不能强判溢价。
3. 只有一个 SKU 有某卖点时不能给出稳定量化。
4. 对照组价格或销量异常时必须 winsorize 或标记异常。
5. 同一 SKU 在不同解释视角下的贡献不能相互求和。
6. 输出的金额、销量必须区分累计窗口和周均窗口。
7. 服务履约、物流安装、售后不得被标记为产品溢价卖点。
8. 所有自然语言解释不得暴露内部表名、规则版本和 JSON 字段，除非处于调试模式。
9. 池级价格差异为负或不显著时，不得输出“强溢价卖点”。
10. SKU 超额解释份额不得被写成单卖点因果金额。
11. 同池高价竞品拦截必须基于同尺寸、同或相邻高价格带、同价值战场/任务/客群，并具备有效市场表现。
12. 组合型增值卖点必须说明它依赖的卖点组合，不能包装成单点溢价。

## 9. 验收标准

1. 能在 TV 当前批次生成卖点价值量化结果。
2. 能对一个 SKU 输出 Top5 主要溢价、销量、基础门槛、组合增值或机会卖点。
3. 能说明每个卖点的参数支撑、评论验证、语义支撑和市场池证据。
4. 能查询某个卖点在某价值战场中的价格溢价和周均销量优势。
5. 能识别基础门槛卖点，避免把同池普遍配置误判为溢价。
6. 能识别厂家主张但评论/市场验证不足的卖点。
7. 能识别竞品具备、本品缺失的机会卖点。
8. 能在报告中用业务语言表达“估算贡献”，不使用因果断言。
9. 能在池级价格差异为负时，正确降级为销量卖点、基础门槛、组合型增值、用户感知不足或高价竞品拦截，不误写成溢价卖点。
10. 能在飞书报告中把“卖点画像”和“卖点价值量化”分开展示。
11. 单元测试不依赖外部 LLM。
12. CLI 和 Skill 能被小奥智能体组合调用。

## 10. 与旧 M11.5 的关系

旧 M11.5 侧重“战场内卖点价值分层”，回答某卖点在某战场里是门槛、绩效、溢价倾向还是弱感知。

M12C 在新版链路中进一步做“价值量化和贡献归因”，回答：

- 这个卖点在可比池中对应多少价格溢价。
- 这个卖点在可比池中对应多少周均销量优势。
- 这个 SKU 的超额表现有多少可以由这些卖点解释。

因此：

- M04C 是卖点事实。
- M05C 是评论验证。
- M09C/M10C/M11C 是语义关系。
- M11D 是任务/客群/战场市场空间和销量分配。
- M12C 是卖点价值量化与 SKU 卖点贡献归因。

旧 M11.5 可以作为历史思想参考，但新版实现应以 M04C/M05C/M07/M09C/M10C/M11C/M11D 为输入，不复用旧 code 和旧结果。
