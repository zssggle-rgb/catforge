# CatForge Analyst 竞品问答 CLI 与小奥 Skill 需求

## 1. 背景

小奥已经能通过 `catforge_analyst` 回答“某 SKU 的竞品有哪些”一类问题，但当前竞品回答还存在三个问题：

1. OpenClaw 直接消费较大的 JSON 时，容易出现解析失败、工具错误外露或自行改写答案。
2. 竞品排序仍偏向“同尺寸、同价位、需求重合、参数相似、销量接近”的通用表达，不能稳定体现专业市场分析判断。
3. 聊天入口适合短结论，不适合承载完整证据矩阵、候选未选原因和详细对比页面。

因此需要把竞品问答拆成两个稳定产物：

- CLI 生成可直接发送的短摘要 `short_answer`，控制在 600 个中文字符以内。
- CLI 生成或返回详细分析报告链接 `report_url`，优先落在飞书文档中。

小奥 Skill 只负责识别意图、调用稳定 CLI、原样发送短摘要和报告链接，不再自行解析大 JSON 或重写竞品结论。

## 2. 核心业务定义

### 2.1 首选竞品定义

首选竞品不是“参数最像”或“价格最近”的 SKU，而是：

> 最容易进入同一批用户最终候选清单，并且会改变用户对目标 SKU 价值判断的 SKU。

系统选择首选竞品必须同时看七类因素：

```text
首选竞品 =
同一购买池
× 主辅价值战场加权重合
× 主辅用户任务加权重合
× 主辅目标客群加权重合
× 关键价值锚点可替代
× 替代压力
× 市场验证
```

其中：

- 同一购买池是前置条件，优先同尺寸、同尺寸档、同价格带或邻近价格带。
- 价值战场、用户任务、目标客群必须区分主辅关系，不能只数重合个数。
- 关键价值锚点包括用户愿意付费的核心卖点、参数能力、场景价值和评论验证。
- 替代压力说明候选 SKU 通过什么方式影响目标 SKU 成交：价格压制、价值替代、配置标杆、品牌/生态替代、下探分流或上探替代。
- 销量只用于验证候选是否具备真实市场分流能力，不能作为首选竞品的主要排序依据。

### 2.2 竞品角色

竞品输出必须至少支持以下业务角色：

| 角色 | 定义 |
| --- | --- |
| 首选直接竞品 | 同一购买池内，对目标 SKU 主支付理由形成最高强度拦截。 |
| 强直接竞品 | 购买池、战场、任务、客群和价值锚点都高度接近，但综合替代压力略低于首选。 |
| 价格贴身竞品 | 价格最接近，但价值战场、任务或客群重合不足以排为首选直接竞品。 |
| 下探分流竞品 | 价格低一个层级或明显更顺，但仍能承接目标 SKU 的部分核心需求。 |
| 上探替代竞品 | 价格更高或品牌/配置更强，可能吸走高预算用户。 |
| 场景替代竞品 | 同尺寸价格不完全贴合，但在某个关键任务、客群或战场上替代性强。 |
| 排除候选 | 有相似点但不足以进入重点竞品，应说明排除原因。 |

## 3. 目标

### 3.1 CLI 目标

`catforge_analyst` 需要把竞品问答升级为可直接面向业务用户的稳定输出：

1. 生成 Top 3 重点竞品摘要，默认只输出三款。
2. 按上述七类因素重新计算竞品业务排序。
3. 输出每款竞品的角色、替代压力、关键重合点、关键差异和市场验证。
4. 输出可直接发送给飞书/聊天入口的 `short_answer`。
5. 输出详细报告 payload，并在配置可用时生成飞书文档 `report_url`。
6. 在 JSON 中保留可复核字段，但不要要求 Skill 解析大 JSON 后再写答案。

### 3.2 Skill 目标

小奥 Skill 需要做到：

1. 识别“竞品是谁”“为什么选第一款”“和谁竞争”“直接竞品有哪些”等竞品意图。
2. 优先调用 CLI 的稳定竞品摘要命令。
3. 如果 CLI 返回 `short_answer`，必须原样发送，不得改写语气、重排名次或增删关键结论。
4. 如果 CLI 返回 `report_url`，必须把链接放入回答末尾。
5. 如果 CLI 返回多候选 SKU，必须让用户二次选择；飞书入口后续可渲染为选择卡片。
6. 如果 CLI 失败，必须给业务化失败提示，不暴露命令、JSON、stderr、堆栈、内部模块名或调试文本。

## 4. 非目标

本次不做以下事项：

- 不重跑 M00-M11D，也不改变事实层、语义层和市场图谱的生成逻辑。
- 不把 OpenClaw 变成竞品排序引擎。
- 不要求聊天回答展示完整证据矩阵。
- 不在聊天回答中展示内部 code、批次号、表名、字段名、规则版本、命令文本或原始 JSON。
- 不把累计销量作为竞品排序主依据。
- 不把服务履约、物流安装、售后体验作为产品竞品主线。
- 不把低价 SKU 简单判定为更强竞品；必须说明其是否仍在同一购买池和同一成交理由中。

## 5. 输入数据

竞品问答必须使用已经生成的事实和图谱结果，不直接读取原始四张表做业务结论。

| 数据 | 作用 |
| --- | --- |
| SKU 解析结果 | 确定目标 SKU，处理同名、模糊型号、Pro/非 Pro 混淆。 |
| M03B 参数事实画像 | 尺寸、尺寸档、核心参数能力、参数档位。 |
| M04C 卖点事实画像 | 标准卖点、卖点维度、卖点是否有参数支撑。 |
| M05C 评论事实画像 | 评论中的用途、人群、价值感知、正负向支撑、竞品提及。 |
| M07 市场画像 | 尺寸内价格带、周均销量、均价、平台结构、观察周。 |
| M09C 用户任务画像 | 主/辅/机会/拖后腿用户任务。 |
| M10C 目标客群画像 | 主/辅/观察/潜在/未满足客群。 |
| M11C 价值战场画像 | 主/辅/机会/品牌声明/用户观察/拖后腿战场。 |
| M11D 语义市场图谱 | 各战场、任务、客群的 SKU 覆盖和销量分配。 |
| M12C 卖点价值量化 | SKU×卖点价值角色、可比产品价格/销量/销额差异、本品可解释价差/销量差份额、竞品拦截、补强建议和拖后腿卖点。 |
| pairwise 销量重叠 | 重叠在售周内周均销量、均价、销额、平台结构。 |

## 6. 竞品选择需求

### 6.1 候选池

候选池必须先从购买池收敛，再做语义和价值判断。

优先级：

1. 同 SKU 尺寸，且同尺寸价格带或邻近价格带。
2. 同 M03B 五档尺寸段，且同尺寸段价格带或邻近价格带。
3. 同价值战场强重合，但价格偏离明显的上探或下探候选。
4. 用户任务、目标客群高度重合，但尺寸或价格偏离的场景替代候选。

必须输出候选进入池的原因，避免用户看到“很多大屏高画质电视都相似，为什么就是这三款”的疑问。

### 6.2 主辅加权重合

价值战场、用户任务、目标客群不能只计算重合数量。必须使用关系状态权重。

建议权重口径：

| 关系状态 | 权重 |
| --- | ---: |
| 主战场 / 主任务 / 主客群 | 1.00 |
| 辅战场 / 辅任务 / 辅客群 | 0.75 |
| 用户观察成立 | 0.45 |
| 机会 / 潜在 | 0.35 |
| 品牌声明 | 0.25 |
| 未满足 / 拖后腿 | -0.30 |
| 排除 / 不支持 | 0 |

评分必须体现主辅相互命中：

- 目标主项命中候选主项，权重最高。
- 目标主项命中候选辅项，仍然是强命中。
- 目标辅项命中候选主项，可说明候选在该方向更聚焦。
- 机会、潜在、品牌声明只能作为补充，不能单独把候选推为首选竞品。
- 拖后腿和未满足是风险，不得被当成正向重合。

### 6.3 关键价值锚点

关键价值锚点必须从业务角度表达，不把回答写成参数清单。

对电视品类，价值锚点至少包括：

- 尺寸和空间升级价值。
- 画质和显示体验。
- 高刷、游戏和运动流畅体验。
- 影音沉浸。
- 智能互联和易用。
- 护眼和长时间观看。
- 外观、贴墙、家装融合。
- 预算内配置获得感。

系统必须判断候选 SKU 是否在目标 SKU 的核心支付理由上提供了可替代方案，而不是仅说明“参数相似”。

### 6.4 替代压力

每个入选竞品都必须生成替代压力说明。

| 替代压力 | 判断口径 |
| --- | --- |
| 价值替代压力 | 在同一购买池内，承接相同主战场和主客群，且关键价值锚点可替代。 |
| 价格压制压力 | 价格更低但仍保留目标 SKU 关键成交理由，削弱目标 SKU 溢价解释。 |
| 配置标杆压力 | 参数或卖点能力更强，抬高用户对目标 SKU 的配置预期。 |
| 场景心智压力 | 在目标 SKU 重要场景中表达更清晰，例如客厅空间、家装融合、游戏场景。 |
| 品牌/生态压力 | 品牌心智、系统生态、渠道表达使其进入同一候选清单。 |
| 下探分流压力 | 用户降低预算后仍能满足部分核心需求。 |
| 上探替代压力 | 用户追加预算后能获得更明确的高端理由。 |

### 6.5 市场验证

市场验证只回答“这是不是纸面竞品”：

- 使用重叠在售周的周均销量、周均销额、均价和平台结构。
- 用于验证候选是否有真实成交能力。
- 不用累计销量作为排序依据。
- 不把销量接近度写成“选择竞品”的核心证据。
- 若候选销量极低或重叠周不足，应降低置信度或标注样本限制。

## 7. CLI 功能需求

### 7.1 命令形态

优先增强现有 `competitor-set`，新增面向小奥的答案输出字段。

```bash
python -m app.cli.catforge_analyst competitor-set \
  --query "海信 65E7Q" \
  --product-category tv \
  --batch-id latest \
  --limit 10 \
  --format json \
  --answer-style xiaoao \
  --with-report feishu-doc
```

`--format text` 必须只输出可直接发送的短摘要：

```bash
python -m app.cli.catforge_analyst competitor-set \
  --query "海信 65E7Q" \
  --product-category tv \
  --batch-id latest \
  --format text \
  --answer-style xiaoao \
  --with-report feishu-doc
```

### 7.2 JSON 输出字段

`competitor-set --format json` 必须新增或稳定以下字段：

```json
{
  "status": "ok",
  "command": "competitor-set",
  "target": {},
  "result": {
    "competitor_answer": {
      "short_answer": "可直接发送的中文摘要",
      "report_url": "https://...",
      "report_status": "created",
      "top_competitors": [],
      "candidate_buckets": {},
      "selection_policy_cn": [],
      "display_policy": {
        "send_short_answer_as_is": true,
        "max_chat_chars": 600,
        "hide_internal_fields": true
      }
    }
  },
  "limitations": []
}
```

### 7.3 短摘要要求

短摘要必须满足：

- 600 个中文字符以内。
- 只讲 Top 3。
- 第一句话直接给结论。
- 使用业务语言，不展示内部 code、分数、命令、模块、批次、字段名。
- 不使用“不是……而是……”作为主要句式。
- 不使用“数据很完整”“下面是回答”“根据工具/CLI/SOP”。
- 不把销量写成首要排序原因。
- 必须以“详细分析报告见飞书链接：{report_url}”收尾；没有链接时写“详细分析报告暂未生成”。

摘要建议结构：

```text
海信 65E7Q 的重点竞品建议看三款：创维 65A7H PRO、TCL 65Q9L PRO 和创维 65A6F ULTRA。
创维 65A7H PRO 排第一，是因为它和海信处在同一 65 寸高价购买池，并且在高端画质升级、影院沉浸和家庭客厅体验上覆盖了海信最核心的成交理由，对海信的技术溢价形成直接替代压力。
TCL 65Q9L PRO 更像配置标杆型竞品，在画质、高刷、影音和游戏娱乐上与海信正面对比，会抬高用户对同价段配置的预期。
创维 65A6F ULTRA 属于下探分流竞品，价格明显更顺，但仍能承接一部分高端画质和家庭观影需求，会分流预算敏感但不愿放弃核心体验的用户。
详细分析报告见飞书链接：{report_url}
```

### 7.4 详细飞书报告要求

飞书报告必须承载聊天中放不下的完整分析。

报告必须包括：

1. 标题：`{目标品牌}{目标型号} 重点竞品分析报告`。
2. 结论摘要：三款竞品、各自角色、核心替代压力。
3. 为什么第一名是首选竞品：按购买池、战场、任务、客群、价值锚点、替代压力、市场验证展开。
4. Top 3 竞品卡：价格、尺寸、周均销量、主要角色、业务压力。
5. 竞争关系矩阵：价值战场、用户任务、目标客群、关键价值锚点、卖点/参数、评论支撑、市场验证。
6. 卖点价值量化：按目标 SKU 和前三竞品列出核心卖点商业价值、价格溢价卖点、销量驱动卖点、基础门槛卖点、组合型增值卖点、竞品拦截与补强建议、卖点有效市场、可比产品价格/销量差异、本品可解释价差/销量差份额和证据支撑强度。
7. 其他候选为什么未入选：价格过远、场景偏离、主客群不同、仅参数相似、销量验证不足等。
8. 口径说明：当前样本范围、线上渠道限制、重叠周口径、销量只是市场验证，M12C 卖点价值是同尺寸价格带和业务市场场景内的可观测估计，不是严格因果。

报告当前不输出产品经理策略、导购话术、应对策略或泛化建议。需要建议类问题时，由独立 SOP 另行生成。

报告中的销量、周均销量、空间销量等“台数”必须展示为整数台，不展示 `113,034.5台` 这类小数台。价值战场、用户任务、目标客群的图谱空间缺失时，不得输出“图谱空间待生成”“暂无该分类市场空间数据”等技术提示，应使用“未纳入当前销量空间测算”。SKU 在主辅维度内暂未分配销量时，使用“当前图谱未分配本品销量”；机会、拖后腿、评论观察等补充关系没有销量分配时，使用“未分配销量，仅作机会或观察证据”。

### 7.5 卖点价值量化要求

竞品比较报告中的“溢价卖点”必须优先使用 M12C，而不是只用“事实卖点 + 评论正向支撑”推断。

报告必须把“卖点画像”和“卖点价值量化”分开：

- 卖点画像只展示事实卖点、评论支持/反向、参数支撑、需复核表达和共同价值锚点。
- 卖点价值量化单独成章，横向对比和每个产品画像中都必须展示。

每个 SKU 的卖点价值量化至少展示：

| 项目 | 业务含义 |
| --- | --- |
| Top5 核心卖点商业价值 | 按强溢价、强销量、组合型增值、机会/拦截和拖后腿风险综合排序，只展示前 5 个，避免长表淹没重点。 |
| 强溢价卖点 | M12C `premium_driver_estimated`，表示有卖点组在同池价格差异为正，且本 SKU 有参数/评论/语义支撑。 |
| 强销量卖点 | M12C `sales_driver_estimated`，表示价格差异不作为价格支撑，但同池有正向周均销量或销额差异。 |
| 组合型增值卖点 | M12C `value_bundle_claim`，表示单项价格支撑不强，但与主战场、主任务、主客群共同组成成交理由。 |
| 基础门槛卖点 | M12C `basic_threshold`，表示用户期望具备但不一定形成额外溢价的卖点。 |
| 用户感知不足卖点 | M12C `weak_user_perception_claim`，表示厂家有表达或参数存在，但评论、任务或市场验证不足，不得作为正向溢价依据。 |
| 高价竞品拦截/价格上探机会 | M12C `high_price_competitor_intercept` / `price_up_opportunity`，表示同池高价或高成交竞品具备、本品缺失或表达弱的卖点。 |
| 厂家主张卖点 | M12C `brand_claim_only`，表示厂家表达较强但市场或评论验证不足。 |
| 机会缺口 | M12C `opportunity_gap`，表示竞品或同池强 SKU 具备且有市场价值，本品缺失或表达弱。 |
| 拖后腿卖点 | M12C `drag_factor`，表示评论、参数或语义支撑不足，会削弱成交理由。 |
| 可比池卖点价格/销量差异 | 有卖点组与对照组的可观测差异，只能作为解释性证据，不能写成“该卖点导致价格/销量增加”。 |
| 本品可解释价差/销量差份额 | 本品相对可比产品基准的价格、周均销量或周均销额差异，按卖点证据权重做解释性分摊。负向、感知不足或拖后腿卖点不得展示为正向分摊。 |

如果某 SKU 没有 M12C 结果，报告可以退回事实卖点口径，但必须写成“卖点价值量化待生成”，不得伪造量化指标。

## 8. Skill 功能需求

小奥 Skill 需要新增竞品问答专项规则。

### 8.1 路由规则

以下问题必须路由到竞品摘要 CLI：

- “某 SKU 的竞品有哪些”
- “和谁竞争”
- “直接竞品是谁”
- “第一款为什么选它”
- “为什么创维 65A7H PRO 是海信 65E7Q 的首选竞品”
- “哪些 SKU 会抢这款的订单”

### 8.2 输出规则

Skill 必须遵守：

1. 如果 CLI 返回 `display_policy.send_short_answer_as_is = true`，直接发送 `short_answer`。
2. 不把 `top_competitors` 重新组织成另一套口径。
3. 不自行补充泛化市场常识。
4. 不暴露英文内部字段和技术模块。
5. 不在答案末尾生成“需要我继续分析吗”一类通用客服语。
6. 后续追问“第一款”“它”“上面那个”时，必须沿用上一轮上下文中的候选，而不是重新模糊解析。

### 8.3 异常规则

| CLI 状态 | Skill 回答要求 |
| --- | --- |
| `ambiguous` | 列出候选 SKU，让用户选择；飞书入口可用卡片。 |
| `not_found` | 说明当前分析样本未找到该型号，请用户补充品牌、型号或 SKU code。 |
| `unsupported` | 说明当前数据暂不支持该问题，并给出可替代问题。 |
| `error` | 说明当前竞品分析包未能生成，不输出内部报错。 |
| `report_failed` | 仍发送短摘要，并说明详细报告暂未生成。 |

## 9. 验收问题

首轮验收使用海信 65 寸电视，不再使用红米样例。

验收问题：

1. “海信 65E7Q 的竞品有哪些？”
2. “为什么创维 65A7H PRO 是海信 65E7Q 的首选竞品？”
3. “TCL 65Q9L PRO 和创维 65A7H PRO 哪个对海信威胁更大？”
4. “为什么小米 L65MC-SP 价格更接近但不是首选竞品？”
5. “详细分析报告链接是什么？”

验收标准：

- 第一问返回三款重点竞品，且聊天摘要不超过 600 字。
- 第一名不是简单按价格最近或销量接近排序。
- 能解释“很多 65 寸高画质电视都相似，为什么这款排第一”。
- 能说明小米 L65MC-SP 是价格贴身竞品，但因任务、客群或价值锚点重合不足，不排第一。
- 详细报告能打开，且报告中有候选未选原因。
- 回答中不出现 CLI、Mxx、JSON、命令、stderr、内部 code、英文 price band。

## 10. 测试要求

测试必须不依赖真实飞书 API 和外部 LLM。

必须覆盖：

- SKU 模糊解析，尤其 Pro/非 Pro 混淆。
- 主辅价值战场加权重合。
- 主辅用户任务加权重合。
- 主辅目标客群加权重合。
- 价格最近但语义弱的候选不得排第一。
- 销量相近不得单独推高为首选竞品。
- `short_answer` 不超过 600 中文字符。
- `short_answer` 不包含内部字段、命令、模块或 JSON。
- 飞书报告生成失败时仍可返回短摘要。
- `--format text` 输出与 JSON 中 `short_answer` 一致。
- 竞品详细报告必须把卖点画像和卖点价值量化拆开：卖点画像展示事实，卖点价值量化展示 M12C 的核心卖点商业价值、可比产品差异、本品可解释价差/销量差份额和业务标签。
- M12C 缺失时报告明确提示“卖点价值量化待生成”，并用事实卖点/评论支撑作为兜底，不得伪造指数。

## 11. 与旧 M12-M15 的关系

旧 M12-M15 是传统核心三竞品结果链路，保留历史兼容。

本需求面向小奥问答入口，不要求沿用旧三槽位输出。它消费 M03B/M04C/M05C/M07/M09C/M10C/M11C/M11D/M12C 等新版事实、语义资产和卖点价值量化结果，生成适合聊天和飞书报告的业务问答结果。

后续如果旧 M12-M15 继续演进，应把本需求中的首选竞品定义、主辅加权重合和报告话术约束同步进去。
