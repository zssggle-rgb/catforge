# M08.6 参数-卖点-评论分层产品锚点校准需求

## 1. 模块定位

M08.6 是 205 真实数据运行后新增的专项修正需求，用于固化“价值战场、用户任务、目标客群不能只靠评论或预设命中，必须有可解释的产品锚点”的业务口径。

M08.6 不是独立替代 M03、M08.4、M08.5 的新业务模块，而是跨模块校准层：

| 涉及模块 | 原职责 | M08.6 对它的修正要求 |
| --- | --- | --- |
| M03 参数抽取 | 从原始参数表生成标准参数值和 SKU 参数画像 | 修正字段误映射，增加相近参数和代理参数规则 |
| M04a/M04b 卖点激活 | 从结构化卖点和评论验证形成卖点激活 | 明确卖点是 claim 证据，不得当成原始参数 |
| M08 SKU 画像 | 汇总参数、卖点、评论、市场信号 | 为下游提供可裁剪的分域证据入口 |
| M08.4 评论原生维度发现 | 发现评论原生业务维度并生成产品锚点 | 输出参数锚点、卖点锚点、评论验证锚点和综合锚点 |
| M08.5 业务维度本体校准 | 生成价值战场、用户任务、目标客群定义 | 消费分层锚点，避免所有 SKU 命中所有维度 |
| M09/M10/M11 | 生成任务、客群、战场 SKU 画像 | 只能消费 M08.5 发布的可用维度和 M08.4 分层锚点 |

## 2. 当前问题

### 2.1 业务问题

当前 205 真实数据运行暴露出三个问题：

1. 部分价值战场、用户任务和目标客群缺少产品锚点，导致“为什么这款 SKU 属于这个战场/任务/客群”解释不足。
2. 部分维度命中范围过宽，出现大量 SKU 同时命中多个互相冲突的战场或客群，削弱分析区分度。
3. 有结构化参数的数据没有被充分使用，反而过度依赖评论或预设分类。

业务上必须改成：

```text
每个 SKU 可以没有结构化卖点，但不能没有产品定位判断。
没有卖点时，优先用参数和市场可比关系判断。
有卖点时，必须结合参数判断卖点是否可信。
评论只验证用户感知，不能单独证明产品能力。
```

### 2.2 数据问题

205 真实数据排查显示：

| 数据域 | 当前情况 | 需求判断 |
| --- | --- | --- |
| 原始参数 | 84 个 SKU 均有清洗后参数 | 可以支撑基础产品定位 |
| 标准参数 | 84 个 SKU 均有 M03 参数画像 | 参数覆盖不是主要问题 |
| 原始卖点 | 只有部分 SKU 有结构化卖点 | 不能要求每个 SKU 都有 claim |
| 卖点激活 | M04a/M04b 可生成更多 claim activation | claim activation 不是原始卖点本身 |
| 评论 | 行数大、重复多、低价值评论多 | 只能作为验证和感知补充 |
| 音效/护眼等参数 | 原始参数缺少稳定字段 | 不能伪造参数支撑，只能标记缺口 |

### 2.3 参数误映射问题

当前 M03 存在典型误映射，必须拦截：

| 原始字段 | 错误映射 | 正确处理 |
| --- | --- | --- |
| 内置WIFI | `speaker_power_w` | 不得作为声音参数 |
| HDMI数量、USB数量、屏幕比例、机身厚度 | `color_depth_bit` | 改成对应接口、尺寸或设计参数，无法判断则 unmapped |
| 人工智能、全屋智控、无缝贴墙 | `motion_compensation_flag` | 改成智能/家居/设计类字段或 unmapped |
| HDR | `peak_brightness_nits` | 改成 `hdr_support_flag` 或 `hdr_format_list` |
| 面板刷新率 | `panel_type` | 应优先匹配刷新率，不得匹配面板类型 |

## 3. 需求目标

### 3.1 业务目标

M08.6 完成后，系统必须支持以下业务表达：

1. 每个 SKU 都能输出“主要产品定位证据”，即便没有结构化卖点。
2. 每个价值战场必须说明由哪些参数、卖点、评论或市场证据支撑。
3. 每个用户任务必须说明其产品支撑点，不能只是评论语义标签。
4. 每个目标客群必须通过任务、战场、价格带、尺寸段或评论人群线索间接支撑，不强行要求直接参数。
5. 服务、安装、物流、售后等履约线索必须剥离为服务语境，不进入产品价值战场。
6. 音效、护眼等当前缺少原始参数的方向，可以由卖点和评论形成候选，但不得伪造成强参数锚点。
7. 下游销量分配和 SKU 综合画像能够区分“参数支撑”“卖点支撑”“评论验证”“证据缺口”。

### 3.2 工程目标

工程上必须实现：

1. M03 字段映射质量增强，避免错误参数污染后续分析。
2. M08.4 输出分层产品锚点，而不是单一 `product_anchor_score`。
3. M08.5 为价值战场和用户任务发布明确的产品锚点规则。
4. M09/M10/M11 后续使用分层锚点做 SKU 分类和权重估算。
5. 205 真实数据重跑后可通过页面或 API 看到每个维度的支撑来源。

## 4. 证据分层定义

### 4.1 参数锚点

参数锚点来自 M03 标准参数，不来自卖点表。

参数锚点分三类：

| 类型 | 定义 | 示例 | 业务强度 |
| --- | --- | --- | --- |
| 强参数锚点 | 直接证明产品能力 | 刷新率、屏幕尺寸、MiniLED、分区背光、RAM、ROM | 高 |
| 代理参数锚点 | 不能直接证明，但可作为相近能力证据 | HDR 支持、背光源、能效等级、机身厚度 | 中 |
| 无效参数锚点 | 字段名或值与能力不匹配 | 内置WIFI 支撑 speaker_power | 无效 |

### 4.2 卖点锚点

卖点锚点来自 M04a/M04b：

1. 原始结构化卖点可形成强 claim 证据。
2. 参数支撑的卖点可形成可信 claim 证据。
3. 仅评论增强但没有结构化卖点或参数支撑的 claim 只能作为弱证据。
4. claim 不能被写成 param，也不能替代缺失的原始参数字段。

### 4.3 评论验证锚点

评论验证锚点来自 M05/M06：

1. 评论说明用户感知到什么，不直接证明产品硬件能力。
2. 评论可增强参数或卖点可信度。
3. 低价值评论、重复模板评论、服务履约评论不得进入产品能力判断。
4. 评论缺失不等于产品能力不存在，只表示感知证据不足。

### 4.4 市场锚点

市场锚点来自 M07：

1. 用于解释价格带、销量、渠道、可比池位置。
2. 可支撑“大屏性价比”“价格段定位”“市场角色”等判断。
3. 市场锚点不能单独证明“护眼”“音效”“游戏”等产品能力。

## 5. 价值战场、用户任务、目标客群的区分

### 5.1 价值战场

价值战场回答：

```text
这款 SKU 靠什么产品价值参与竞争？
```

必须有产品锚点。优先来自参数和卖点，评论只能验证。

示例：

| 价值战场 | 产品锚点 |
| --- | --- |
| 游戏体育流畅战场 | 高刷新率、HDMI 2.1、运动补偿、游戏卖点 |
| 大屏沉浸画质战场 | 屏幕尺寸、分辨率、亮度、HDR、背光源、画质卖点 |
| 智能交互易用战场 | 系统、内存、存储、语音、AI、大模型 |
| 家庭护眼舒适战场 | 低蓝光、无频闪、护眼卖点、护眼评论 |

### 5.2 用户任务

用户任务回答：

```text
用户买这台电视主要为了完成什么使用任务？
```

用户任务必须有场景线索和产品支撑，不要求每个任务都有独立硬件参数，但必须能回到战场或产品锚点。

示例：

| 用户任务 | 支撑关系 |
| --- | --- |
| 客厅大屏观影 | 大屏、画质、音画沉浸、客厅评论场景 |
| 游戏娱乐 | 刷新率、HDMI、低延迟卖点、游戏评论场景 |
| 长辈易用观看 | 语音、系统易用、遥控/操作评论、长辈场景 |

### 5.3 目标客群

目标客群回答：

```text
这台电视主要服务哪类购买人群？
```

目标客群可以通过任务、价格带、尺寸段、战场和评论人群线索间接推断。目标客群不直接要求“参数字段”，但必须能解释到产品和使用任务。

示例：

| 目标客群 | 支撑关系 |
| --- | --- |
| 客厅家庭升级人群 | 大尺寸、高销量价格带、家庭观影任务 |
| 游戏影音年轻人群 | 游戏体育战场、游戏任务、高刷/HDMI 支撑 |
| 长辈家庭人群 | 智能易用战场、长辈易用任务、语音/操作评论 |

## 6. 下游消费规则

### 6.1 M08.4 输出要求

M08.4 对每个 SKU 和每个候选业务维度输出：

| 字段 | 含义 |
| --- | --- |
| `param_anchor_score` | 强参数直接支撑分 |
| `proxy_param_anchor_score` | 代理参数支撑分 |
| `claim_anchor_score` | 卖点支撑分 |
| `comment_validation_score` | 评论验证分 |
| `market_anchor_score` | 市场支撑分 |
| `overall_anchor_score` | 综合锚点分 |
| `anchor_source_status` | `param_only`、`claim_plus_param`、`claim_only`、`proxy_param_only`、`comment_only`、`no_direct_anchor` |
| `anchor_quality_flags` | 映射可疑、参数缺口、卖点缺口、评论低质等 |

### 6.2 M08.5 发布要求

M08.5 发布每个业务维度时必须带：

1. 维度名称和业务定义。
2. 维度类型：价值战场、用户任务、目标客群、服务语境。
3. 强参数锚点列表。
4. 代理参数锚点列表。
5. 卖点锚点列表。
6. 评论主题验证列表。
7. 是否允许进入 SKU 分类。
8. 是否允许进入销量分配。
9. 不允许进入的原因。

### 6.3 M09/M10/M11 消费要求

后续模块必须遵守：

1. 价值战场不能消费 `comment_only` 作为强支撑。
2. 用户任务可以消费评论场景，但必须有战场或产品锚点回链。
3. 目标客群可以间接消费任务和战场，但不能只靠泛化好评。
4. 服务语境永远不进入产品价值战场和销量分配。
5. SKU 在多个维度中可以多重归属，但权重合计必须可校验。

## 7. 质量规则

### 7.1 必须阻断

以下情况必须阻断进入强锚点：

1. 参数字段命中黑名单误映射。
2. 参数值为空、`unknown`、`-`、无法解析。
3. 原始字段名与目标参数语义明显不一致。
4. 仅服务评论支撑产品价值战场。
5. 仅低价值重复评论支撑业务维度。

### 7.2 必须降级

以下情况可保留为候选，但不得作为强支撑：

1. 只有卖点但没有参数或评论验证。
2. 只有评论但没有参数或卖点。
3. 只有代理参数，没有直接参数。
4. 当前数据缺少该能力的原始字段，例如音效功率、低蓝光、无频闪。

### 7.3 必须复核

以下情况进入复核：

1. 某维度命中 SKU 占比过高，导致区分度不足。
2. 某 SKU 同时强命中多个业务互斥维度。
3. M03 参数映射 confidence 低但被下游使用。
4. M08.4 锚点综合分高但分层证据单一。

## 8. 验收标准

M08.6 的验收必须基于 205 真实数据和本地 fixture 双口径：

1. M03 重跑后，已知误映射样例不再作为强参数。
2. 84 个 SKU 均保留参数画像，不因卖点缺失失去定位能力。
3. 有结构化卖点的 SKU 能看到 `claim_plus_param` 或 `claim_only` 状态。
4. 没有结构化卖点的 SKU 能通过参数形成 `param_only` 或 `proxy_param_only` 状态。
5. 价值战场不再出现所有 SKU 全量强命中。
6. 用户任务能回链到产品锚点或价值战场。
7. 目标客群能回链到任务、战场、价格带或评论人群线索。
8. 服务履约语境不进入产品价值战场。
9. 页面或 API 能解释“为什么这个 SKU 属于这个战场/任务/客群”。
