# CatForge Analyst 用户卖点价值看板需求

## 1. 背景

M12C 已经能够生成 SKU 粒度的用户卖点支付价值分析，并支持 Markdown/飞书详细报告。但在飞书会话中，单纯返回短答或长文档链接还不够直观，业务用户需要像竞品分析卡片一样，在会话主屏先看到清晰的“卖点价值看板”：

- 这款 SKU 当前真正能形成用户支付价值的卖点是什么。
- 哪些卖点是高溢价、份额转化、人无我有、门槛、待激活或竞品拦截。
- 正向卖点主要在哪些价值战场成立。
- 每个重点卖点能解释多少价格或销量表现。
- 完整证据和计算过程通过飞书文档继续下钻。

因此需要在 `catforge_analyst sku-claim-value` 和 `ask` 路由上新增稳定的卡片式看板输出，复用 M12C 已有量化结果和飞书报告能力，不让 OpenClaw/小奥自己解析大 JSON 后二次改写答案。

## 2. 模块定位

用户卖点价值看板是 CatForge Analyst 的展示/交互层，位于 M12C 量化结果之后。

```text
M03B/M04C/M05C/M07/M09C/M10C/M11C/M11D
  -> M12C 用户卖点支付价值量化
  -> CatForge Analyst 用户卖点价值看板
  -> 飞书卡片 / 文本短答 / 飞书详细报告
```

看板不改变 M12C 的计算逻辑，不新建独立量化算法，只负责把已经计算好的 M12C 结果裁剪成业务用户能直接理解的会话主屏。

## 3. 业务目标

### 3.1 面向的问题

看板必须支持回答：

- “海信 65E7Q 的用户卖点价值是什么？”
- “这款 SKU 哪些卖点能支撑溢价？”
- “5200nits 高亮对用户支付价值有多大？”
- “这款 SKU 有哪些门槛卖点和待激活卖点？”
- “竞品从哪些卖点拦截本品？”
- “卖点价值详细报告在哪里？”

### 3.2 主屏要回答的三个问题

| 问题 | 看板回答 |
| --- | --- |
| 当前正向支付价值在哪里 | 展示高溢价/份额转化/客户获得价值/人无我有型支付价值卖点 Top 项 |
| 为什么这些卖点成立 | 展示成立价值战场、可解释金额/销量、参数竞争力、用户评论和市场承接摘要 |
| 哪些卖点不能当溢价卖点 | 展示门槛、待激活、厂家主张、竞品拦截和价格压力，并说明业务含义 |

## 4. 核心业务定义

### 4.1 看板主结论

看板结论必须使用业务语言表达：

```text
海信 65E7Q 的正向支付价值集中在 300Hz 高阶刷新率、5200nits 高亮、98% 色彩表现和 1920 分区控光。
其中高亮、色彩和分区控光主要支撑高端画质升级战场；300Hz 高刷主要支撑游戏体育流畅战场。
HDMI2.1、MiniLED 和杜比更多是入围门槛，不单独解释加价。
```

不得把“价值分”作为主结论，不得把内部字段名、模块名、批次号、SQL、JSON 或命令文本暴露给用户。

### 4.2 卖点分类

看板沿用 M12C 的业务分类，但展示顺序必须固定：

1. 高溢价卖点
2. 份额转化卖点
3. 客户获得价值卖点
4. 人无我有型支付价值卖点
5. 门槛卖点
6. 待激活卖点
7. 厂家主张卖点
8. 竞品拦截卖点
9. 价格压力卖点
10. 样本不足待复核

分类含义必须用业务解释，而不是技术状态：

| 分类 | 看板解释 |
| --- | --- |
| 高溢价卖点 | 用户愿意为该卖点支付更高价格，并且市场承接没有明显反证 |
| 份额转化卖点 | 不一定抬价，但能解释同价或相近价格下的销量/份额承接 |
| 客户获得价值卖点 | 用户觉得更值，形成选择理由，但价格没有完全收取 |
| 人无我有型支付价值卖点 | 本品具备稀缺能力，可能提高用户最高支付意愿，但当前对照不足，不展示金额 |
| 门槛卖点 | 有了不加价，缺了会掉队 |
| 待激活卖点 | 产品事实或厂家表达存在，但用户感知和市场验证不足 |
| 厂家主张卖点 | 当前主要是厂家表达，尚未形成稳定用户支付价值 |
| 竞品拦截卖点 | 竞品具备并形成表达或验证，本品缺失或较弱 |
| 价格压力卖点 | 卖点没有有效支撑当前价格，可能削弱成交理由 |
| 样本不足待复核 | 当前样本或对照不足，只能作为观察线索 |

## 5. 输入数据

看板只消费已有分析结果，不直接读取原始表生成结论。

| 输入 | 作用 |
| --- | --- |
| SKU 解析结果 | 确定目标 SKU，处理型号模糊和 Pro/非 Pro 混淆 |
| M07 市场画像 | 展示均价、周均销量、尺寸价格带等目标概览 |
| M12C `sku_level_claim_values` | 构建卖点价值总榜、分类结构、Top 卖点 |
| M12C `claim_values` | 构建分价值战场来源和证据明细 |
| M12C `attributions` | 补充整机归因摘要和机会/风险说明 |
| M12C Markdown/飞书报告 | 作为完整证据和计算过程入口 |

## 6. 看板内容需求

### 6.1 顶部结论

顶部必须包含：

- 目标 SKU 名称。
- 当前正向支付价值的核心卖点。
- 卖点主要成立的价值战场。
- 门槛/待激活/竞品拦截的关键提醒。
- 完整报告入口。

顶部结论控制在 120-180 个中文字符，不能展开长推导。

### 6.2 卖点价值结构

必须展示各分类数量，形成结构化判断：

```text
高溢价 4 个｜门槛 3 个｜待激活 3 个｜竞品拦截 3 个
```

飞书卡片优先用图表呈现；如果卡片图表不可用，使用文字结构条。

### 6.3 Top 卖点表

Top 卖点表默认展示 5 个以内，优先展示正向支付价值卖点。

字段：

| 字段 | 说明 |
| --- | --- |
| 卖点 | 业务可读卖点名，优先使用参数档位化名称，例如 `5200nits 高亮档位` |
| 分类 | 高溢价、份额转化、客户获得价值、人无我有等 |
| 成立战场 | 主要价值战场，最多 2 个 |
| 可解释金额/潜力 | 正向量化卖点展示金额；人无我有展示潜力等级；门槛/待激活不展示金额 |
| 可解释销量 | 正向量化卖点展示周均销量解释；不可量化显示“暂不量化” |
| 证据 | 参数竞争力、评论感知、市场承接的业务摘要 |

硬约束：

- 不展示 `pool_claim_price_delta_abs`、`pool_claim_weekly_sales_delta_abs` 等原始组间差。
- 不把上千元原始可比池差异写成卖点支付价值。
- 不把“价值分 84.3”作为 Top 表默认列。

### 6.4 价值战场来源

看板必须说明正向卖点主要在哪些价值战场成立：

| 战场 | 典型展示 |
| --- | --- |
| 高端画质升级 | 5200nits 高亮、98% 色彩、1920 分区控光 |
| 游戏体育流畅 | 300Hz 高刷 |
| 家庭护眼舒适 | 若只有门槛或待激活，不放入正向溢价战场 |

飞书卡片可用横向条形图或短表。

### 6.5 风险与待激活

看板必须给出风险提示，但不能把风险项混进本品正向卖点 Top。

至少展示：

- 门槛卖点：有助于入围，不单独加价。
- 待激活卖点：有产品基础，但用户感知或市场验证不足。
- 竞品拦截卖点：竞品有、本品弱或缺失。
- 价格压力卖点：卖点无法支撑当前价格。

### 6.6 报告入口

飞书卡片必须包含“查看完整报告”按钮或链接，链接指向同一次 CLI 生成的飞书详细报告。

如果飞书报告生成失败：

- 卡片仍可展示摘要和结构。
- 展示“详细报告暂未生成，请稍后重试或查看 Markdown 输出。”
- 不把本地 Markdown 路径发给普通飞书用户。

## 7. CLI 与 Skill 需求

### 7.1 CLI

`sku-claim-value` 和 `ask` 必须支持：

```bash
python -m app.cli.catforge_analyst sku-claim-value \
  --query "海信 65E7Q" \
  --answer-style xiaoao \
  --with-report feishu-doc \
  --format json
```

JSON 输出必须包含：

```json
{
  "result": {
    "claim_value_answer": {
      "short_answer": "...",
      "dashboard_payload": {},
      "feishu_card_payload": {},
      "report": {
        "status": "created",
        "url": "https://..."
      }
    }
  }
}
```

`--format text` 输出只允许包含可直接发送给用户的短答，不输出内部 JSON。

### 7.2 飞书卡片发送

飞书入口如果传入 `--feishu-reply-message-id`，CLI 应优先发送 `claim_value_answer.feishu_card_payload`。

发送策略应与竞品看板一致：

- `--feishu-card-only` 时 stdout 只输出卡片发送状态。
- 未提供 message_id 时，不尝试发送卡片。
- 卡片发送失败时，返回业务化失败原因，不暴露 stderr、堆栈或命令。

### 7.3 小奥 Skill

小奥 Skill 必须把以下问题路由到 `sku-claim-value --answer-style xiaoao --with-report feishu-doc`：

- “某 SKU 的用户卖点价值是什么”
- “某 SKU 的溢价卖点有哪些”
- “某 SKU 哪些卖点值钱”
- “某 SKU 的卖点支付价值”
- “某个卖点能支撑多少钱”

Skill 不得重写排序和分类，不得把卡片结果改写成通用大模型回答。

## 8. 非目标

- 不修改 M12C 量化算法。
- 不重跑 M00-M12C。
- 不新建独立飞书机器人。
- 不要求聊天卡片承载完整计算过程。
- 不在卡片里展示原始可比池组间差、原始 JSON、表名、字段名和批次号。
- 不把竞品拦截项列入本品正向卖点 Top。
- 不用累计销量判断卖点价值。

## 9. 降级与边界

| 场景 | 降级回答 |
| --- | --- |
| SKU 未找到 | 提示可能有多个或没有匹配 SKU，要求用户二次选择 |
| M12C 缺失 | 提示需要先执行卖点价值量化 |
| 正向卖点为空 | 展示门槛、待激活、竞品拦截和样本限制 |
| 飞书文档创建失败 | 返回短答，提示完整报告暂未生成 |
| 飞书卡片发送失败 | 返回卡片发送失败原因，可降级为短答 |
| 卡片内容过大 | 裁剪 Top 卖点和风险项，完整内容进入报告 |

## 10. 验收标准

以海信 65E7Q 为首个验收样例：

1. 飞书会话能生成用户卖点价值卡片。
2. 卡片顶部能说清正向支付价值集中在哪些卖点。
3. 卡片 Top 表展示 300Hz 高刷、5200nits 高亮、98% 色彩、1920 分区控光等正向结果。
4. HDMI2.1、MiniLED、杜比等门槛或待验证项不得被展示为高溢价 Top。
5. 卡片和报告都不得展示上千元原始可比池价差作为卖点价值。
6. 卡片包含飞书详细报告按钮，链接权限为获得链接的人可阅读。
7. `--format text` 输出 600 字以内业务短答。
8. 单元测试不依赖真实飞书 API，可通过 mock 验证卡片 payload 和发送逻辑。
