# CatForge Analyst 与小奥家电市场分析专家需求

## 1. 背景

M00-M11D 已经把原始量价、参数、卖点、评论处理成事实画像、用户任务、目标客群、价值战场、语义市场图谱和销量分配。

下一步需要把这些确定性结果交给 205 上的 OpenClaw + DeepSeek，形成“小奥家电市场分析专家”。用户只用业务语言提问，不需要知道 M00/M01/Mxx，也不需要直接查数据库。

本需求文档定义两件事：

1. `catforge_analyst` 复合分析 CLI 要提供哪些原子能力和 SOP 编排能力。
2. 小奥 Skill / Agent 如何使用这些能力，既能稳定回答高频 SOP 问题，也能在问题超出预设 SOP 时组合原子能力回答。

## 2. 总体目标

小奥必须能做到：

- 对高频问题，按固定 SOP 调用程序，输出稳定、可复核的分析包。
- 对非高频但可拆解的问题，调用原子能力组合回答。
- 对当前数据不支持的问题，明确边界、缺失数据和可替代分析口径。
- 对所有结论保留 evidence、口径、置信度、样本范围和限制说明。
- 对用户输出必须是专业市场分析语言，不暴露 CLI、Mxx 模块、内部代码、批次号、字段名、JSON、命令或调试错误。

## 3. 非目标

本层不做以下事情：

- 不重新生成 M03B/M04C/M05C/M07/M09C/M10C/M11C/M11D 结果。
- 不直接读取原始四张表做业务判断。
- 不把 LLM 自由推理作为竞品、溢价卖点、销量差异、市场空间的唯一依据。
- 不输出无证据的确定因果。
- 不计算广告 ROI、库存影响、渠道资源投放效果，除非未来引入广告、库存、流量、促销数据。
- 不泄露 API key、数据库密码、环境变量或 205 连接凭据。

## 4. 分层定位

```text
用户自然语言
  -> 小奥 Agent
  -> xiaoao-home-appliance-market-analysis Skill
  -> catforge_analyst CLI
       -> 原子分析能力
       -> SOP 编排能力
  -> catforge_insight / catforge_pipeline
  -> M00-M11D 结果表
```

各层职责：

| 层 | 职责 | 是否允许自由推理 |
| --- | --- | --- |
| M00-M11D | 生成事实、语义画像、图谱和销量分配 | 否 |
| `catforge_insight` | 查询事实、图谱、销量分配 | 否 |
| `catforge_pipeline` | 执行数据准备和画像生成 | 否 |
| `catforge_analyst` 原子能力 | 完成单个稳定分析动作 | 否 |
| `catforge_analyst` SOP | 编排多个原子能力回答高频问题 | 否 |
| 小奥 Skill | 问题分类、路由、约束回答格式 | 少量 |
| 小奥 Agent | 业务语言表达和开放问题拆解 | 是，但必须受 CLI 结果约束 |

## 5. 两级 CLI 能力模型

`catforge_analyst` 必须同时提供两级能力：

### 5.1 原子能力

原子能力是可复用的分析积木，只完成一个明确动作。

| 原子能力 | 作用 |
| --- | --- |
| `resolve-sku` | 把品牌、型号、SKU code、自然语言 query 解析到唯一 SKU 或候选 SKU。 |
| `sku-fact-brief` | 汇总单 SKU 的参数、卖点、评论、市场、任务、客群、战场、销量分配。 |
| `same-size-price-candidates` | 按同尺寸、同价格带、邻近价格带找可比 SKU 候选。 |
| `semantic-overlap` | 计算两个或多个 SKU 的价值战场、用户任务、目标客群重合。 |
| `sales-overlap` | 计算两个 SKU 在重叠在售周内的周均销量、周均销额、价格和平台结构差异。 |
| `param-claim-overlap` | 计算参数、卖点、卖点位置、参数档位的重合和差异。 |
| `comment-support` | 查询评论对任务、客群、战场、参数、卖点的正负向支撑。 |
| `semantic-dimension-space` | 查询某个任务、客群、战场的市场空间、SKU 贡献和品牌/尺寸/价格分布。 |
| `opportunity-gaps` | 查询某 SKU 的机会战场、拖后腿战场、用户观察战场，以及价格/参数/卖点/评论卡点。 |

原子能力必须输出结构化 JSON，并包含：

- `status`
- `input`
- `result`
- `evidence`
- `limitations`
- `source_modules`
- `rule_versions`

### 5.2 SOP 编排能力

SOP 编排能力是高频业务问题的稳定执行流程，内部按顺序调用多个原子能力。

| SOP 命令 | 回答的问题 |
| --- | --- |
| `competitor-set` | 某 SKU 的直接竞品、价格压力竞品、上探/下探替代竞品是谁。 |
| `why-sales-diff` | A 为什么比 B 卖得好或卖得差。 |
| `premium-claim-drivers` | 某 SKU 哪些卖点支撑用户选择，哪些是溢价卖点，哪些拖后腿。 |
| `battlefield-space` | 某个价值战场、用户任务或目标客群空间有多大，哪些 SKU 承接。 |
| `battlefield-opportunity` | 某 SKU 能否通过价格、参数、卖点调整进入更多战场。 |
| `sku-business-brief` | 某 SKU 的综合业务画像和核心分析摘要。 |
| `ask` | 自然语言路由到 SOP 或原子能力，不承担复杂计算。 |

## 6. 高频 SOP 需求

### 6.1 竞品是谁

问题示例：

- “海信 65E7Q 的竞品是谁？”
- “这款电视和谁竞争？”

必须按以下优先级判断：

1. 同尺寸价格。
2. 同价值战场。
3. 同用户任务 / 目标客群。
4. 同参数 / 卖点。
5. 销量接近。

输出必须分为：

- `direct_competitors`：直接竞品。
- `price_pressure_competitors`：更低价但同尺寸/同战场的价格压力竞品。
- `uptrade_alternatives`：高价/高配替代。
- `downgrade_alternatives`：低价/低配替代。
- `excluded_candidates`：被排除的候选及原因。

小奥面向用户回答时，不直接展示内部分组字段或原始分数。竞品问题默认先给三款重点竞品，并用业务角色解释：

- 最直接竞品：同尺寸、同价位、价值战场/用户任务/目标客群重合最高。
- 价格贴身竞品：价格最接近，产品卖点和参数可比。
- 分流竞品：低价下探、同战场强替代或高价上探替代。

回答结构必须是“结论先行 + 判断依据 + 分析过程 + 口径与限制”。不得说“根据 CLI 结果”“SOP 候选顺序”“Mxx 模块返回”。

### 6.2 为什么 A 比 B 卖得好或卖得差

必须使用重叠在售周的周均销量和周均销额。累计销量只能作为展示字段，不参与胜负判断。

必须比较：

- 重叠在售周数。
- 周均销量、周均销额、加权均价。
- 尺寸、价格带、价格差。
- 平台结构。
- 价值战场、用户任务、目标客群重合。
- 参数优势和短板。
- 卖点支撑和缺失。
- 评论正负向支撑。

输出必须区分：

- `likely_reasons`：有证据支持的可能原因。
- `not_supported_reasons`：数据不支持的常见解释。
- `risks`：样本、渠道、时间窗口、评论量不足等风险。

### 6.3 哪些卖点支撑用户选择 / 哪些是溢价卖点

必须结合：

- M04C 卖点事实。
- M05C 评论正负向支持。
- M09C 主/辅用户任务。
- M10C 主/辅目标客群。
- M11C 主/辅价值战场。
- M11D 销量分配权重。
- M07 尺寸内价格带和同池销量表现。

卖点分类：

| 类型 | 需求口径 |
| --- | --- |
| 用户选择卖点 | 与主/辅任务、主/辅客群、主/辅战场有关，且有评论或参数支撑。 |
| 销量支撑卖点 | 对 M11D 分配权重高的任务/客群/战场有支撑。 |
| 溢价卖点 | SKU 处于较高价格位置，卖点支撑高价值战场或高端客群，并有评论/参数/市场验证。 |
| 基础卖点 | 用户需要但同池 SKU 普遍具备，不能单独解释溢价。 |
| 厂家声明卖点 | 卖点存在但评论或参数支撑不足。 |
| 拖后腿卖点 | 用户有需求或厂家声明，但评论负向或参数不足。 |

### 6.4 战场空间

问题示例：

- “家庭护眼舒适战场空间多大？”
- “大屏换新性价比战场有哪些 SKU，支撑多少销量？”

必须使用 M11D 结果，不临时重算。

输出：

- 维度名称、code、analysis_population、market_window。
- 覆盖 SKU 数、分配 SKU 数。
- 估算销量、估算销额、周均销量、周均销额。
- 品牌分布、尺寸/价格分布。
- Top SKU 贡献及 allocation weight。
- 口径说明：这是估算解释，不是真实购买归因。

### 6.5 战场机会

问题示例：

- “海信 65E7Q 能不能进入更多战场？”
- “通过价格、参数、卖点调整能抢到更大的市场吗？”

必须基于：

- 当前主/辅战场。
- `opportunity_battlefield`。
- `user_observed_battlefield`。
- `drag_factor_battlefield`。
- 价格门槛、尺寸门槛。
- 参数缺口。
- 卖点表达缺口。
- 评论负向或未满足需求。

输出必须分为：

- `short_term_actions`：页面卖点、价格、表达方式等短期动作。
- `mid_term_actions`：参数补齐、体验修复等中期动作。
- `high_cost_actions`：产品规格、渠道资源等高成本动作。
- `battlefield_impact`：每个动作对应可能进入或强化的战场。
- `evidence_and_limits`：依据和限制。

### 6.6 SKU 综合业务画像

必须把单 SKU 事实和图谱组合成一页业务摘要：

- 基础信息：品牌、型号、尺寸、价格带。
- 参数画像。
- 卖点事实画像。
- 评论事实画像。
- 市场画像。
- 主/辅用户任务。
- 主/辅目标客群。
- 主/辅价值战场。
- M11D 销量分配。
- 主要竞争池。
- 关键优势、短板、机会、风险。

## 7. 超出预设 SOP 的问题处理

当问题没有命中固定 SOP，小奥不能直接自由编结论，必须按以下流程处理：

1. 判断问题是否能拆成已有原子能力。
2. 若能拆，输出内部分析计划并调用原子能力。
3. 汇总原子能力结果，形成证据链。
4. 若部分问题无法回答，明确写出无法回答的部分和原因。
5. 如果完全缺少必要数据，给出可替代分析口径。
6. 如果该类问题高频出现，后续沉淀为新的 SOP 命令。

开放问题示例：

| 用户问题 | 处理方式 |
| --- | --- |
| “海信在 65 寸高端画质市场表现如何？” | 组合 `semantic-dimension-space`、`same-size-price-candidates`、品牌过滤和 M11D 贡献。 |
| “TCL 在大屏换新战场里谁最强？” | 查询战场空间和 Top SKU，再按品牌过滤。 |
| “65 寸高价带里，游戏体育和高端画质哪个更有空间？” | 分别查询两个战场/任务空间，限制尺寸和价格带后比较。 |
| “投 100 万广告能多卖多少？” | 当前无广告/流量/转化数据，不能直接测算；可提供战场空间和假设情景。 |

## 8. 边界回答需求

当数据不支持时必须明确说明：

- 缺少什么数据。
- 当前可用数据能回答什么。
- 当前结论的边界是什么。
- 是否可以用假设情景替代。

典型边界：

| 问题类型 | 当前边界 |
| --- | --- |
| 广告 ROI | 缺广告花费、曝光、点击、转化链路。 |
| 库存影响 | 缺库存、断货、发货能力数据。 |
| 促销影响 | 缺活动日历、券补贴、实际成交价拆解。 |
| 线下市场 | 当前主要是线上渠道数据，不能推出线下结论。 |
| 真实购买因果 | M11D 是销量解释，不是真实消费者归因。 |

## 9. Skill 需求

小奥 Skill 必须包含：

- 工具目录：`catforge_pipeline`、`catforge_insight`、`catforge_analyst`。
- 问题分类规则。
- SOP 路由规则。
- 原子能力组合规则。
- 边界回答规则。
- 输出格式要求。
- 禁止事项。

Skill 必须明确：

- 常见问题优先调用 SOP 命令。
- 非常见问题先拆解为原子能力。
- 不允许直接绕过 CLI 查数据库生成业务结论。
- 不允许把累计销量作为销量胜负判断。
- 不允许把估算解释说成确定因果。

## 10. Agent 需求

小奥 Agent 必须面向业务用户，不暴露内部模块术语，除非用户主动问实现细节。

Agent 回答必须包含：

- 结论。
- 关键证据。
- 解释逻辑。
- 数据口径。
- 限制和不确定性。

Agent 不应输出：

- 长 SQL。
- API key、密码、环境变量。
- 未经 CLI 支撑的确定性结论。
- “一定”“必然”“证明因果”等过度表述。

## 11. 验收问题

固定 SOP 验收以海信 65 寸 SKU 为主，例如 `65E7Q`：

1. “海信 65E7Q 的直接竞品是谁？”
2. “海信 65E7Q 为什么比某个竞品卖得好或卖得差？”
3. “海信 65E7Q 哪些卖点支撑用户选择？”
4. “海信 65E7Q 哪些卖点是溢价卖点？”
5. “海信 65E7Q 在哪些价值战场里，销量怎么切？”
6. “海信 65E7Q 能不能通过价格、参数、卖点调整进入更多战场？”
7. “海信 65E7Q 所在价值战场空间多大？”

开放问题验收：

1. “海信在 65 寸高端画质市场表现如何？”
2. “TCL 在大屏换新战场里哪些 SKU 最强？”
3. “65 寸高价带里，游戏体育和高端画质哪个更有空间？”
4. “预算 3000 元左右买 65 寸电视，主要竞品怎么分？”
5. “如果没有广告数据，能不能评估投放后销量增长？”

验收标准：

- 命中固定 SOP 时，必须调用对应 SOP 命令。
- 未命中固定 SOP 时，必须列出使用的原子能力。
- 输出必须包含数据口径和限制。
- 不得使用累计销量判断输赢。
- 不得编造当前数据没有的广告、库存、促销结论。

3. 实现 `same-size-price-candidates`、`semantic-overlap`、`sales-overlap`。
4. 实现 `param-claim-overlap`、`comment-support`、`opportunity-gaps`。
5. 实现 `competitor-set`、`sku-business-brief`。
6. 实现 `why-sales-diff`、`premium-claim-drivers`。
7. 实现 `battlefield-space`、`battlefield-opportunity`。
8. 实现 `ask` 路由。
9. 编写小奥 Skill。
10. 编写小奥 Agent `AGENTS.md`。
11. 安装到 205 OpenClaw。
12. 用固定 SOP 问题和开放问题验收。
