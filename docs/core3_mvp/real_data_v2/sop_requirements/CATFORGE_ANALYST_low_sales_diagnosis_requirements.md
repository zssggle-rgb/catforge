# CatForge Analyst 单品低销量诊断需求

## 1. 背景

CatForge 当前已经能通过 `catforge_analyst` 回答竞品是谁、A 为什么比 B 卖得好或差、哪些卖点支撑溢价、某个价值战场空间多大、某个 SKU 能否进入更多战场等问题。

但业务用户经常不会先指定竞品，而是直接问：

- “这款电视为什么卖不好？”
- “这个 SKU 问题出在哪里？”
- “销量低应该怎么改？”
- “如果按照价值棒理论看，它是价格问题还是价值问题？”

这类问题需要一个新的 SOP 编排能力，把现有原子能力和 SOP 结果整理成单品诊断报告。该能力不重新生成事实层，不直接读取原始表，不把 LLM 自由推理作为原因判断依据。

## 2. 能力名称

建议新增 SOP：

```text
low-sales-diagnosis
```

中文业务名：

```text
单品低销量诊断
```

该能力属于 `catforge_analyst` 专业市场分析层。它不是新的 Mxx 数据生产模块，而是只读编排层。

对应详细设计见：[CatForge Analyst 单品低销量诊断详细设计](../sop_detailed_design/CATFORGE_ANALYST_low_sales_diagnosis_design.md)。

## 3. 总体目标

`low-sales-diagnosis` 要回答七件事：

1. 这款 SKU 是否真的在可比市场中卖得弱。
2. 低销量是“没有进入用户选择集”，还是“进入选择集后被替代品打掉”。首版没有曝光、搜索、流量数据，因此只能在市场成交口径下判断“被替代品打掉”，不能判断曝光不足。
3. 本品用户愿付价值是否被具体购买任务、目标客群、价值战场和可验证卖点抬高。
4. 当前价格是否拿走了超过用户感知价值的空间，以及这是应该降价、加权益、补价值证明，还是重设定位。
5. 哪些外部竞品提供了更高用户剩余价值：更强购买锚点、更低价格、更低选择风险，或三者组合。
6. 是否存在同品牌/同尺寸/更低价 SKU 分流，导致本 SKU 的价格架构和产品线定位被挤压。
7. 对应应该采取哪些验证动作和短期、中期、高成本动作。

输出必须面向业务用户，不暴露 CLI、Mxx 模块、内部表名、字段名、SQL、JSON 过程或调试错误。

## 4. 价值棒低销量归因口径

本能力使用价值棒理论作为业务解释框架，但所有判断都必须由 CatForge 已生成证据支撑。首版只考虑现有市场数据，不考虑企业侧成本、广告、库存、渠道和毛利。

### 4.1 核心公式

低销量不是先问“竞品是否卖得好”，而是问用户在同一购买场景中为什么没有选择本品。

```text
用户选择概率 ≈ 本品用户剩余价值 相对 竞品用户剩余价值 的优势

本品用户剩余价值 = 本品愿付价值 WTP - 本品实际支付价格 P - 风险/麻烦/不确定性成本
竞品用户剩余价值 = 竞品愿付价值 WTP - 竞品实际支付价格 P - 风险/麻烦/不确定性成本
```

### 4.2 低销量归因体系

| 归因层 | 核心问题 | 业务含义 | CatForge 证据 |
| --- | --- | --- | --- |
| 是否真的弱 | 本品在可比成交口径下是否弱 | 不能因为用户说“卖不好”就默认低销量 | M07 市场画像、`why-sales-diff` 重叠在售周销量/销额 |
| 愿付价值 WTP | 用户为什么愿意为本品多付钱或优先选本品 | 购买任务、目标客群、价值战场和卖点证据是否清晰 | M09C、M10C、M11C、M11D、M12C、M05C |
| 价格 P | 当前价格是否拿走太多价值 | 价格高不等于必须降价，要区分有效价格问题和价值证明问题 | M07 价格分位、价差、同池低价竞品表现 |
| 竞品替代 | 谁给了更高用户剩余价值 | 竞品可能通过具体购买锚点、更低价格或更低风险拦截本品 | `competitor-set`、`why-sales-diff`、`claim-value-compare` |
| 同品牌产品线分流 | 自家更低价 SKU 是否截流 | 用户想买本品牌时是否下探到更便宜款 | same-size-price candidates、同品牌同尺寸销量/价格 |
| 风险与不确定性 | 用户是否不信、看不懂或担心体验 | 评论、参数、证据缺口会降低 WTP | M05C、M03B、M04C、M12C 降级原因 |

### 4.3 原因不是事实，必须给出机制

以下表达不合格：

- “竞品卖得好。”
- “价格高。”
- “卖点不足。”
- “愿付价值不足。”

合格表达必须包含：

1. **事实证据**：谁、哪个价格/销量/卖点/参数口径。
2. **真正原因**：用户为什么因此不选本品。
3. **价值棒机制**：影响 WTP、P、替代选择，还是风险成本。
4. **决策含义**：降价测试、权益补偿、补证据、改战场，还是调整产品线。
5. **验证动作**：需要什么下一步市场或内容测试验证。

### 4.4 M12C 卖点标签口径

`claim-value-compare` 中的 `claim_value_role` 和 `business_value_label` 是 M12C 的**市场观测角色**，不是卖点天然层级。低销量诊断不得把它直接写成“强卖点”。

必须区分两层：

| 层级 | 字段 | 可表达为 | 禁止表达为 |
| --- | --- | --- | --- |
| 市场观测 | `claim_value_role`、`business_value_label` | 市场观测：溢价相关 / 销量相关 / 基础门槛 / 样本不足 | 该卖点本身是强卖点 |
| 参数/事实口径 | `parameter_competitiveness.overall_parameter_competitiveness_level_cn`、`downgrade_reason_cn` | 参数口径：领先优势 / 较强优势 / 基础门槛 / 弱或缺失 / 样本不足 | 忽略基础门槛和降级原因 |

示例：

- “影音认证锚点：市场观测为销量相关，但参数口径为基础门槛，不能单独解释为强卖点。”
- “画质锚点：市场观测为溢价相关，参数口径为领先优势，可以作为高价价值锚点候选。”

### 4.5 因果边界

业务表达中允许说“更可能”“当前证据显示”“可观测结果支持”，不得说“证明因果”“必然导致”。补齐某个竞品卖点也不得承诺销量追平竞品，因为销量还受价格、品牌、流量、促销、库存、评价数量和产品线分流影响。

## 5. 非目标

本能力不做以下事情：

1. 不重新执行 M03B/M04C/M05C/M07/M09C/M10C/M11C/M11D/M12C。
2. 不直接从 `week_sales_data`、`attribute_data`、`selling_points_data`、`comment_data` 原始表生成业务结论。
3. 不判断广告投放、库存、促销、导购、线下渠道、毛利和 BOM 原因，除非未来引入对应数据。
4. 不把累计销量作为“卖得好或卖得差”的判断依据。
5. 不把样本不足或缺失值当作负向结论。
6. 不输出无证据的动作建议。

## 7. 数据前置条件

首版 TV 低销量诊断要求目标 SKU 至少具备以下结果：

| 上游结果 | 用途 | 缺失时处理 |
| --- | --- | --- |
| M07 市场画像 | 判断价格、销量、同尺寸池位置 | 缺失则不能判断卖得弱，只能提示补跑 M07 |
| M03B 参数画像 | 判断参数缺口和硬能力 | 缺失则参数类原因降级为未知 |
| M04C 卖点事实画像 | 判断厂家卖点和事实卖点 | 缺失则卖点类原因降级为未知 |
| M05C 评论事实画像 | 判断用户感知和负向体验 | 缺失则不能强判用户愿付价值，只能提示评论验证不足 |
| M09C 用户任务画像 | 判断主任务和拖后腿任务 | 缺失则任务错位类原因降级 |
| M10C 目标客群画像 | 判断主客群和未满足客群 | 缺失则客群错位类原因降级 |
| M11C 价值战场画像 | 判断主战场、机会战场、拖后腿战场 | 缺失则不能输出战场诊断 |
| M11D 语义市场图谱 | 判断战场空间和 SKU 分配 | 缺失则不能判断战场空间影响 |
| M12C 卖点价值量化 | 判断市场观测角色、参数/事实口径、待激活、竞品拦截和价格压力卖点 | 缺失则卖点价值诊断降级为事实卖点和评论支撑判断 |

## 8. 触发问题

自然语言命中以下意图时，应路由到 `low-sales-diagnosis`：

- “为什么卖不好”
- “为什么卖不动”
- “销量低是什么原因”
- “这款电视问题在哪”
- “怎么提升销量”
- “给这款 SKU 做诊断”
- “按照价值棒理论看问题在哪”
- “价格是不是太高”
- “是不是卖点不行”

如果用户同时给出两个 SKU，例如“TV00029112 为什么比 TV00030001 卖得差”，应优先路由到 `why-sales-diff`，因为用户已经指定对比对象。

## 9. 诊断主流程

`low-sales-diagnosis` 必须按以下顺序执行：

1. 解析目标 SKU。
2. 拉取单 SKU 事实画像。
3. 找同尺寸、同尺寸内价格带或邻近价格带候选。
4. 生成重点竞品集合。
5. 对重点竞品逐个执行销量差异对比。
6. 判断目标 SKU 在可比池中是否真的卖得弱。
7. 对重点竞品逐个执行卖点价值对比，抽取“竞品已成立、本品未形成正向对照”的具体购买锚点。
8. 拉取卖点支付价值和卖点机会缺口。
9. 拉取战场机会、拖后腿战场、价格/参数/卖点/评论缺口。
10. 识别同品牌、同尺寸、更低价或相邻价格带 SKU 的产品线分流信号。
11. 生成低销量原因排序。
12. 生成短期、中期和高成本动作建议。
13. 输出结构化 JSON、短答案和可选报告。

## 10. “卖得弱”的判断口径

低销量诊断必须先判断目标是否真的弱。不能因为用户问“卖不好”就默认它卖不好。

### 10.1 首选口径

使用同购买池重点竞品的重叠在售周周均销量和周均销额。

对 Top 3 重点竞品计算：

```text
target_vs_competitor_sales_ratio = target_avg_weekly_sales / competitor_avg_weekly_sales
target_vs_competitor_amount_ratio = target_avg_weekly_amount / competitor_avg_weekly_amount
```

### 10.2 判断状态

| 状态 | 判断条件 |
| --- | --- |
| `weak` | 多数重点竞品在重叠在售周的周均销量或周均销额明显高于目标，且目标没有明显高价销额承接 |
| `not_weak` | 目标周均销量或销额不低于主要竞品，或高价下销额承接稳定 |
| `mixed` | 目标对部分竞品弱、对部分竞品不弱 |
| `uncertain` | 可比竞品不足、重叠周不足或市场画像缺失 |

### 10.3 禁止口径

不得使用累计销量直接判断卖得弱。累计销量只可作为展示字段。

## 11. 原因分类

原因输出必须归入以下类型之一。原因必须是“事实证据 + 机制解释”，不得只是结论标签。

### 11.1 关键购买锚点被竞品占位

成立信号：

- `claim-value-compare` 中存在 `candidate_advantage_claims`。
- 该竞品在重叠在售周销量或销额强于目标。
- 候选卖点具备市场观测信号，并且参数口径不是纯样本不足。

业务解释：

```text
竞品不是因为“卖得好”才成为原因，而是它在若干具体购买锚点上形成了已成立信号，本品没有同等清晰的正向对照，用户无法理解为什么选本品。
```

输出必须列出：

- 竞品名称。
- 具体卖点名称。
- 市场观测：溢价相关 / 销量相关 / 基础门槛 / 样本不足。
- 参数口径：领先优势 / 较强优势 / 基础门槛 / 弱或缺失 / 样本不足。
- 降级原因，尤其是“基础门槛”“不能单独支撑溢价”等。

示例：

```text
重点竞品 A 在某个画质锚点上有溢价相关市场观测，参数口径为领先优势；另一个影音认证卖点虽然有销量相关市场观测，但参数口径为基础门槛，不能单独当作强卖点。
```

### 11.2 价格价值不匹配

成立信号：

- 同尺寸价格分位高但销量分位不高。
- M12C 出现 `价格压力卖点`。
- 竞品价格更低但保留目标 SKU 的核心成交理由。

业务解释：

```text
本品当前价格拿走了较多用户价值，但卖点、评论或市场承接没有充分支撑，可能削弱转化。
```

### 11.3 核心愿付价值不足

成立信号：

- 能够抬高 WTP 的购买锚点数量少，或只停留在基础门槛卖点。
- 有市场观测信号的卖点缺少参数/评论/战场支撑，不能稳定解释更高价格。
- 主价值战场中缺少可稳定解释价格或销量的卖点。
- 主任务、主客群和主战场弱或缺失。

业务解释：

```text
用户在这个购买池里没有看到足够明确的支付理由，产品进入候选清单后缺少让用户选择本品的核心价值。
```

### 11.4 卖点未激活

成立信号：

- M12C 中待激活卖点多。
- 厂家主张卖点多，但评论和市场验证弱。
- 参数事实存在，但 M04C 表达弱或 M05C 用户感知不足。

业务解释：

```text
产品能力可能存在，但用户还没有形成稳定感知，当前卖点没有转化为支付理由或选择理由。
```

### 11.5 外部竞品替代截流

成立信号：

- Top 竞品在同池有更强销量或销额验证。
- 竞品具备本品缺失或弱表达的正向卖点。
- 竞品在相同主战场、主任务或主客群中替代压力高。

业务解释：

```text
用户仍然有需求，但同购买池竞品用更清晰或更有性价比的价值锚点拦截了本品。
```

### 11.6 战场客群错位

成立信号：

- 无主价值战场、无主用户任务或无主目标客群。
- 存在拖后腿战场、拖后腿任务、未满足客群。
- 机会战场空间较大，但本品只停留在机会或用户观察状态。

业务解释：

```text
本品并非没有价值，而是当前价值表达和用户实际需求没有稳定对齐。
```

### 11.7 同品牌产品线分流

成立信号：

- 同品牌、同尺寸、相邻或更低价格带 SKU 周均销量明显高于目标。
- 目标 SKU 价格更高，但没有形成可抵消价差的明确购买锚点。
- 用户如果买该品牌，存在更便宜、销量更强的替代选择。

业务解释：

```text
本品不只是被外部竞品打掉，也可能被自家产品线中更便宜、更容易理解的 SKU 分流。
```

该原因在首版可作为市场数据洞察输出；若缺少同品牌 SKU，必须降级为未知，不得硬判。

### 11.8 用户体验或证据风险

成立信号：

- 评论负向事实较多。
- 评论事实句不足或置信度低。
- 参数冲突、参数未知较多。
- 样本不足或可比池不足。

业务解释：

```text
当前证据不足或存在体验负向，不能把相关卖点直接作为提价或转化理由。
```

## 12. 原因排序要求

原因排序必须同时考虑：

1. 目标是否真的低销量。
2. 该原因的证据强度。
3. 该原因与价值棒的关系。
4. 该原因能否被业务动作影响。
5. 该原因是否有竞品对照。

输出字段：

| 字段 | 说明 |
| --- | --- |
| `rank` | 原因排序 |
| `reason_type` | 原因分类 |
| `severity` | `high`、`medium`、`low`、`unknown` |
| `evidence_strength` | 0 到 1 |
| `summary_cn` | 面向业务的原因摘要 |
| `observation_points` | 事实证据，列出具体竞品、价差、销量、卖点或参数口径 |
| `root_cause_cn` | 真正原因，不得只复述“竞品强/价格高/卖点不足” |
| `value_stick_mechanism_cn` | 该原因如何影响 WTP、P、替代选择或风险成本 |
| `decision_implication_cn` | 业务决策含义，例如降价测试、补证据、改战场、避开正面竞争 |
| `validation_cn` | 下一步验证动作 |
| `value_stick_position` | `customer_wtp`、`price_capture`、`competitor_alternative`、`evidence_quality` |
| `evidence_refs` | 证据摘要，不暴露内部字段 |
| `recommended_action_keys` | 关联动作 |

## 13. 动作建议

动作必须按成本和周期分层。

### 13.1 短期动作

适用于不改产品规格的动作：

- 价格带校准。
- 券后价、权益包或套装测试。
- 详情页卖点重排。
- 直播/导购话术调整。
- 用用户场景重写参数卖点。
- 针对竞品拦截点补充对比表达。

### 13.2 中期动作

适用于需要一段时间积累证据或修体验的动作：

- 补充评测、样张、体验证明。
- 针对负向评论修复体验。
- 针对待激活卖点补充内容教育。
- 重构主战场和主客群表达。
- 优先收集缺失评论和参数证据。

### 13.3 高成本动作

适用于下一代产品或产品线策略：

- 参数规格补齐。
- SKU 定位重设。
- 产品线价格梯度调整。
- 渠道资源重配。
- 高价形象款和走量款拆分。

每条动作都必须包含：

- `action_type`
- `priority`
- `cost_level`
- `time_horizon`
- `summary_cn`
- `why_cn`
- `expected_impact_cn`
- `evidence_refs`
- `not_supported_note_cn`

## 15. 短答案要求

`--format text --answer-style xiaoao` 输出必须是业务短答案。

结构：

1. 第一句先说明是否真的卖得弱。
2. 第二段给 2 到 3 个主要原因。
3. 第三段给 2 到 3 个优先动作。
4. 最后说明数据边界。

示例：

```text
这款电视在当前可比线上样本中属于相对偏弱，但不是单纯低价竞争问题。
主要压力来自三点：第一，价格处在同尺寸较高位置，但可被用户明确感知的溢价卖点不足；第二，同购买池竞品在高端画质和家庭观影场景上形成拦截；第三，部分卖点停留在厂家表达，评论和市场承接还不够。
优先建议先做价格带校准和权益包测试，同时把详情页主卖点聚焦到已被评论验证的画质、影音或大屏场景；中期再补齐竞品拦截卖点的参数证据和用户体验内容。
当前无法判断广告、库存、促销和毛利原因，因为数据中没有对应链路。
```

## 16. 报告要求

当 `with_report=markdown` 或 `feishu-doc` 时，报告必须包含：

1. 结论摘要。
2. 销量是否真的弱。
3. 价值棒拆解。
4. 重点竞品拦截证据。
5. 卖点支付价值和价格压力。
6. 战场、任务、客群缺口。
7. 动作建议。
8. 数据口径和限制。

完整报告可以展示内部业务 code，但不得展示 SQL、表名、数据库凭据、堆栈或调试日志。

## 17. 降级规则

| 场景 | 输出要求 |
| --- | --- |
| SKU 解析到多个候选 | 返回 `ambiguous`，要求用户选择 SKU |
| 找不到 SKU | 返回 `not_found` |
| 缺 M07 | 不能判断销量强弱，提示先生成市场画像 |
| 缺 M12C | 不输出确定的溢价相关市场观测或价格压力分类，降级为事实卖点和评论支撑 |
| 缺评论 | 不强判用户愿付价值，提示评论验证不足 |
| 可比竞品不足 | `sales_status=uncertain`，仍可输出画像和缺口 |
| 目标并不弱 | `sales_status=not_weak`，输出“不是低销量问题”的解释和优化建议 |

## 18. 小奥 Skill 要求

小奥识别低销量诊断意图后：

1. 优先调用 `catforge_analyst low-sales-diagnosis`。
2. 不自行从 JSON 重新排序原因。
3. 如果 CLI 返回 `short_answer`，优先原样使用。
4. 如果返回多 SKU 候选，要求用户选择。
5. 如果数据不支持，必须解释缺什么数据和当前可替代口径。
6. 不得把缺失数据的问题编造成业务原因。

## 19. 验收问题

首版以 TV SKU 验收：

1. “TV00029112 为什么卖不好？”
2. “海信 65E7Q 销量低是什么原因？”
3. “这款电视是不是价格太高？”
4. “按照价值棒理论看 65E7Q 问题在哪？”
5. “这个 SKU 怎么提升销量？”
6. “TV00029112 为什么比 TV00030001 卖得差？”应路由到 `why-sales-diff`，不是 `low-sales-diagnosis`。

## 20. 验收标准

1. 命中单 SKU 低销量问题时，路由到 `low-sales-diagnosis`。
2. 命中双 SKU 销量差问题时，路由到 `why-sales-diff`。
3. 如果目标 SKU 实际不弱，不能硬输出低销量原因。
4. 原因排序必须至少引用价格、竞品、卖点、战场或评论中的一种证据。
5. 动作建议必须分短期、中期、高成本。
6. 输出必须明确广告、库存、促销、毛利等不支持项。
7. 测试不得调用外部 LLM。
8. `--format text` 必须输出可直接给业务用户看的中文短答。
9. 竞品卖点差距必须同时展示市场观测角色和参数/事实口径，不能把 M12C 标签直接写成“强溢价卖点”或“强销量卖点”。
10. 原因必须包含“事实证据、真正原因、价值棒机制、决策含义、验证动作”，不能只输出“竞品强、价格高、愿付价值不足”等结论。
11. 不得承诺“卖点比过竞品即可销量追平”，必须说明价格、品牌、流量、促销、库存、评价数量和产品线分流等未覆盖因素。
