# M08.4 评论原生业务维度发现 SOP 需求

## 0. 文档定位

本文是 CatForge 彩电核心三竞品真实数据 MVP 的新增需求文档，位于 M08 SKU 综合信号画像之后、M08.5 业务维度本体融合发布之前。

当前只定义需求，不写详细设计，不拆开发任务，不开发代码。

新增 M08.4 的原因是 205 真实数据验证发现：现有 M09 用户任务、M10 目标客群、M11 价值战场大量沿用预设维度做全量候选，导致 84 个 SKU 几乎都命中所有任务、客群和战场。这个结果可以数学平账，但业务上没有区分度，不能支撑核心三竞品推导。

M08.4 必须先从真实评论中发现“用户真实在表达什么”，再和预设任务、客群、价值战场做对照。它不能先用预设名称套评论，也不能因为 seed 里有某个维度就强行保留该维度。

## 1. 模块目标

M08.4 的目标是生成“评论原生业务维度候选”和“预设维度对齐建议”，为后续 M08.5 生成当前项目可用的业务维度本体提供依据。

本模块要回答五个问题：

1. 真实评论里自然出现了哪些人群、场景、购买动机、产品体验、服务体验和风险线索。
2. 这些原生线索能否组合成用户任务、目标客群、产品价值战场、服务语境、风险语境或购买动机。
3. 哪些原生维度与现有 seed 维度一致，哪些需要改名、拆分、合并、降级、废弃或新增。
4. 哪些线索只是通用背景、服务履约或低价值泛化好评，不能进入产品战场和销量分配。
5. 哪些维度具备足够证据，可进入后续 SKU 分类；哪些只能作为候选或复核提示。

M08.4 不生成单个 SKU 的最终主任务、主客群、主战场，也不做销量分配。它只生成“当前数据告诉我们应该怎样定义维度”的候选和建议。

## 2. 设计依据

本模块依据：

- `cankao/CatForge_竞品生成SOP_详细指导_v1.md` 中任务、客群、价值战场应由卖点、参数、评论和市场共同推导的要求。
- `cankao/CatForge_核心竞品展示页_UI设计规范_v1.md` 对业务高层展示“先结论、再证据、再推导”的要求。
- `docs/core3_mvp/real_data_v2/validation/comment_native_dimension_alignment_205_20260615.md` 的 205 真实评论原生线索分析。
- M05 评论基础证据层、M06 评论下游信号、M08 SKU 综合信号画像的已生成数据。
- 数据分层原则：下游模块不能绕过上游产物直接读取原始表做业务判断。

## 3. 本模块解决什么

### 3.1 从真实评论发现维度

M08.4 必须从 M05/M06 的可下游消费评论中提取原生语义，而不是直接从 seed 维度出发。

原生语义包括：

| 原生类型 | 示例 | 可能支撑的后续维度 |
| --- | --- | --- |
| 人群 `person` | 老人、父母、孩子、家人、单位、公司 | 目标客群、购买场景 |
| 场景 `scene` | 客厅、卧室、新家、挂墙、宿舍、看球、游戏 | 用户任务、目标客群、价值战场 |
| 动机 `motive` | 换新、国补、性价比、预算、送礼、装修 | 用户任务、购买动机、价格价值 |
| 产品体验 `product_experience` | 画质、音质、高刷、护眼、语音、系统、投屏、大屏 | 产品价值战场、卖点价值 |
| 服务体验 `service_experience` | 安装、配送、售后、师傅、退换、物流 | 服务语境、履约风险 |
| 风险 `risk` | 广告、卡顿、坏点、做工、破损、噪音 | 风险语境、复核线索 |

### 3.2 从原生语义形成候选维度

M08.4 要把低层线索组合成可评审的候选维度：

| 候选维度类型 | 定义 | 典型形成逻辑 |
| --- | --- | --- |
| 原生用户任务 `native_task` | 用户买这台电视主要要完成什么任务 | 场景 + 动作/动机 + 产品体验 |
| 原生目标客群 `native_target_group` | 谁在买、为什么是这类人买 | 人群 + 场景/任务 + 价格或产品证据 |
| 原生产品价值战场 `native_product_value_battlefield` | SKU 靠什么产品价值参与竞争 | 产品体验 + 参数/卖点锚点 + 市场支撑 |
| 服务语境 `service_context` | 安装、配送、售后等履约体验 | 服务体验评论聚合 |
| 风险语境 `risk_context` | 影响购买和体验的风险点 | 负向评论、低分原因、质量问题 |
| 购买动机 `purchase_motive` | 价格、补贴、换新、装修等购买驱动 | 动机词 + 价格/渠道/评论证据 |

MVP 至少要落地 `native_task`、`native_target_group`、`native_product_value_battlefield`、`service_context`。`risk_context` 和 `purchase_motive` 可以先作为候选输出，但不能丢失。

### 3.3 生成预设维度对齐建议

M08.4 要把评论原生候选与 seed 里的任务、客群、价值战场做对照，并输出建议：

| 对齐动作 | 含义 | 示例 |
| --- | --- | --- |
| `keep` | 原生证据与预设定义基本一致 | 大屏性价比有价格价值和大屏证据支撑 |
| `rename` | 原生表达和预设名称有偏差，需要改名 | “家庭观影升级”可能更接近“客厅影音体验” |
| `split` | 一个预设维度混入多个不同原生线索 | “游戏体育”在任务层应拆游戏、看球/体育 |
| `merge` | 多个预设维度在真实评论中无法区分 | 两个过近的泛化家庭维度可合并 |
| `downgrade` | 预设维度不应作为产品维度，只能做服务、风险或背景 | 服务保障战场降级为服务语境 |
| `deprecate` | 当前数据没有足够证据支撑，暂不启用 | 覆盖低、无产品锚点、无市场支撑的维度 |
| `new` | 评论里出现稳定新线索，seed 没有覆盖 | 单位/公司采购、国补换新驱动 |

对齐建议不是最终发布结果。M08.5 才负责融合 seed、原生维度和复核意见，形成当前项目的正式本体版本。

## 4. 本模块不做什么

| 不做事项 | 原因 | 后续处理 |
| --- | --- | --- |
| 不生成 SKU 主任务、主客群、主战场 | 本模块定义维度，不给 SKU 下最终结论 | M09/M10/M11 |
| 不做销量分配 | 销量权重要基于已发布维度和 SKU 画像 | M11.6 |
| 不做销量交叉校验 | 这是对画像和权重的检查 | M11.7 |
| 不改 seed 文件 | seed 是基础资产，本模块只输出项目级建议 | M08.5/规则治理 |
| 不直接读取原始评论表 | 必须使用清洗、去重、低价值过滤后的评论句 | M05/M06 |
| 不把服务履约当产品战场 | 服务是体验和风险，不是产品核心价值 | 服务语境/风险语境 |
| 不把泛化好评当维度证据 | “不错、很好、满意”不能说明任务、客群或战场 | 低价值过滤/弱证据 |

## 7. 处理逻辑需求

### 7.1 数据读取

M08.4 只能读取已经清洗和过滤后的上游产物：

1. 从 M05 读取 `usable_for_downstream=true` 且 `low_value_flag=false` 的评论句。
2. 从 M06 读取评论主题、情感、强弱线索和服务/价格/痛点等标签。
3. 从 M08 读取 SKU 的参数、卖点、评论、市场综合画像。
4. 从 seed 读取预设维度，仅用于后置对齐，不用于前置聚类。

低价值评论、重复模板评论、只有“好评、不错、满意、值得”这类泛化表达的评论不能进入维度发现主流程。

### 7.2 原生线索抽取

每条评论句至少要判断：

- 是否包含人群线索。
- 是否包含使用场景。
- 是否包含购买动机。
- 是否包含产品体验。
- 是否包含服务体验。
- 是否包含风险或负向体验。

同一句评论可以有多个线索，但必须保留线索类型和证据来源。例如：

- “给爸妈买的，语音操作方便”应抽出 `person=长辈/父母`、`product_experience=语音易用`。
- “客厅看电影画质不错”应抽出 `scene=客厅/观影`、`product_experience=画质`。
- “师傅安装很快”只能进入 `service_experience=安装服务`，不能支撑产品价值战场。

### 7.3 原生维度聚合

候选维度不能只靠关键词出现。必须综合：

1. 评论线索的共现关系。
2. 覆盖 SKU 数和句数。
3. 强证据句比例。
4. 是否有参数或卖点锚点。
5. 是否有市场销量、价格、渠道支撑。
6. 是否与其他候选维度明显重叠。
7. 是否被服务履约或低价值泛化好评污染。

如果某个候选覆盖绝大多数 SKU，但缺乏区分度，应标记为 `over_broad_dimension`，不能直接进入后续 SKU 分类。

### 7.4 维度命名

原生维度名称必须来自真实评论表达归纳，不能默认复用 seed 名称。

命名规则：

- 优先使用评论中高频、业务可理解的表达。
- 名称要能区分相邻维度。
- 名称不能把服务语境包装成产品战场。
- 名称不能过度营销化，必须能被证据解释。

例如：

- 如果评论主要是“国补、优惠、价格合适、值得”，候选名称应偏向“价格价值/补贴换新驱动”，不能直接写成“大屏性价比战场”。
- 如果评论主要是“师傅、安装、送货”，候选名称应是“安装配送服务体验”，不能写成“服务保障战场”。

### 7.5 与预设维度对齐

对齐必须发生在原生维度生成之后。

对齐时要分别比较：

| 比较项 | 说明 |
| --- | --- |
| 名称相似 | 名称是否接近，但不能只看名称 |
| 定义相似 | 业务含义是否一致 |
| 证据结构 | 评论、参数、卖点、市场支撑是否一致 |
| SKU 覆盖 | 是否覆盖同一批 SKU |
| 排除规则 | 是否有相同边界 |
| 服务污染 | 是否混入服务履约 |
| 区分度 | 是否能把 SKU 分开 |

如果名称相同但证据结构不同，应输出 `rename` 或 `split`，不能输出 `keep`。

### 7.6 可进入后续分类的条件

一个原生维度要进入后续 M09/M10/M11 分类候选，至少满足：

1. 有足够评论证据，且不是低价值泛化好评。
2. 有明确业务定义。
3. 有包含规则和排除规则。
4. 能与相邻维度区分。
5. 产品价值战场必须有产品锚点。
6. 服务语境不能进入产品价值战场。
7. 对于目标客群，不能只有“家里、家人”等泛化词，必须有更明确的人群、场景或购买动机。

## 8. 205 真实数据约束

根据 205 当前批次 `m00_20260613004311_d548f6dc` 的验证报告，M08.4 首版必须能处理以下真实基线：

| 指标 | 数值 |
| --- | ---: |
| SKU 数 | 84 |
| 原始评论行 | 32,454 |
| 去重评论单元 | 32,454 |
| 句级评论数 | 68,257 |

## 9. 与其他模块关系

### 9.1 与 M08 的关系

M08 是 SKU 综合信号画像，提供参数、卖点、评论、市场的统一 SKU 入口。

M08.4 在 M08 之后运行，用 M08 的画像判断评论原生维度是否有参数、卖点和市场支撑。M08.4 不反向修改 M08。

### 9.2 与 M08.5 的关系

M08.4 负责“发现和建议”：

- 发现评论原生线索。
- 生成原生业务维度候选。
- 生成 seed 对齐建议。
- 输出复核问题。

M08.5 负责“融合和发布”：

- 消费 M08.4 原生候选。
- 结合 seed 和复核结果。
- 生成当前项目 active 的任务、客群、价值战场定义版本。
- 为 M09/M10/M11 提供正式可消费的本体版本。

因此，后续 M08.5 的详细设计需要调整为“原生发现结果 + seed + 复核意见”的融合发布模块，而不是 seed-first 的简单校准模块。

### 9.3 与 M09/M10/M11 的关系

M09/M10/M11 不应再直接遍历 seed 维度给所有 SKU 打分。

它们应消费 M08.5 发布后的本体版本，并使用 M08.4/M08.5 定义的：

- 包含规则。
- 排除规则。
- 必需证据。
- 可选证据。
- 服务剥离规则。
- `eligible_for_profile` 和 `allocation_eligible` 判断。

### 9.4 与 M11.6/M11.7 的关系

M11.6 只能对可进入 SKU 画像的维度做权重分配，不能把所有 `weak` 或 `insufficient` 候选归一化。

M11.7 要校验销量纵横一致性，同时校验业务稀疏性：如果所有 SKU 都命中所有维度，即使销量守恒，也应判定业务维度失真。

## 10. 质量规则

### 10.1 低价值过滤

低价值评论不能进入原生维度发现主流程。典型低价值评论包括：

- 只有“好评、不错、满意、值得、挺好”。
- 与产品无关的物流模板话术。
- 重复模板评论。
- 不能提供人群、场景、动机、产品体验、服务体验或风险线索的泛化表达。

### 10.2 服务剥离

服务履约必须单独归为 `service_context` 或 `risk_context`。

安装、配送、售后可以影响购买体验，但不能单独支撑：

- 产品价值战场。
- 产品卖点价值层。
- 核心竞品的产品能力对打理由。

### 10.3 产品战场锚点

产品价值战场必须有产品锚点。

产品锚点包括：

- 参数锚点：高刷、HDMI、Mini LED、亮度、控光、护眼、音频、内存、语音等。
- 卖点锚点：结构化卖点、宣传卖点、参数支撑卖点、评论验证卖点。
- 产品体验锚点：画质、音质、护眼、智能、游戏体验等强评论证据。

只有服务评论、价格背景或泛化好评，不能形成产品价值战场。

### 10.4 过宽维度处理

如果某个候选维度覆盖绝大多数 SKU，且缺少清晰排除规则，应标记为过宽。

过宽维度不能直接进入后续 SKU 分类，除非详细设计中证明它具备足够区分度，例如：

- SKU 间支持强度差异明显。
- 维度内部存在可解释的子类。
- 与参数、卖点、市场有稳定差异。

### 10.5 缺失不等于否定

缺少评论、参数或卖点时，只能输出 `unknown`、`insufficient` 或 `review_required`，不能推导为该 SKU 不具备某项能力。

### 10.6 维度不能互相污染

- 目标客群不能由任务自动映射生成，必须有人群、场景、动机或价格证据。
- 用户任务不能由单一卖点直接生成，必须说明使用任务。
- 价值战场不能由服务履约生成，必须说明产品价值竞争点。
- 购买动机可以支撑任务和客群，但不等同于产品战场。

## 11. 复核条件

以下情况必须进入复核队列：

1. 原生维度与 seed 名称相同，但证据结构明显不同。
2. 一个 seed 维度被多个原生维度同时命中，需要拆分。
3. 多个 seed 维度被同一批评论线索支撑，无法区分。
4. 维度覆盖 SKU 过多，可能成为无区分度背景维度。
5. 产品战场缺少产品锚点。
6. 服务履约进入产品战场。
7. 新原生维度在 seed 中不存在，但句数、SKU 覆盖或市场支撑达到门槛。
8. 维度名称无法从评论代表短语解释。
9. 代表证据主要来自低价值评论。

## 12. 业务展示需求

M08.4 的结果面向业务复核，不面向最终高层报告。

页面或查询结果应优先展示：

1. 真实评论中出现了哪些原生业务线索。
2. 哪些线索能组成用户任务、目标客群、产品价值战场。
3. 哪些预设维度被真实数据支持。
4. 哪些预设维度需要改名、拆分、合并、降级或废弃。
5. 哪些线索被判定为服务语境、风险语境或泛化背景。
6. 每个判断背后的代表评论、参数、卖点和市场证据。

展示语言必须是业务语言，不能出现内部字段名、英文枚举、哈希、SQL 错误或 AI 过程描述作为主展示内容。

## 13. 验收标准

### 13.1 功能验收

M08.4 完成后，应满足：

1. 能从 M05/M06/M08 产物中生成评论原生线索。
2. 能形成原生任务、原生客群、原生产品价值战场、服务语境等候选。
3. 能对 seed 维度输出 keep、rename、split、merge、downgrade、deprecate、new 等建议。
4. 能明确指出服务履约不能作为产品价值战场。
5. 能识别游戏、体育、卧室/副屏、单位采购等窄场景，不把它们泛化到所有 SKU。
6. 能识别家庭泛化、泛化好评等弱线索，不直接升级为目标客群或战场。
7. 能为每个候选维度提供代表评论和证据摘要。

### 13.2 205 数据验收

基于 205 当前真实批次运行时，应满足：

1. 发现的高频原生线索应覆盖画质、服务、价格价值、音质、智能易用、客厅观影、护眼、长辈、装修/家居、换新、儿童、游戏、体育、卧室/副屏、单位采购等方向。
2. 服务履约必须被归为服务语境或风险语境，不进入产品价值战场。
3. 游戏和体育不能因为 seed 中合并为“游戏体育战场”就覆盖绝大多数 SKU。
4. 长辈相关评论可以形成目标客群候选，但只有具备语音、易用、少广告、适老等产品锚点时，才能支持产品战场。
5. “家庭泛化”不能直接生成“家庭换新用户”或“家庭观影战场”。
6. 输出必须包含与原 seed 的逐项对照，说明保留、调整或废弃原因。

### 13.3 后续支撑验收

M08.4 输出必须能支撑后续详细设计：

1. M08.5 能基于 M08.4 输出生成当前项目 active 本体版本。
2. M09/M10/M11 能基于新本体避免所有 SKU 全量命中所有维度。
3. M11.6 能只对可分配维度做销量权重估算。
4. M11.7 能做销量守恒和业务稀疏性双重校验。
5. 如果后续新增原始数据，M00-M08 重新处理后，M08.4 能基于新批次增量更新原生维度候选。

## 14. 通过 / 不通过判断

### 14.1 通过

满足以下条件才算通过：

- 评论原生线索先于 seed 对齐生成。
- 原生维度名称可由真实评论代表短语解释。
- 对每个 seed 维度都有明确对齐建议。
- 服务履约被剥离。
- 过宽维度被标记，不直接进入分类。
- 输出能解释为什么某些维度适合后续 SKU 分类，为什么某些只适合作为候选或复核。

### 14.2 不通过

出现以下任一情况视为不通过：

- 先用 seed 维度名称套评论，再声称“名称不变说明预设准确”。
- 所有 SKU 仍然命中所有任务、客群或战场。
- 服务安装、配送、售后继续作为产品价值战场支撑。
- 游戏、体育、卧室/副屏等窄场景被扩展到大多数 SKU，且无产品锚点解释。
- 只有泛化好评就生成用户任务、目标客群或价值战场。
- 输出没有代表评论和证据，无法业务复核。
