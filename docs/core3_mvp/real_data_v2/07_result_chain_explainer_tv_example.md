# 07 彩电真实数据结果链路解释

本文补充说明 `M00-M16` 不是一串“生成了多少行表”的流水账，而是一条从原始四表到可解释竞品结论的结果生成链路。

重点回答四个问题：

1. 每个阶段到底产出什么业务内容。
2. 这个产出为什么必须存在。
3. 没有它会怎样。
4. 当前 205 `catforge_dev` 的彩电真实数据里，一个 SKU 实际生成了什么结果，哪些结果可用，哪些需要复核。

本文样例来自 205 PostgreSQL `catforge_dev`，批次 `m00_20260613004311_d548f6dc`，项目 `TV`，目标 SKU 为 `TV00009549`，商品为康佳 `LED32E330CE`。

## 0. 阅读目录和使用方式

这份文档的读法不是从表数量开始，而是从结果链路开始：先看某一步解决什么问题，再看它实际生成什么、价值在哪里、没有它会怎样、当前样例有哪些风险。

| 如果你想确认 | 先看章节 | 重点看什么 |
| --- | --- | --- |
| 原始四表进来后系统如何知道重算范围 | `3. M00-M02` | 批次登记、row hash、impacted SKU、evidence atom |
| 参数、卖点、评论为什么会影响后面结论 | `4. M03-M06` | 参数标准化、基础卖点、评论证据、评论污染风险 |
| 市场画像和 SKU 综合信号到底有什么用 | `5. M07-M08.5` | 市场画像、SKU signal profile、证据矩阵、维度本体校准 |
| 为什么要先生成任务、客群、战场 | `6. M09-M11.7` | 用户任务、目标客群、价值战场、销量分配和对账 |
| 为什么最终不是简单 TopN 竞品 | `7. M12-M16` | 候选召回、组件/角色分、三槽位选择、证据卡、发布门禁 |
| 康佳样例从头到尾如何生成结果 | `8. 一个完整的结果生成链路` | 康佳 `LED32E330CE` 到三槽位竞品的完整推导 |
| 当前链路哪些可用、哪些必须修 | `9. 怎么判断结果有没有用、错不错` | 评论污染、窗口信号、候选审计、M16 口径 |
| 新数据丰富后怎么程序化执行 | `10. 新数据进来后应如何程序化执行` | 按原始表变化触发模块，按 hash 和门禁判断下游是否重跑 |
| 后续页面和自动化还缺什么 | `11. 后续需要补齐的产品化展示` | 结果链路视图、候选审计、质量风险和验收看板 |

## 1. 先区分两类流程

| 类型 | 名称 | 作用 | 是否产生业务结论 |
| --- | --- | --- | --- |
| 数据生产线 | `M00-M16` | 把原始四表变成参数、卖点、评论、市场、任务、客群、战场、竞品和报告 | 是 |
| 运维/执行方式 | GitHub 同步、热修复、镜像构建、批处理、resume、Multica/Codex/Claude/OpenClaw | 让代码和任务在 205 上可运行、可恢复、可远程协作 | 否 |

所以要判断“结果有没有用、错不错”，只看 `M00-M16` 的业务产物；不要把 GitHub 同步、热修、构建镜像这些运行方式混进业务 SOP 里。

## 2. 原始四表到最终报告的清晰链路

```text
原始四表
  -> M00 批次登记：知道这次处理了哪些行、影响哪些 SKU
  -> M01 清洗诊断：知道哪些字段可用、哪些缺失/冲突/重复
  -> M02 evidence atom：把所有可引用事实变成证据 ID
  -> M03/M04/M05/M06/M07：分别生成参数、卖点、评论、市场事实
  -> M08 SKU 综合信号画像：把上面事实合并成一个 SKU 可消费画像
  -> M09/M10/M11/M11.5：把 SKU 画像翻译成任务、客群、价值战场、卖点价值层
  -> M12/M13/M14：召回候选、评分、选出三槽位核心竞品
  -> M15：生成证据卡和高层报告
  -> M16：复核、发布门禁、增量重算
```

关键原则：下游模块不应回头直接读原始四表做业务判断。它应该读上游已经结构化、带证据、带置信度、带质量标记的结果。

## 3. M00-M02 数据入口和证据层

M00-M02 是整条链路的地基。它们不负责判断“谁是竞品”，但决定后面所有结论能不能被追踪、复算和复核。

| 没有会怎样 | 新数据进来后只能全量重跑或靠人工猜测，无法知道“哪个 SKU 因为什么变化要重算” |

### 3.2 M01 清洗规范化和质量诊断

| 项目 | 说明 |
| --- | --- |
| 输入 | M00 登记后的原始行和 row hash |
| 生成内容 | `core3_clean_sku`、`core3_clean_market_weekly`、`core3_clean_attribute`、`core3_clean_claim`、`core3_clean_comment`、`core3_data_quality_issue` |
| 业务价值 | 把原始字段变成统一口径，同时把缺失、横杠、冲突、重复、价格校验等问题显式标出来 |
| 没有会怎样 | 下游会把 `-`、空值、unknown 误当成“没有这个能力”，或者把脏评论/重复评论当成真实用户声音 |
| 质量判断 | 看 clean 表是否保留原值和规范值、unknown 是否标记为 unknown、issue 是否可追溯到原始行 |

康佳 `LED32E330CE` 在 M01 的实际结果：

| 清洗对象 | 实际结果 | 业务解释 |
| --- | --- | --- |
| SKU 主记录 | 品牌康佳、型号 `LED32E330CE`，来源覆盖参数/卖点/周销 | 主体识别稳定，没有品牌和型号冲突 |
| 市场清洗 | 45 条周销记录，覆盖 `26W01` 起的线上专业电商/平台电商 | 可进入 M07 市场画像 |
| 参数清洗 | 90 条属性，其中 55 条是 `dash/unknown_value` | 参数可用，但缺失很多，后续参数结论要降级 |
| 卖点清洗 | 13 条卖点，示例包括高清硬屏、多接口、低碳环保、立体声效 | 可进入 M04a 基础卖点激活 |
| 评论清洗 | 0 条 | 后续评论层对这个 SKU 不应强行构造用户体验结论 |
| 质量 issue | 55 条 `unknown_value`，severity 为 `info` | 不是阻断错误，但必须传给 M03/M08/M15 做置信度提示 |

这里的关键设计理念是“缺失是未知，不是否定”。例如参数表里 `HDR = -`，M01 不能把它解释为“不支持 HDR”，只能记录为 `value_presence=dash`、`quality_flags=["unknown_value"]`。这会防止后面错误地说该 SKU 没有某项能力。

### 3.3 M02 evidence atom

| 项目 | 说明 |
| --- | --- |
| 输入 | M01 clean 表和质量 issue |
| 生成内容 | `core3_evidence_atom` |
| 业务价值 | 把可引用事实变成稳定证据 ID，后续参数、卖点、评论、市场、竞品和报告都引用 evidence，而不是直接引用原始行 |
| 没有会怎样 | 报告只能给结论，不能回答“这句话来自哪张表哪一行”，也无法做审计和复核 |
| 质量判断 | 看 evidence 是否有来源表、clean 表、字段、置信度、质量标记、是否 current |

康佳 `LED32E330CE` 在 M02 的实际证据构成：

| evidence 类型 | 数量 | 平均置信度 | 业务含义 |
| --- | ---: | ---: | --- |
| `param_raw` | 90 | 0.5639 | 参数原始事实，有较多 unknown，置信度中等 |
| `quality_issue` | 55 | 0.45 | 参数缺失/横杠问题本身也成为证据，供下游降级 |
| `market_fact` | 45 | 0.95 | 周度销量、销额、价格证据，市场事实较强 |
| `promo_sentence` | 15 | 0.8 | 卖点句级证据，供 M04a/M04b 使用 |
| `promo_raw` | 13 | 0.85 | 原始卖点行证据 |
| `sku_fact` | 1 | 0.8 | SKU 主体证据 |

M02 的价值在于建立“证据账本”。例如 M07 说这个 SKU 有周度市场量价证据，实际可以追到 `market_fact` evidence；M03 说某个参数未知，可以追到 `param_raw` 和 `quality_issue` evidence；M15 报告里不能只写“数据不完整”，而要能说明是哪类字段导致不完整。

### 3.4 M00-M02 这一层当前是否有用、错不错

| 检查点 | 康佳样例结论 | 判断 |
| --- | --- | --- |
| 批次登记是否完整 | 四表全量登记 195203 行，84 个 impacted SKU | 可用 |
| SKU 影响范围是否清晰 | 康佳影响来自参数、卖点、周销；评论未覆盖 | 可用，且能防止评论强推 |
| 清洗是否保留未知 | `HDR = -` 等被标记为 unknown，不被当成 false | 设计正确 |
| 质量问题是否下传 | 55 条 unknown issue 进入 M02 evidence | 可用 |
| 风险点 | M00 全量批次下所有行都是 insert，下一批增量时要验证 `no_change/update` 是否准确 | 后续新增数据时必须重点验收 |

所以 M00-M02 的结论是：这一层目前对彩电样例是有用的，能清楚告诉后续“哪些事实可用、哪些未知、哪些 SKU 要重算”。它不直接产生竞品结论，但没有这一层，后续所有结论都无法稳定复算和审计。

## 4. M03-M06 参数、卖点和评论层

M03-M06 是把“清洗后的事实”变成“可被业务使用的产品能力和用户感知”。这一层仍然不负责选择竞品，它负责回答：这个 SKU 有哪些参数能力、能激活哪些卖点、用户评论是否支持这些卖点、评论里还能提炼出哪些原生任务和体验信号。

| 环节 | 输入 | 产出内容 | 价值 | 没有会怎样 | 质量/复核点 |
| --- | --- | --- | --- | --- | --- |
| M03 参数字段画像与标准参数抽取 | M01 参数 clean 表、M02 参数 evidence、参数本体 | 标准参数值、字段覆盖率、冲突值、SKU 参数画像 | 把“原始属性名和值”翻译成统一 `param_code`，让不同品牌不同写法可比较 | 下游只能读散乱属性，无法比较刷新率、分辨率、尺寸、背光等能力 | unknown 不能当 false，冲突值要进入 review，低置信参数不能强说 |
| M04a 基础卖点激活 | M03 参数、M02 卖点 evidence、卖点本体 | 基础卖点命中、来源状态、基础激活分 | 先用参数和宣传语判断“商品自己声称或具备什么卖点” | M08/M15 只能看参数，无法表达清爽系统、老人友好、安装服务等业务卖点 | promo only、param only、param+promo 要分开，避免宣传语单独决定强结论 |
| M05 评论去重和句级 evidence | M01 评论 clean 表、M02 评论 evidence | 评论单元、句级 evidence、评论质量画像 | 把评论拆成可引用句子，并判断样本是否足够、重复是否严重 | M06 会把重复评论或脏文本当真实用户声音 | 空评论、重复评论、质量诊断文本不得混进用户评论 |
| M06 评论原生信号 | M05 句级 evidence、任务/卖点/战场词典 | 评论侧任务线索、卖点验证、战场支撑、风险信号 | 让用户声音独立参与判断，而不是只听参数和宣传语 | 系统无法知道“用户实际在夸什么、抱怨什么、为什么买” | 低样本、低置信、非用户评论信号必须降权或阻断 |
| M04b 参数+卖点+评论增强 | M04a 基础卖点、M06 评论信号 | 最终卖点激活、评论验证状态、review issue | 判断卖点是否被评论增强、削弱或需要复核 | 卖点会停留在商家宣传层，无法区分“声称有”和“用户认可” | 评论污染会直接污染 final claim，必须有门禁 |

这一层的设计理念是分离三种声音：参数是“机器可读能力”，卖点是“商品自我表达”，评论是“用户实际反馈”。三者一致时才能强说；三者冲突或缺失时必须降级。

### 4.1 M03 参数字段画像与标准参数抽取

康佳 `LED32E330CE` 在 M03 的实际结果：

| 内容 | 实际结果 | 业务解释 |
| --- | --- | --- |
| 参数完整度 | 0.555556 | 一半多参数可形成画像，但不能当作完整规格书 |
| 已知/未知参数 | 已知 14 个、未知 14 个 | 参数能力有基础，但缺口明显 |

| 标准参数 | 原始含义 | 抽取值 | 置信度 | 用途 |
| --- | --- | --- | ---: | --- |
| `native_refresh_rate_hz` | 原生刷新率 | `60HZ`，数值 60 | 0.95 | 判断是否普通 60Hz 入门屏 |

需要复核的例子是 `color_depth_bit`。当前抽取里出现过把 `曲面弧度=0`、`三维电视=2D`、`HDMI数量=0` 等原始字段映射到同一标准参数的情况，这说明字段画像和抽取规则还需要校准。M03 的价值正是在这里：它不是把所有原始属性“包装成结论”，而是把可用、未知、冲突、低置信全部显式暴露出来。

如果没有 M03，后续 M04/M08/M14 会直接读原始属性。结果会出现两个问题：一是不同品牌同义字段无法比较，二是 `-`、空值、unknown、0 这类值容易被误读成确定结论。

M04b 会把 M06 评论信号接回来，判断卖点是被评论增强、削弱还是证据不足。当前康佳样例里，`高性价比` 的 final activation 是 0.6072，状态为 `comment_enhanced`、`validated`，评论验证分 0.8303；但这个结果不能直接信，因为这个 SKU 的 M05/M06 评论层存在污染，见下一节。也就是说，M04b 的设计是对的，但它对评论质量高度敏感。

M04a/M04b 的价值在于把“卖点”拆成三层：商品有没有说、参数是否支持、用户是否认可。没有这一层，报告会只能堆参数，或者把宣传语当事实，业务可信度会很弱。

### 4.3 M05 评论去重和句级 evidence

M05 的目标不是做情感分析，而是先回答“这批评论能不能被用”。它会生成评论单元、句级 evidence、去重率、空维度率、样本是否足够等质量结果。

康佳 `LED32E330CE` 的 M05 暴露了一个关键问题：

| 内容 | 实际结果 | 业务判断 |
| --- | --- | --- |
| M01 评论清洗 | 0 条 | 这个 SKU 没有真实评论覆盖 |
| M05 评论质量画像 | raw comment row 55、comment unit 55、sentence 55 | 与 M01 不一致 |
| distinct comment id/text | 0/0 | 不像真实评论 |
| duplicate text rate | 1.0 | 高度重复 |
| empty dimension rate | 1.0 | 评论维度为空 |
| sample status | `insufficient` | 不应进入强评论判断 |
| 样例文本 | “TV00009549 存在空值、unknown 或横杠等未知值” | 这是质量诊断句，不是用户评论 |

这说明当前链路里有质量诊断 evidence 泄漏到评论层。这个发现很重要，因为它不是“数据量多少”的问题，而是评论输入边界错了。M05 必须增加门禁：`quality_issue` 类 evidence 不能生成用户评论 unit，也不能进入 M06 原生信号。

为了说明 M05 在真实评论充足时应该长什么样，另一个彩电 SKU `TV00028168`，TCL `85T7L ULTRA`，给出了干净样例：

| 内容 | 实际结果 | 业务解释 |
| --- | --- | --- |
| raw comment row | 1129 | 评论样本充足 |
| distinct comment id/text | 1129/1129 | 去重质量好 |
| sentence count | 2723 | 可拆成句级证据 |
| usable sentence | 1975 | 大量句子可供下游使用 |
| duplicate text rate | 0.0 | 没有明显重复灌水 |
| empty dimension rate | 0.005876 | 维度缺失很低 |
| usability score | 0.85123 | 可进入 M06 |
| downstream ready | true | 评论层可下游消费 |

这个 clean 样例里，句级 evidence 能看到“画质清晰、色彩鲜艳、安装好、价格合适、老人小孩容易操作”等真实用户表达。它和康佳样例形成对照：不是所有 SKU 都有足够评论，系统必须能区分“没有评论”和“评论足够”。

### 4.5 M03-M06 这一层当前是否有用、错不错

| 检查点 | 康佳样例结论 | 判断 |
| --- | --- | --- |
| 参数抽取是否有用 | 能抽出 60Hz、1366x768、LCD、二级能效等基础事实 | 有用，但低置信和冲突多 |
| 卖点激活是否有用 | 能识别清爽系统、性价比、安装服务、智能易用等候选卖点 | 有用，但很多需要复核 |
| 评论层是否可信 | 康佳没有真实评论，M05/M06 却生成了质量诊断句评论 | 当前对该 SKU 不可信 |
| 评论充足 SKU 是否跑通 | TCL `85T7L ULTRA` 能形成 1129 条评论单元、1975 条可用句子和高置信原生信号 | M05/M06 机制可用 |
| 最大缺陷 | `quality_issue` 泄漏为评论 evidence，导致强任务/卖点/战场信号误生成 | 必须修复 |

这一层的阶段性结论是：M03 和 M04a 已经能提供可解释的参数和基础卖点；M05/M06 的算法形态是有价值的，但必须先堵住“质量诊断句进入评论层”的入口。否则后续 M04b 的评论增强和 M08 的综合画像会把错误评论信号放大。

## 5. M07-M08.5 市场、SKU画像和维度本体层

M07-M08.5 是从“单 SKU 产品事实”进入“可比较、可推导、可复用业务维度”的关键层。它解决四个问题：

1. M07 判断这个 SKU 在市场里的量价和可比池位置。
2. M08 把参数、卖点、评论、市场合并成下游统一读取的 SKU 事实包。
3. M08.4 从真实评论和产品锚点里发现原生任务、客群、战场候选，防止预设维度套所有 SKU。
4. M08.5 把原生候选、seed 本体和复核规则融合成当前项目可用的业务维度版本。

| 环节 | 输入 | 产出内容 | 价值 | 没有会怎样 | 质量/复核点 |
| --- | --- | --- | --- | --- | --- |
| M07 市场画像 | M01 周销 clean 表、M03 尺寸参数、M02 市场 evidence | SKU 市场画像、市场信号、可比池、池成员关系 | 让下游知道量价位置、渠道位置、趋势和候选比较边界 | 竞品选择会只看参数/卖点，不知道谁在真实市场里挤压谁 | analysis window、样本状态、渠道范围、同池样本数必须展示 |
| M08 SKU 综合信号画像 | M03 参数、M04b 卖点、M06 评论、M07 市场 | SKU 总画像、证据矩阵、下游特征视图 | 给 M09-M15 一个统一事实入口，避免各模块各自拼表 | 任务、客群、战场、竞品和报告会口径不一致，增量重算也无法判断影响范围 | JSON 摘要要可读，证据矩阵要暴露缺失、风险和污染 |
| M08.4 产品锚点/原生维度候选 | M08 画像、M05/M06 评论证据、M03 参数、M04 卖点、M07 市场 | 原生维度候选、SKU 支撑、seed 对齐建议、复核问题 | 从真实用户表达和产品锚点反推“应该有哪些任务/客群/战场” | M09-M11 会按 seed 全量套维度，84 个 SKU 几乎都像什么都命中 | 评论污染、服务履约、弱产品锚点、维度过宽必须被拦截 |
| M08.5 业务维度本体校准 | M08.4 候选和对齐建议、seed 本体、M08/M07 支撑 | 当前项目 active 本体版本、维度定义、证据锚点、校准 issue | 发布给 M09/M10/M11 使用的任务、客群、战场定义和分配规则 | 下游会把服务、泛化好评、弱锚点都当成正式业务维度 | 战场必须有产品锚点；服务只能做 context；过宽维度要 warning 或 review |

### 5.1 M07 市场画像到底是什么

M07 不是最终市场报告，也不是竞品结论。它回答的是：这个 SKU 在当前样例市场里的量价位置、渠道位置、趋势、可比池基础是否成立。

### 5.3 M08.4 产品锚点和评论原生维度候选

M08.4 是后面补出来但非常关键的一层。它的直接背景是：205 真实数据验证发现，如果 M09/M10/M11 直接套 seed 维度，84 个 SKU 会几乎都命中所有任务、客群和战场。数学上能出分，业务上没有区分度。

M08.4 的设计理念是先问“真实评论和产品锚点告诉我们有哪些业务维度”，再去和 seed 做对齐。它不发布最终任务、客群、战场，只产出候选和建议。

## 6. M09-M11.7 业务语义、销量分配和对账层

M09-M11.7 是把 SKU 画像翻译成业务语言的层。它不直接选择三竞品，而是回答：这个 SKU 主要服务什么任务、面向什么客群、处在哪些价值战场、哪些卖点在战场里有价值、销量应该如何分配到这些业务维度，以及这些分配是否守恒。

| 环节 | 输入 | 产出内容 | 价值 | 没有会怎样 | 质量/复核点 |
| --- | --- | --- | --- | --- | --- |
| M09 用户任务 | M08 `for_module=M09` 视图、M08.5 task 本体、证据矩阵 | SKU 任务候选、任务分、证据拆解、复核 issue | 把参数/卖点/评论/市场翻译成“用户买它来干什么” | 战场和客群只能靠 seed 或卖点硬套，缺少使用场景 | 评论污染、参数冲突、任务重复 current 行、低证据任务要降级 |
| M10 目标客群 | M09 任务、M08/M08.5 客群定义、市场和评论线索 | SKU 客群候选、客群分、证据拆解、复核 issue | 把任务翻译成“谁在买/谁适合” | 后续报告只知道功能，不知道面向人群 | 客群不能只由任务映射，必须看评论/价格/渠道/市场是否支撑 |
| M11 价值战场 | M08、M09、M10、M08.5 battlefield 定义 | 战场分、战场 portfolio、召回语境、复核 issue | 把产品能力、任务、客群、市场汇成“在哪个价值战场竞争” | M12/M13 不知道按什么竞争语境召回和评分 | 战场必须有产品锚点；服务不能变成产品战场；portfolio 与单项分要一起看 |
| M11.5 战场内卖点价值层 | M11 战场、M04b 卖点、M07/M08 可比池、评论感知 | 卖点在战场内的价值分层、证据拆解、summary、复核 issue | 判断某个卖点在某个战场里是核心竞争力、弱感知还是仅 seed 线索 | 报告只能说“有卖点”，不能说“这个卖点有没有竞争价值” | 评论缺失、宣传缺失、样本不足、claim only 都要复核 |
| M11.6 业务画像/销量分配 | M09/M10/M11/M11.5、M07 市场销量销额 | SKU 业务画像、主任务/主客群/主战场、维度权重、销量销额分配 | 把多张分数表压成一个可给 M12-M15 消费的 SKU 业务 profile | 下游每次都要重新选择主语义和分配销量，口径会漂移 | 主维度选择、权重归一、异常溢价标签、弱证据主战场要复核 |
| M11.7 销量对账 | M11.6 分配表、M07 市场销量销额 | 维度销量 summary、SKU contribution、守恒检查、对账 issue | 证明业务维度分配没有凭空创造或丢失销量/销额 | 市场规模口径不可信，维度份额可能被重复相加 | 每类维度内部守恒；不要把 task、客群、战场、卖点四类加总 |

## 7. M12-M16 竞品选择、报告、复核发布层

M12-M16 是把“业务语义画像”变成“可展示、可复核、可发布竞品结论”的层。它和前面的最大区别是：前面回答 SKU 是什么、用户为什么买、在哪个战场竞争；这一层回答哪些 SKU 值得进入候选、为什么打这些分、最终三槽位是谁、报告怎么展示、发布前能不能过门禁。

| 环节 | 输入 | 产出内容 | 价值 | 没有会怎样 | 质量/复核点 |
| --- | --- | --- | --- | --- | --- |
| M12 候选召回 | M07 可比池、M08 SKU 画像、M11 战场、M11.5 卖点价值、M11.6 业务画像 | 候选池、召回原因、候选特征快照、召回复核 issue | 先把“可能相关”的 SKU 收进池子，并保留为什么进入候选 | M13/M14 只能在全量 SKU 上硬排，慢且不可解释 | 候选不能等于最终竞品；强召回也可能因角色不合适被淘汰 |
| M13 组件分/角色分 | M12 候选和特征快照、M07-M11.5 证据 | 组件分、角色分、组件解释、评分复核 issue | 把每个候选拆成价格、渠道、参数、卖点、市场、评论等分项，并判断适合哪个竞品角色 | 只剩总分，无法解释为什么某个 SKU 是价格挤压而不是正面对打 | 低组件必须展示 gap；服务类角色不能被过度当核心竞品 |
| M14 三槽位核心竞品选择 | M13 组件分/角色分、M11 战场、选择策略 | 三槽位 selection、slot decision、候选审计、选择复核 issue | 从候选池里选出 0-3 个核心竞品，并区分正面对打、价格/销量挤压、高端标杆/潜在下探 | 会退化成 TopN 排名，业务不知道每个竞品承担什么竞争压力 | 每槽最多一个；空槽要保留原因；未选候选要有审计原因 |
| M15 证据卡和高层报告 | M14 选择、M13 分项、M08-M11.5 画像、M02 evidence | 证据卡、目标报告 payload、报告章节、导出包、报告复核 issue | 把数据库结果改写成业务可读的报告和证据卡 | 只有表结果，没有给业务展示和复核的页面材料 | 报告不能暴露内部字段/UUID；必须写数据范围和风险口径 |
| M16 复核、发布门禁、增量编排 | M00 impacted SKU、M12-M15 产物、review issue、验收规则 | review queue、acceptance report、发布门禁、重算计划 | 判断本轮结果能不能发布、哪些问题要人工复核、下次增量重算什么 | 新数据进来后靠人工判断是否重跑；低质量报告可能直接发布 | 门禁口径必须和 M14/M15 实际产物一致，验收字段不能自相矛盾 |

这一层的核心设计理念是：候选、评分、选择、报告、发布必须拆开。M12 说“值得进池”，M13 说“各组件和角色分是多少”，M14 说“最终三槽位选谁”，M15 说“怎么给业务解释”，M16 说“能不能发布和下次怎么重算”。如果把它们合成一个 TopN 排名，结果会看起来简单，但业务无法判断为什么选、错在哪、如何复核。

## 9. 怎么判断结果有没有用、错不错

不能只看“生成了几千行”。应该按下面的质量门禁看。

| 检查点 | 应该怎么看 | 康佳样例状态 | 结论 |
| --- | --- | --- | --- |
| 来源是否全 | M08 `source_coverage_json` 是否覆盖参数、卖点、评论、市场、质量 | 全部 covered | 基础可用 |
| 市场是否可说 | M07 是否有销量、价格、渠道、样本状态和质量标记 | 有 23 周、45 行、线上样本，置信度 high | 可以说线上样例市场，不可说全渠道全年 |
| 参数是否可靠 | M03/M08 是否有冲突、unknown、置信度 | 有不少 unknown 和冲突 | 参数结论要降级表达 |
| 评论是否干净 | M05/M06 是否真实来自用户评论句，而不是质量诊断句 | 当前 M08 摘要里出现“存在空值、unknown 或横杠”等质量句作为评论线索 | 这是需要复核/修复的问题 |
| 市场信号是否矛盾 | 同一 signal 是否跨窗口重复，是否带 analysis window | M07 里同一 SKU 可同时出现不同窗口下的价格高/低信号 | UI/报告必须带窗口，否则会误读 |

## 10. 新数据进来后应如何程序化执行

新增或更新原始四表后，不应该人工判断每个表要重跑什么。标准流程是：

| 原始变化 | 先触发 | 后续影响 |
| --- | --- | --- |
| `week_sales_data` 新增/变化 | M00、M01、M02、M07 | M08-M15 受影响，因为市场画像和可比池变了 |
| `attribute_data` 新增/变化 | M00、M01、M02、M03 | M08-M15 受影响，因为参数画像变了 |
| `selling_points_data` 新增/变化 | M00、M01、M02、M04a、M04b | M08-M15 受影响，因为卖点激活变了 |
| `comment_data` 新增/变化 | M00、M01、M02、M05、M06、M04b | M08-M15 受影响，因为评论信号和卖点验证变了 |

M00 的 `impacted_sku` 是增量执行入口。M16 应该读取 impacted SKU 和上游依赖 hash，决定哪些 SKU、哪些模块需要重算。重算后，M08 的 `profile_hash` 是关键分界点：如果画像 hash 没变，下游 M09-M15 理论上可以复用；如果变了，下游必须按依赖关系重算。

程序化执行时建议把门禁拆成四层：

| 门禁 | 通过条件 | 不通过时怎么办 |
| --- | --- | --- |
| 数据入口门禁 | M00 能识别 impacted SKU，M01/M02 有 clean 和 evidence | 先修导入、清洗或 evidence，不继续下游 |
| 画像门禁 | M03-M08 产出 profile，且评论/参数质量风险被显式标记 | 可继续跑下游，但报告必须带风险；严重污染应阻断 |
| 业务语义门禁 | M09-M11.7 有任务、客群、战场、销量分配，并且 M11.7 对账通过 | 对账不通过时不能发布市场份额类结论 |
| 报告发布门禁 | M12-M15 形成三槽位/空槽原因和证据卡，M16 无 blocker | 有 blocker 或验收字段不自洽时进入 review queue |
