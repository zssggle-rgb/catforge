# CatForge 彩电核心三竞品真实数据 MVP 最终验收清单

## 1. 本轮验收目标

本轮验收只验证本地可复现链路，不部署、不连接 205。验收对象是 M00-M16、业务 API、独立前端页面在 `local_validation_fixture_v1` 上形成的完整业务闭环。

验收结论必须回答三件事：

1. 数据是否从原始四表进入清洗、证据、画像、候选、评分、报告和门禁分层。
2. 最终页面是否用业务语言先展示三竞品是谁，再解释为什么成立。
3. 205 恢复后是否具备直接用真实 PostgreSQL 数据复跑的条件。

## 2. 本地样例覆盖

`local_validation_fixture_v1` 覆盖 6 个彩电 SKU，目标型号为 `TV900001 / 海信 85E8Q Pro`。

| 覆盖项 | 验收要求 |
| --- | --- |
| 原始市场数据 | `week_sales_data` 有目标、竞品、弱相关和负样本的周销量、价格、渠道 |
| 原始参数数据 | `attribute_data` 覆盖尺寸、刷新率、背光、分区、亮度、色域、接口、运动补偿、音响 |
| 原始卖点数据 | `selling_points_data` 覆盖目标和候选 SKU 的业务卖点 |
| 原始评论数据 | `comment_data` 覆盖游戏、体育、家庭观影、画质、安装服务、价格感知 |
| 同品牌竞品 | `TV900004 / 海信 85U7N` 不因同品牌被过滤，可作为高端标杆竞品 |
| 负样本 | `TV900006 / 创维 65A5D` 应被排除，验证候选过滤能力 |

## 3. 链路验收

| 阶段 | 要求 |
| --- | --- |
| M00-M02 | 能登记原始表、清洗后生成证据，保留来源和置信度 |
| M03-M06 | 能形成参数画像、卖点激活、评论单元和评论信号 |
| M04b-M08 | 能形成卖点评论互证、市场画像、可比池和 SKU 综合信号 |
| M09-M11.5 | 能推导用户任务、目标客群、价值战场和卖点价值层 |
| M12-M14 | 能召回候选、组件评分、选择核心三竞品和三类角色 |
| M15-M16 | 能生成业务报告、证据卡、导出、验收报告和发布门禁 |
| API | 能通过 v2 业务 API 查询总览、目标列表、报告、竞品、证据卡、章节、导出和证据追溯 |
| FRONTEND | 独立页面 `/core3-real-data` 能展示业务报告，不混入旧页面 |

## 4. 业务展示验收

高层页面必须满足：

1. 先展示三竞品角色和型号，再展示成立理由。
2. 展示价值战场、市场压力、参数证据、卖点证据、评论证据和策略含义。
3. 不展示 UUID、SQL、原始表名、内部字段、候选池内部编号、算法过程文案。
4. 不出现 `AI 认为`、`模型判断`、`生成过程`、`market_aggregate`、`comment_signal`、`task_battlefield` 等内部表述。
5. 对 85E7Q 真实样例的卖点缺口必须表述为“宣传卖点数据缺口”，不能表述为“没有卖点”。

## 5. 205 恢复后复跑条件

205 恢复后需要执行真实数据阶段运行，并保留落库结果。复跑前不需要改业务代码，但必须确认：

1. PostgreSQL 原始四表仍可读，清洗表、证据表、画像表、结果表可写。
2. M00-M16 pipeline 使用上一阶段落库结果，不能跳过中间模块。
3. 真实数据可能继续增加，验收采用水位、覆盖率和变化摘要，不写死固定总数。
4. 当前 85E7Q 如仍缺结构化卖点，应触发数据缺口说明和复核提示。
5. 如新增外部品牌或更多同品牌 SKU，应重新观察核心三竞品角色是否变化。

## 6. 当前本地通过标准

本地验收通过需要同时满足：

1. 后端核心三竞品测试通过。
2. M00-M16 本地样例全链路测试通过。
3. 业务 API 守卫测试通过。
4. 前端 v2 页面测试通过。
5. 前端构建通过。
6. 本文档和测试结果能说明剩余风险，而不是把样例结果误当成真实 205 结论。
