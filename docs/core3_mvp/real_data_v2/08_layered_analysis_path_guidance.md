# 08 分层分析路径指导

本文用于重新整理 Core3 真实数据分析路径。它不是继续给 `M00-M16` 打补丁，而是把现有模块按业务对象重新分层，明确每一层先生成什么、禁止做什么、下游如何使用。

目标是形成一条更清晰的分析主线：

```text
原始数据
  -> 清洗层
  -> SKU 事实层
  -> 品类语义能力层
  -> SKU 与业务维度画像层
  -> 竞品分析层
  -> 复核、报告和增量治理
```

`M00-M16` 仍然保留为当前工程模块编号，但后续设计和页面解释应以“层”和“业务对象”为主，而不是以模块流水账为主。

## 1. 为什么需要重排思路

`07_result_chain_explainer_tv_example.md` 已经证明现有链路能跑通，也暴露了几个结构性问题：

| 问题 | 表现 | 新路径处理 |
| --- | --- | --- |
| 模块解释过长 | 文档按 M00-M16 展开，业务主线被技术模块淹没 | 按清洗、事实、语义能力、画像、竞品五层组织 |
| 清洗和分析边界不清 | 评论质量诊断句曾进入评论分析，污染任务、战场和报告 | 清洗层必须决定哪些行可分析，脏评论不能进入事实层 |
| 事实和推理混在一起 | SKU 参数、卖点、评论、市场事实还没稳定，就进入任务和战场判断 | 先生成 SKU 事实画像，再进入语义推导 |
| 品类标准不稳定 | 任务、客群、战场更像直接套 seed，而不是从品类事实校准 | 增加品类语义能力层，先生成 TV 专用品类预设 |
| 画像方向单一 | 只看 SKU 命中了哪些任务/客群/战场，不够看每个维度包含哪些 SKU | 画像层必须同时产出 SKU 画像和维度画像 |
| 竞品候选过宽 | 候选池容易接近全量 SKU，后续只能靠评分硬筛 | 竞品先按价值战场建库，再按任务、客群、价格、尺寸和卖点收敛 |

新的原则是：先把事实做干净，再把品类语义标准定清楚，最后再做 SKU 画像和竞品选择。

## 2. 新分析路径总览

```text
L1 清洗层
  输入：原始周销、参数、卖点、评论
  输出：可分析行、不可分析行、清洗质量问题、受影响 SKU

L2 SKU 事实层
  输入：清洗后的参数、周销、卖点、评论
  输出：SKU 事实画像，包括参数事实、市场事实、卖点事实、评论事实和相互印证关系

L3 品类语义能力层
  输入：全品类 SKU 事实画像、可用评论、卖点、尺寸、价格带和 seed 本体
  输出：当前品类可用的用户任务、目标客群、价值战场和判定规则

L4 SKU 与业务维度画像层
  输入：SKU 事实画像、品类语义能力
  输出：SKU -> 任务/客群/战场画像，以及任务/客群/战场 -> SKU 画像

L5 竞品分析层
  输入：SKU 画像、业务维度画像、市场可比池、战场内卖点价值
  输出：每个 SKU 的战场竞品库、2-3 个核心竞品、证据卡和展示说明
```

### 2.1 原始输入边界

真实数据链路的原始分析输入只来自四张业务表：

```text
week_sales_data
attribute_data
selling_points_data
comment_data
```

文档中提到的“质量信息”“质量问题”“质量风险”不是第五张输入表，也不是独立的业务事实来源。它们只能由清洗、抽取、对账和复核过程从四张原始表派生出来，用作：

- 判断某行是否可进入下游。
- 解释覆盖缺口、字段冲突、样本不足、低质评论等问题。
- 降低置信度、触发复核或阻断展示。
- 给人工复核提供审计线索。

质量信息不能作为用户任务、目标客群、价值战场或竞品结论的正向证据。举例：`comment_data` 中识别出的“安装快、客服好、物流慢”只能生成服务履约质量标记或服务背景统计，不能被当作“电视产品体验好”的评论事实，也不能进入产品价值战场。

M00、M02、M16 是横向支撑能力：

| 支撑能力 | 作用 | 放在新路径中的位置 |
| --- | --- | --- |
| M00 批次和行登记 | 识别新数据、变化行、受影响 SKU | 服务 L1-L5 的增量边界 |
| M02 evidence atom | 给可分析事实、画像、竞品报告提供证据 ID；给质量问题提供审计 ID | 事实证据贯穿 L2-L5；质量证据只服务门禁和复核 |
| M16 复核、门禁和增量编排 | 判断能不能发布、哪些要重算 | 贯穿全链路，不再被理解为“最后又生成一个结论” |

## 3. L1 清洗层

### 3.1 目标

清洗层只回答三个问题：

1. 哪些原始行可以进入后续分析。
2. 哪些行只能作为质量提示、服务履约背景或复核样例，不能进入产品分析。
3. 哪些 SKU 因新数据或变化数据需要重算。

清洗层不生成用户任务、目标客群、价值战场，也不生成竞品结论。

### 3.2 输入和输出

| 输入 | 输出 |
| --- | --- |
| `week_sales_data` | 可分析周销事实、异常量价记录、SKU 周度覆盖解释 |
| `attribute_data` | 可分析参数事实、unknown/dash/null 标记、字段冲突 |
| `selling_points_data` | 可分析卖点句、空卖点、模板卖点、低质卖点 |
| `comment_data` | 可分析产品评论、低质评论、默认评论、重复评论、服务类评论、不可分析评论 |

评论清洗要先做快速过滤，避免大量无效评论拖垮机器：

| 评论类型 | 是否进入后续产品分析 | 处理方式 |
| --- | --- | --- |
| 空文本、纯符号、默认好评、过短文本 | 否 | 直接过滤，只统计数量和比例 |
| 重复评论 | 否或只保留一条 | 按 `comment_id` 和正文 hash 去重 |
| 质量诊断句、系统生成句 | 否 | 不能进入评论事实层 |
| 服务、物流、安装、客服、售后、退换、发票、保修类评论 | 否 | 在 M01 阶段过滤出产品分析链路，只保留服务背景统计和少量复核样例 |
| 产品体验评论 | 是 | 进入 L2 评论事实 |

服务类评论的默认规则必须前置到 M01：只要评论主要表达履约、配送、安装、客服、售后、退换、发票、保修等服务过程，就不能进入 M02 产品事实 evidence、M05 评论事实、M06 评论信号、M09-M11 用户任务/客群/战场，也不能进入 M12-M15 竞品和报告。若一条评论同时包含产品体验和服务内容，应先做句级拆分：产品体验句可以继续下传，服务句只进入服务背景统计。

周销量价覆盖要在清洗层解释清楚：

| 覆盖现象 | 清洗层解释 |
| --- | --- |
| 某 SKU 前段周缺失 | 可能是新品或晚进入样本，不直接当异常 |
| 某 SKU 后段周缺失 | 可能是退市、断货或样本结束，不直接当异常 |
| 同一周只有一个平台有数据 | 正常，说明该 SKU 只在该平台售卖或只有该平台被采集 |
| 内部周断点 | 标记为覆盖缺口，影响趋势和可比池置信度 |
| 均价与销售额/销量不一致 | 标记量价计算问题，必要时阻断该行进入市场画像 |

### 3.3 对现有模块的调整

| 现有模块 | 新定位 | 调整建议 |
| --- | --- | --- |
| M00 | 保留 | 继续做批次、row hash、impacted SKU，不参与业务判断 |
| M01 | 强化 | 增加快速评论过滤、服务类评论隔离、量价覆盖解释、低质数据不下传 |
| M02 | 保留但收口 | 只为可分析事实生成下游 evidence；质量问题只能生成审计 evidence，不能伪装成用户评论或产品事实 |

### 3.4 清洗层门禁

进入 L2 前必须满足：

- 可分析事实和质量问题分开存储。
- `unknown` 只能表示未知，不能表示否定。
- 评论产品体验、服务履约、低质无效三类分开；服务履约在 M01 阶段即被阻断出产品分析链路。
- SKU 周度覆盖带解释，不把新品、退市、平台特供误判为异常。
- 大批量评论处理必须按 SKU 或批次分块执行。

## 4. L2 SKU 事实层

### 4.1 目标

事实层回答“这个 SKU 在样例数据中客观有什么”。

一个 SKU 的事实画像至少包括：

| 事实域 | 说明 |
| --- | --- |
| 参数事实 | 尺寸、分辨率、刷新率、屏幕技术、接口、能效、系统、音响等 |
| 市场事实 | 周销量、销额、均价、价格带、平台、渠道、趋势、可比池基础 |
| 卖点事实 | 商品宣传了什么卖点，卖点来自标题、正文还是结构化卖点 |
| 评论事实 | 用户真实评论说了什么产品体验，样本是否充足 |
| 相互印证关系 | 参数、卖点、评论之间是否一致、增强、冲突或缺失 |

事实层仍然不直接生成用户任务、目标客群、价值战场和竞品。

### 4.2 事实画像的核心结构

每个 SKU 应形成一份 `sku_fact_profile`，建议包含：

```text
sku_identity
market_fact_summary
param_fact_summary
claim_fact_summary
comment_fact_summary
cross_evidence_matrix
fact_quality_summary
source_coverage_summary
```

`cross_evidence_matrix` 是事实层的关键，不是简单拼表：

| 关系 | 含义 | 示例 |
| --- | --- | --- |
| `param_claim_consistent` | 参数和卖点互相支持 | 参数有高刷，卖点宣传游戏流畅 |
| `claim_comment_validated` | 卖点被评论验证 | 卖点说画质清晰，评论大量提到画质好 |
| `claim_comment_unverified` | 有卖点但评论不足 | 宣传清爽系统，但评论没有系统体验反馈 |
| `param_claim_gap` | 参数强但宣传未表达 | 有高端画质参数，但卖点没讲清 |
| `claim_param_weak` | 宣传强但参数支撑弱 | 宣传高端画质，但亮度/分区/HDR 参数缺失 |
| `comment_conflict` | 评论和卖点相反 | 宣传系统流畅，但评论抱怨卡顿 |

### 4.3 对现有模块的调整

| 现有模块 | 新定位 | 调整建议 |
| --- | --- | --- |
| M03 | 保留 | 作为参数事实抽取，不直接推任务和战场 |
| M04a | 保留 | 作为宣传卖点事实，不使用评论做最终增强 |
| M05 | 保留并前置门禁 | 只处理 L1 通过的产品体验评论 |
| M06 | 收敛 | 只生成评论事实和验证信号，不直接发布任务/客群/战场结论 |
| M04b | 调整为事实验证 | 输出卖点是否被评论验证、削弱或证据不足 |
| M07 | 保留 | 作为市场事实和可比池基础，不直接选竞品 |
| M08 | 重命名语义 | 从“SKU 综合信号画像”调整为“SKU 事实画像”更清晰 |

### 4.4 事实层门禁

进入 L3/L4 前必须满足：

- 每个事实都有来源、置信度和质量标记。
- 评论不足时，事实画像明确 `comment_sample_status=insufficient`。
- 卖点不能只因为宣传出现就被当成强事实。
- 参数冲突不阻断全链路，但必须进入事实质量摘要。
- 市场画像必须写明窗口、平台和样本边界。

## 5. L3 品类语义能力层

### 5.1 目标

语义能力层回答“对于彩电这个品类，系统应该使用哪些用户任务、目标客群和价值战场来分析 SKU”。

这一层生成的是品类能力，不是单个 SKU 的结论。

输入包括：

- 全部 SKU 的事实画像。
- 可用产品评论和高频表达。
- 卖点事实和参数锚点。
- 尺寸段、价格带、渠道和销量结构。
- seed 本体和人工确认规则。

输出包括：

| 对象 | 输出内容 |
| --- | --- |
| 用户任务 | 彩电品类适用的任务集合、定义、触发信号、排除条件 |
| 目标客群 | 客群集合、典型需求、价格/尺寸/渠道特征、评论线索 |
| 价值战场 | 战场集合、核心价值、必要产品锚点、市场校验规则 |
| 维度关系 | 任务和客群如何映射到战场，哪些卖点属于哪个战场 |
| 品类规则版本 | 本轮 TV 语义能力版本、证据来源、复核状态 |

### 5.2 彩电语义标准示例

彩电的任务、客群、战场不能泛化成所有品类通用标签，应先基于 TV 品类形成标准。

| 用户任务 | 可能证据 |
| --- | --- |
| 家庭观影升级 | 大尺寸、画质、音效、客厅、家庭观看评论 |
| 卧室/副屏补充 | 小尺寸、低价、安装方便、轻量观看 |
| 游戏体育流畅 | 高刷、HDMI、低延迟、运动画面、游戏/看球评论 |
| 性价比购买 | 低价、销量、促销、够用、价格正向评论 |
| 长辈易用 | 语音、遥控简单、系统清爽、字体/操作友好 |

| 目标客群 | 可能证据 |
| --- | --- |
| 性价比用户 | 低价带、高销量、价格敏感评论 |
| 家庭换新用户 | 大尺寸、客厅、画质音效、家庭观影任务 |
| 游戏体育用户 | 高刷、运动流畅、接口、游戏体育任务 |
| 卧室副屏用户 | 小尺寸、低价、安装方便、低空间占用 |
| 长辈家庭用户 | 语音、简单操作、少广告、护眼 |

| 价值战场 | 必要锚点 |
| --- | --- |
| 大屏沉浸观影 | 尺寸、画质、音效、家庭观影任务 |
| 高端画质升级 | 高端显示参数、画质卖点、高价格带或高销额支撑 |
| 游戏体育流畅 | 刷新率、低延迟、接口、运动流畅评论或卖点 |
| 智能交互易用 | 系统、语音、少广告、长辈/易用评论 |
| 性价比竞争 | 价格带、销量、基础配置够用、价格正向反馈 |

### 5.3 对现有模块的调整

| 现有模块 | 新定位 | 调整建议 |
| --- | --- | --- |
| M08.4 | 保留但改名为“品类原生语义候选发现” | 从全品类事实中发现候选任务、客群、战场，不直接给 SKU 分配 |
| M08.5 | 保留但改名为“品类语义能力校准” | 把 seed、本轮事实和人工复核合并为 TV 语义能力版本 |
| M09/M10/M11 | 下沉到 L4 | 不再承担“发现品类维度”的职责，只做 SKU 分配和画像 |

### 5.4 语义能力层门禁

发布到 L4 前必须满足：

- 每个价值战场有产品锚点，不能只有泛化好评。
- 服务、物流、安装默认不是产品价值战场，且应已在 M01 从产品分析链路中过滤；后续只能做履约背景或复核风险。
- 任务、客群、战场都有排除条件，避免所有 SKU 都命中。
- 品类语义能力需要版本号，后续新增数据可复核和演进。
- seed 只能作为起点，不能替代真实数据激活。

## 6. L4 SKU 与业务维度画像层

### 6.1 目标

画像层回答两个方向的问题：

1. 对每个 SKU：它主要服务哪些用户任务、目标客群和价值战场。
2. 对每个任务、客群、价值战场：它包含哪些 SKU，市场规模、价格结构和代表卖点是什么。

这一层开始生成业务判断，但仍然不是竞品结论。

### 6.2 SKU 画像

每个 SKU 应生成：

```text
主用户任务
辅助用户任务
主目标客群
辅助目标客群
主价值战场
机会价值战场
战场内卖点价值层
市场角色
证据强弱
质量风险
```

任务、客群、战场的分配必须来自 L2 事实画像和 L3 品类语义能力，而不是直接从评论贴标签。

### 6.3 业务维度画像

每个用户任务、目标客群、价值战场也应生成画像：

| 维度画像 | 内容 |
| --- | --- |
| 包含 SKU | 命中该维度的 SKU 列表、主/辅/弱关系 |
| 市场规模 | 维度内销量、销额、均价、份额 |
| 价格结构 | 价格带分布、尺寸段分布 |
| 代表卖点 | 该维度内高频且有证据的卖点 |
| 代表品牌/SKU | 销量、销额、证据强度靠前的 SKU |
| 风险提示 | 评论不足、参数缺失、维度过宽、服务信号污染 |

这里的风险提示不是新的原始数据源，而是基于四张原始表在清洗、事实抽取和画像分配过程中的派生质量标记。它只能解释“为什么某个画像需要降级、复核或不展示”，不能把服务类信息转换成产品价值证据。

这样业务问题可以双向回答：

- “这个 SKU 的目标客户是谁？”
- “性价比用户包含哪些 SKU？”
- “家庭观影舒适战场里，哪些卖点最能带来溢价？”
- “游戏体育用户里，哪些 SKU 只是宣传强、参数弱？”

### 6.4 溢价卖点规则

卖点是否是溢价卖点，不能只看卖点分高。建议按下面规则判断：

```text
溢价卖点 =
  卖点属于 SKU 的主价值战场
  + 在该战场内有参数或宣传证据
  + 在可比池内具备区分度
  + 能支撑更高价格、销额份额或高端标杆角色
  + 没有严重评论冲突或质量阻断
```

因此：

| 情况 | 判断 |
| --- | --- |
| 卖点在主价值战场内，且有价格/销额/高端角色支撑 | 可标为溢价卖点 |
| 卖点在主价值战场内，但只证明“够用” | 标为竞争绩效卖点 |
| 卖点只在非主战场出现 | 标为机会卖点或辅助卖点 |
| 卖点只有宣传，没有参数和评论支撑 | 不标溢价，只标宣传卖点 |
| 卖点来自服务履约 | 不进入产品溢价卖点 |

### 6.5 对现有模块的调整

| 现有模块 | 新定位 | 调整建议 |
| --- | --- | --- |
| M09 | SKU 用户任务分配 | 消费 L3 任务定义，输出 SKU->任务和任务->SKU |
| M10 | SKU 目标客群分配 | 消费 L3 客群定义，输出 SKU->客群和客群->SKU |
| M11 | SKU 价值战场分配 | 消费 L3 战场定义，输出 SKU->战场和战场->SKU |
| M11.5 | 战场内卖点价值 | 强化“主价值战场内溢价卖点”判断 |
| M11.6 | SKU 业务画像 | 保留，作为 L4 的 SKU 画像汇总 |
| M11.7 | 维度市场对账 | 保留，作为任务/客群/战场市场规模守恒门禁 |

## 7. L5 竞品分析层

### 7.1 目标

竞品分析层回答“在具体价值战场中，哪些 SKU 对目标 SKU 构成竞争压力，最终应该展示哪 2-3 个核心竞品”。

竞品不再从全量 SKU 直接 TopN 排名，而是先建立战场竞品库。

### 7.2 候选库生成顺序

对每个目标 SKU：

1. 读取它的主价值战场和机会价值战场。
2. 在每个战场内找候选 SKU。
3. 优先同用户任务、同目标客群。
4. 再用价格带、尺寸段、平台渠道、销量压力、卖点差异收敛。
5. 每个战场形成候选库和候选解释。
6. 从最相关的战场候选库中提炼 2-3 个核心竞品。

候选优先级建议：

| 优先级 | 条件 |
| --- | --- |
| P1 | 同主价值战场 + 同主用户任务 + 同主目标客群 |
| P2 | 同主价值战场 + 同任务或同客群 |
| P3 | 同主价值战场 + 同价格带或相邻价格带 + 同/相邻尺寸 |
| P4 | 同任务/客群，但价值战场不同，作为替代或标杆 |
| P5 | 只有市场压力，没有语义重合，只能作为价格/销量挤压候选 |

### 7.3 评分结构

竞品评分应以业务语义为主，价格、尺寸、卖点为辅：

| 评分组件 | 作用 |
| --- | --- |
| 用户任务重合 | 判断是否满足同一购买目的 |
| 目标客群重合 | 判断是否争夺同一类消费者 |
| 价值战场重合 | 判断竞争语境是否一致 |
| 价格带/尺寸段 | 判断购买替代性和升级/下探关系 |
| 卖点差异 | 判断候选在哪些价值上压制或被目标压制 |
| 市场压力 | 判断销量、销额、渠道和趋势是否形成真实压力 |
| 证据质量 | 判断结论能否展示或需要复核 |

### 7.4 核心竞品选择

最终 2-3 个竞品不是总分 Top3，而是要覆盖不同竞争压力：

| 竞品角色 | 选择逻辑 |
| --- | --- |
| 正面对打 | 同任务、同客群、同战场最强，价格/尺寸接近 |
| 价格/销量挤压 | 市场压力强，价格或销量对目标形成直接挤压 |
| 高端标杆/潜在下探 | 高端卖点或参数显著更强，可能影响目标升级空间 |

如果某个角色没有足够证据，应保留空槽原因，而不是强行补齐。

### 7.5 展示要求

每个入选竞品都要展示：

- 为什么进入候选库。
- 和目标 SKU 共同的用户任务、目标客群和价值战场。
- 它在哪些价格、尺寸、卖点或市场指标上形成压力。
- 证据来自哪些事实域。
- 哪些证据不足或需要复核。

未入选候选也要保留审计原因：

- 任务或客群不一致。
- 战场不一致。
- 价格/尺寸距离太远。
- 卖点证据不足。
- 市场压力不足。
- 质量风险太高。

### 7.6 对现有模块的调整

| 现有模块 | 新定位 | 调整建议 |
| --- | --- | --- |
| M12 | 战场竞品库召回 | 先按战场、任务、客群建库，再用价格/尺寸/市场补充，不应默认接近全量 SKU |
| M13 | 竞品组件和角色评分 | 提高任务/客群/战场权重，价格、尺寸、卖点为辅助解释 |

## 9. 推荐的新执行顺序

### 阶段 A：清洗和事实准备

```text
M00 -> M01 -> M02
```

验收重点：

- 无效评论不下传。
- 服务类评论在 M01 阶段隔离，并阻断出产品分析链路。
- 周销覆盖有解释。
- 质量问题不会伪装成用户事实。

### 阶段 B：SKU 事实画像

```text
M03 -> M04a -> M05 -> M06 -> M04b -> M07 -> M08
```

验收重点：

- 每个 SKU 有参数、市场、卖点、评论四类事实摘要。
- 参数、卖点、评论能互相印证或标记缺失。
- M08 输出的是事实画像，不直接给业务语义下结论。

### 阶段 C：品类语义能力

```text
M08.4 -> M08.5
```

验收重点：

- TV 专用品类任务、客群、战场定义清楚。
- 每个战场有产品锚点和排除条件。
- 服务类维度不进入产品战场。

### 阶段 D：SKU 与维度画像

```text
M09 -> M10 -> M11 -> M11.5 -> M11.6 -> M11.7
```

验收重点：

- 每个 SKU 有主任务、主客群、主战场。
- 每个任务、客群、战场有 SKU 列表和市场画像。
- 主价值战场内的卖点才能参与溢价卖点判断。
- 销量和销额分配守恒。

### 阶段 E：竞品分析和展示

```text
M12 -> M13 -> M14 -> M15 -> M16
```

验收重点：

- 每个 SKU 先有不同价值战场下的竞品库。
- 核心竞品以同任务、同客群为主，以价格、尺寸、卖点为辅。
- 最终展示 2-3 个核心竞品，保留证据和未选原因。
- 发布前必须通过 M16 门禁。

## 10. 对后续实现的直接指导

### 10.1 先修清洗层

当前最先做的是清洗层，不进入 M02/M05 之前先过滤明显无效评论，并在 M01 阶段阻断服务、物流、安装、客服、售后、退换、发票、保修类评论进入产品分析链路。后续可以再统计服务类评论占比，但统计结果只用于决定是否单独建设服务履约背景模块，不再用于决定是否提前拦截。

清洗 CLI 和 Claude Skill 只应该触发 L1，不应该伪装成“全链路分析”。

### 10.2 再重建事实层输出

下一步应让 M08 输出更清楚的 SKU 事实画像：

- 参数事实是否充分。
- 市场事实是否充分。
- 卖点事实是否充分。
- 评论事实是否充分。
- 四者之间如何互相验证。

业务用户问“某个 SKU 是什么样”时，先回答事实画像，而不是直接回答竞品。

### 10.3 然后校准 TV 语义能力

TV 的用户任务、目标客群、价值战场应先形成品类标准，再给 SKU 分配。不能让每个 SKU 直接从 seed 上套所有任务和战场。

### 10.4 最后优化竞品召回

竞品召回应从“全量候选评分”改成“按主价值战场和同任务/同客群建库”。价格、尺寸、卖点用于解释竞争关系和角色差异，不应替代任务、客群、战场。

## 11. 质量门禁总表

| 层 | 可继续下游的条件 | 阻断或降级条件 |
| --- | --- | --- |
| 清洗层 | 产品体验、服务履约、低质无效和质量问题分离 | 大量空评论、质量诊断句、服务履约评论、异常量价未隔离 |
| 事实层 | SKU fact profile 完整，证据和置信度可追溯 | 评论不足、参数冲突、市场窗口不足未标记 |
| 语义能力层 | TV 任务、客群、战场有定义、证据和排除条件 | 维度过宽、服务信号进入产品战场 |
| 画像层 | SKU 和维度双向画像可生成，销量分配守恒 | 主任务/主客群/主战场弱证据却强表达 |
| 竞品层 | 战场竞品库和 2-3 核心竞品有证据卡 | 候选池过宽、只按 TopN、未选原因缺失 |

## 12. 一句话结论

新的分析路径应从“模块流水线”改为“业务对象分层生成”：

```text
先清洗出可信事实，
再形成 SKU 事实画像，
再校准 TV 品类语义能力，
再生成 SKU 与任务/客群/战场的双向画像，
最后按价值战场建立竞品库并提炼 2-3 个核心竞品。
```

这样既保留现有 M00-M16 的工程资产，又避免继续在旧链路上补丁叠补丁。
