# M08.4/M08.5 业务维度边界优化补充设计

## 1. 文档定位

本文是对以下两个详细设计的补充和修订约束：

- `M08_4_comment_native_dimension_discovery_design.md`
- `M08_5_business_dimension_ontology_calibration_design.md`

本文不新增独立业务模块，不替代 M09/M10/M11。它解决 205 真实数据运行后暴露出的核心问题：

1. 当前 M08.4 实现把同一个评论信号同时派生成用户任务、目标客群和价值战场。
2. 候选名称只是加前缀，例如“任务：画质清晰与色彩表现”“客群：游戏流畅与低延迟”“产品价值：儿童护眼与学习陪伴”，导致三类维度边界混淆。
3. 产品价值战场没有稳定绑定参数和卖点锚点，出现只有少数战场有产品锚点，其余战场主要靠评论主题成立的问题。
4. 服务履约、泛化好评和价格感知容易进入产品价值战场，影响后续 SKU 分类和销量分配。

本文的目标是把 M08.4/M08.5 改成可支撑专业市场分析的维度发现和本体校准流程。

## 2. 当前问题诊断

### 2.1 业务问题

当前输出中，“画质清晰”“游戏流畅”“客厅家庭观影升级”“儿童护眼”等词组既可被解释成任务，也可被解释成客群，也可被解释成价值战场。这说明系统没有先识别低层语义要素，而是直接把主题词映射到多个业务维度。

这会导致三个后果：

1. **用户任务不清楚**：不知道用户具体要完成什么事。
2. **目标客群不清楚**：不知道谁在买、谁在用、为什么是这类人。
3. **价值战场不清楚**：不知道 SKU 靠什么产品能力参与竞争。

### 2.2 工程问题

当前实现存在以下模式：

```text
评论命中一个 signal
  -> signal 配置多个 candidate_types
  -> 直接生成 任务/客群/产品价值 的同名候选
```

这个模式必须废弃。后续应改为：

```text
评论、参数、卖点、市场
  -> 抽取原子线索
  -> 组合用户任务候选
  -> 组合目标客群候选
  -> 组合价值战场候选
  -> M08.5 校准并发布当前项目维度定义
```

### 2.3 数据问题

205 当前评论数据足够支持原生线索发现，但不支持把所有高频主题都直接升级为产品价值战场：

- 评论有大量画质、价格、音质、智能、服务履约线索。
- 游戏、体育、卧室、副屏等是较窄场景，不应覆盖大多数 SKU。
- 服务安装高频，但不属于产品核心价值战场。
- “画质好”“不错”“满意”“性价比高”属于弱语义，需要参数、卖点或市场对比补强。

## 3. 专业定义和方法

### 3.1 用户任务

用户任务回答：

```text
用户在什么场景下，买这台电视要完成什么事，并期待什么结果。
```

分析方法采用 JTBD 口径：

```text
场景/触发条件 + 动作任务 + 期望结果
```

示例：

| 合格用户任务 | 不合格写法 | 原因 |
| --- | --- | --- |
| 客厅大屏追剧观影 | 画质清晰 | 画质是产品体验，不是任务 |
| 主机游戏低延迟娱乐 | 高刷低延迟 | 高刷是能力，不是任务 |
| 体育赛事流畅观看 | 运动流畅 | 运动流畅是体验结果，需要补场景 |
| 给长辈买一台易操作电视 | 长辈易用 | 长辈是人群，易用是价值，需要组合成任务 |
| 新家装修配一台大屏电视 | 家居美学 | 家居美学是价值方向，不是任务 |

### 3.2 目标客群

目标客群回答：

```text
谁会买，谁会用，为什么是这类人买。
```

分析方法采用 STP 分群口径：

```text
人群/家庭阶段/决策角色 + 核心任务 + 预算/渠道/购买动机 + 产品适配证据
```

示例：

| 合格目标客群 | 不合格写法 | 原因 |
| --- | --- | --- |
| 家庭客厅升级用户 | 客厅家庭观影升级 | 后者是任务/场景，不是人群 |
| 主机游戏和年轻娱乐用户 | 游戏流畅与低延迟 | 后者是产品体验，不是人群 |
| 体育赛事重度观看用户 | 体育赛事流畅观看 | 后者是任务，不是人群 |
| 长辈家庭代购用户 | 长辈易用 | 需要说明购买/使用角色 |
| 有儿童家庭用户 | 儿童护眼 | 需要说明家庭阶段和使用者 |
| 新家装修用户 | 新家装修搭配 | 后者是任务，不是客群 |
| 价格敏感换新用户 | 性价比 | 需要说明预算和换新动机 |

### 3.3 价值战场

价值战场回答：

```text
SKU 靠什么产品价值参与竞争，并在哪些参数、卖点和市场事实上可被验证。
```

分析方法采用价值主张和竞争定位口径：

```text
产品能力组合 + 参数锚点 + 卖点锚点 + 任务/客群相关性 + 市场/竞品对比
```

示例：

| 合格价值战场 | 不合格写法 | 原因 |
| --- | --- | --- |
| 大屏沉浸画质战场 | 客厅家庭观影升级 | 后者是任务场景，战场要落到产品能力 |
| 高刷低延迟游戏战场 | 游戏用户 | 后者是客群 |
| 运动流畅观看战场 | 体育赛事观看 | 后者是任务 |
| 智能交互易用战场 | 长辈家庭用户 | 后者是客群 |
| 护眼舒适观看战场 | 儿童家庭用户 | 后者是客群 |
| 声画沉浸战场 | 影院观影 | 后者是任务场景 |
| 大屏价值性价比战场 | 价格划算 | 后者是购买动机，必须加尺寸/配置/价格对比 |

## 4. 三类维度的判别规则

### 4.1 判别优先级

任意文本或信号进入系统后，必须先判断它属于哪类原子线索：

| 原子线索 | 示例 | 首选下游 |
| --- | --- | --- |
| 人群 | 老人、父母、孩子、学生、单位、公司 | 目标客群 |
| 决策角色 | 给爸妈买、给孩子看、公司采购、自己玩游戏 | 目标客群、用户任务 |
| 场景 | 客厅、卧室、新家、出租屋、宿舍、看球、游戏 | 用户任务 |
| 动作任务 | 追剧、观影、看球、玩游戏、投屏、学习陪伴 | 用户任务 |
| 期望结果 | 流畅、不拖影、清楚、不刺眼、操作简单 | 用户任务、价值战场 |
| 产品能力 | 高刷、亮度、控光、音响、护眼、语音、系统 | 价值战场 |
| 参数 | 120Hz、Mini LED、HDMI2.1、低蓝光、85 寸 | 价值战场 |
| 卖点 | 分区控光、AI 画质、远场语音、杜比音效 | 价值战场 |
| 价格动机 | 国补、优惠、性价比、预算 | 购买动机、价值性价比战场弱证据 |
| 服务履约 | 安装、配送、师傅、售后 | 服务语境，不进入产品战场 |

### 4.2 禁止规则

后续实现必须禁止以下直接映射：

| 禁止映射 | 原因 |
| --- | --- |
| 产品体验词直接生成目标客群 | “高刷”“画质”不是人 |
| 人群词直接生成价值战场 | “长辈”“儿童”不是产品能力 |
| 服务词生成产品价值战场 | 安装配送不是产品价值 |
| 价格词单独生成产品战场 | 价格是市场/动机，必须结合配置和销量 |
| 泛化好评生成任务/客群/战场 | “不错、满意、值得”缺少业务含义 |
| 一个 signal 同名生成三类维度 | 会造成任务、客群、战场不可区分 |

### 4.3 允许一源多用，但必须不同组合

同一个原子线索可以参与多类维度，但不能用同一个名称直接派生。

示例：

```text
评论：给爸妈买的，语音操作方便，画面清楚。
```

应生成：

| 维度 | 输出 |
| --- | --- |
| 人群线索 | 父母/长辈 |
| 任务候选 | 给长辈买一台易操作电视 |
| 客群候选 | 长辈家庭代购用户 |
| 价值战场候选 | 智能交互易用战场、基础画质体验 |
| 参数锚点要求 | 语音、遥控、系统易用、开机广告、字体/模式等 |

不能生成：

```text
任务：长辈易用
客群：长辈易用
产品价值：长辈易用
```

## 5. 参数锚点设计

### 5.1 产品价值战场必须有参数或卖点锚点

M08.4/M08.5 后续应为每个产品价值战场维护 `battlefield_anchor_spec`。每个战场定义：

- 必需锚点。
- 强锚点。
- 弱锚点。
- 排除锚点。
- 可比池。
- 参数缺失时的降级策略。

### 5.2 MVP 战场锚点表

| 价值战场 | 必需或强参数锚点 | 强卖点锚点 | 评论只可作为 |
| --- | --- | --- | --- |
| 大屏沉浸画质战场 | 尺寸、分辨率、亮度、分区控光、Mini LED/OLED/QLED、色域、HDR、画质芯片 | 高画质、AI 画质、控光、广色域、影院画质 | 感知验证 |
| 高刷低延迟游戏战场 | 刷新率、HDMI2.1、VRR、ALLM、MEMC、游戏模式、输入延迟 | 游戏模式、高刷、低延迟、主机适配 | 场景验证 |
| 运动流畅观看战场 | 刷新率、MEMC、运动补偿、响应速度、拖影控制 | 体育模式、运动补偿、流畅观看 | 场景验证 |
| 智能交互易用战场 | 系统、芯片、内存、存储、语音、远场语音、遥控器、开机广告 | 智能语音、系统流畅、老人易用、投屏 | 易用验证 |
| 护眼舒适观看战场 | 低蓝光、无频闪、护眼认证、环境光调节、亮度调节、儿童模式 | 护眼、低蓝光、防频闪、儿童护眼 | 感知验证 |
| 声画沉浸战场 | 扬声器功率、声道、低音、杜比视界、杜比全景声、DTS、音响系统 | 杜比音效、影院音响、环绕声 | 感知验证 |
| 大屏价值性价比战场 | 尺寸、价格、价格/英寸、关键配置强度、销量分位、促销 | 大屏高配低价、国补价、同价位配置优势 | 价值验证 |
| 家居空间适配战场 | 尺寸、厚度、边框、挂装、底座、外观、壁纸屏/艺术模式 | 超薄、全面屏、家居融合 | 场景验证 |

说明：

1. 评论不能单独支撑正式产品战场。
2. 参数或卖点缺失时，战场最多为 `candidate_review`，不能进入 `allocation_eligible`。
3. 如果参数名称在真实数据中不标准，M03/M08 必须提供标准参数映射后才能作为锚点。

### 5.3 参数锚点评分

建议 `product_anchor_score` 按以下方式计算：

```text
product_anchor_score =
  required_anchor_hit_ratio * 0.35
  + strong_anchor_hit_ratio * 0.25
  + claim_anchor_hit_ratio * 0.20
  + same_pool_advantage_score * 0.20
```

其中：

- `required_anchor_hit_ratio`：必需锚点命中比例。
- `strong_anchor_hit_ratio`：强锚点命中比例。
- `claim_anchor_hit_ratio`：卖点锚点命中比例。
- `same_pool_advantage_score`：在相同尺寸、相近价格、相同渠道可比池内的参数优势。

如果 `product_anchor_score < 0.30`，产品价值战场不得发布为正式可分配维度。

## 6. M08.4 修订后的处理流程

### 6.1 输入读取

只读取以下上游结果：

1. M05 可下游消费评论句。
2. M06 评论下游信号。
3. M08 SKU 综合画像、证据矩阵和 feature view。
4. M03/M04b/M07 的必要锚点明细，优先通过 M08 汇总读取。

### 6.2 原子线索抽取

新增原子线索层，建议输出到 `core3_comment_native_signal` 或后续新表中。

字段建议：

| 字段 | 说明 |
| --- | --- |
| `signal_code` | 原子线索 code，例如 `person_senior`、`scene_living_room`、`capability_high_refresh` |
| `signal_family` | person、role、scene、action、outcome、capability、param_anchor、claim_anchor、price_motive、service、risk |
| `source_evidence_ids` | 来源评论/参数/卖点/市场 evidence |
| `sku_count` | 覆盖 SKU 数 |
| `sentence_count` | 评论句数 |
| `strong_sentence_count` | 强证据句数 |
| `support_score` | 原子线索支撑分 |
| `eligible_for_task` | 是否可支撑任务 |
| `eligible_for_target_group` | 是否可支撑客群 |
| `eligible_for_battlefield` | 是否可支撑战场 |
| `exclusion_reason` | 服务、低价值、泛化、缺产品锚点等 |

### 6.3 用户任务候选组合

任务候选由以下组合生成：

```text
scene/action/motive + action/outcome + product_experience/capability
```

示例规则：

| 规则 | 输出任务 |
| --- | --- |
| 客厅/全家/追剧/观影 + 大屏/画质/音质 | 客厅大屏追剧观影 |
| 游戏/主机 + 高刷/低延迟/HDMI | 主机游戏低延迟娱乐 |
| 看球/体育/赛事 + 流畅/不拖影/MEMC | 体育赛事流畅观看 |
| 父母/长辈 + 语音/简单/遥控 | 给长辈买一台易操作电视 |
| 新家/装修/挂墙 + 大屏/外观/尺寸 | 新家装修配大屏电视 |
| 孩子/儿童 + 护眼/低蓝光/学习 | 儿童长期观看护眼陪伴 |

任务输出必须采用“动作型”名称，不使用单纯产品能力词。

### 6.4 目标客群候选组合

客群候选由以下组合生成：

```text
person/role/life_stage + task + price/channel/motive + product_fit
```

示例规则：

| 规则 | 输出客群 |
| --- | --- |
| 家庭/全家 + 客厅观影 + 大屏/音画 | 家庭客厅升级用户 |
| 游戏/主机 + 高刷低延迟任务 + 高刷参数 | 主机游戏和年轻娱乐用户 |
| 看球/赛事 + 运动流畅任务 | 体育赛事重度观看用户 |
| 父母/长辈 + 易用任务 + 语音/遥控/系统 | 长辈家庭代购用户 |
| 孩子/儿童 + 护眼任务 + 护眼参数 | 有儿童家庭用户 |
| 新家/装修 + 空间适配任务 + 外观/尺寸 | 新家装修用户 |
| 国补/预算/性价比 + 换新任务 + 价格分位 | 价格敏感换新用户 |

客群输出必须采用“人群型”名称，不使用纯场景或产品能力词。

### 6.5 价值战场候选组合

战场候选由以下组合生成：

```text
capability_bundle + param_anchor + claim_anchor + task_relevance + market_comparison
```

示例规则：

| 规则 | 输出战场 |
| --- | --- |
| 大屏 + 画质参数 + 观影评论 + 同池优势 | 大屏沉浸画质战场 |
| 高刷/HDMI/VRR + 游戏评论 + 同池优势 | 高刷低延迟游戏战场 |
| MEMC/刷新率 + 体育评论 + 拖影评价 | 运动流畅观看战场 |
| 语音/系统/遥控 + 易用评论 | 智能交互易用战场 |
| 护眼参数 + 儿童/长期观看评论 | 护眼舒适观看战场 |
| 音响/杜比 + 音质评论 | 声画沉浸战场 |
| 尺寸/配置 + 价格分位 + 销量分位 | 大屏价值性价比战场 |

战场输出必须采用“产品价值型”名称，不使用纯人群或纯任务名称。

### 6.6 服务和风险剥离

服务履约、安装配送、售后、客服、物流只允许进入：

- `service_context`
- `risk_context`
- M15 的履约体验展示
- 新家装修任务/客群的弱侧面证据

不得进入：

- `native_product_value_battlefield`
- M11 正式价值战场
- M11.6 产品价值战场销量分配

## 7. M08.5 修订后的本体发布规则

### 7.1 本体定义字段

M08.5 发布的每个维度必须包含：

| 字段 | 说明 |
| --- | --- |
| `dimension_type` | task、target_group、battlefield、service_context、risk_context、purchase_motive |
| `dimension_code` | 当前项目稳定 code |
| `dimension_name_cn` | 业务中文名 |
| `business_definition_cn` | 明确说明该维度是什么 |
| `positive_criteria_json` | 成立条件 |
| `negative_criteria_json` | 排除条件 |
| `required_signal_families_json` | 必需原子线索类型 |
| `required_anchor_json` | 参数/卖点/市场锚点，仅战场必需 |
| `allowed_evidence_domains_json` | 可使用的证据域 |
| `allocation_eligible_rule_json` | 是否可进入 M11.6 分配 |
| `source_native_dimension_ids` | 来源 M08.4 原生候选 |
| `seed_alignment_action` | keep、rename、split、merge、downgrade、new、disable |
| `review_status` | active、active_with_warning、review_required、disabled |

### 7.2 发布准入

| 维度类型 | 发布为 active 的最低条件 |
| --- | --- |
| 用户任务 | 有场景/动作/结果中的至少两类证据，且不是纯产品能力词 |
| 目标客群 | 有人群/角色/家庭阶段线索，且能关联至少一个任务或购买动机 |
| 价值战场 | 有产品能力组合，且 `product_anchor_score >= 0.30` |
| 服务语境 | 服务词稳定高频，且明确标记不进入产品战场 |
| 风险语境 | 负向体验或履约问题稳定出现 |
| 购买动机 | 价格、补贴、换新、装修等动机稳定出现 |

### 7.3 禁用和降级

满足以下任一条件时，不能发布为正式可分配维度：

- 覆盖 SKU 超过 70%，但强证据 SKU 低于 35%，属于过宽维度。
- 价值战场缺参数和卖点锚点。
- 主要证据来自服务履约。
- 主要证据来自低价值评论。
- 任务名称是纯产品能力。
- 客群名称没有人群或决策角色。
- 战场名称没有产品能力或竞争价值。

## 8. 对 M09/M10/M11 的接口影响

### 8.1 M09 用户任务

M09 不再直接使用 seed 默认任务或 M06 宽泛 task cue 做全量判断。M09 应优先消费 M08.5 发布的 `task` 维度定义。

M09 计算 SKU 任务时必须验证：

```text
任务线索 + 任务相关产品体验 + 任务相关市场/参数/卖点弱支撑
```

任务可以没有强参数锚点，但不能只有泛化好评。

### 8.2 M10 目标客群

M10 不再把 M09 默认映射客群当结论。M10 应优先消费 M08.5 发布的 `target_group` 维度定义。

M10 计算 SKU 客群时必须验证：

```text
人群/角色/家庭阶段 + 任务 + 价格/渠道/购买动机或产品适配
```

如果没有人群或决策角色线索，只能作为弱客群或不适用。

### 8.3 M11 价值战场

M11 不再把任务或客群直接映射为战场。M11 应优先消费 M08.5 发布的 `battlefield` 维度定义。

M11 计算 SKU 战场时必须验证：

```text
产品能力组合 + 参数锚点 + 卖点锚点 + 评论感知 + 市场/竞品可比
```

如果缺参数/卖点锚点，不能进入主战场，只能进入 `candidate_review`。

### 8.4 M11.6 销量分配

M11.6 只能对 M09/M10/M11 输出的 `allocation_eligible=true` 结果做销量分配。

不允许把以下结果纳入分配：

- weak 但无业务准入的候选。
- 服务语境。
- 泛化好评。
- 缺参数锚点的产品战场候选。
- 没有人群证据的目标客群。

## 9. 评分和稀疏化规则

### 9.1 维度候选评分

建议 M08.4 对候选维度使用以下结构：

```text
dimension_candidate_score =
  native_signal_support * 0.20
  + evidence_diversity_score * 0.15
  + specificity_score * 0.15
  + product_anchor_score * battlefield_weight
  + task_or_group_fit_score * task_group_weight
  + market_support_score * 0.15
  + distinctiveness_score * 0.15
```

其中：

| 维度类型 | battlefield_weight | task_group_weight |
| --- | ---: | ---: |
| 用户任务 | 0.10 | 0.25 |
| 目标客群 | 0.10 | 0.25 |
| 价值战场 | 0.30 | 0.05 |

### 9.2 SKU 级稀疏化

对每个 SKU：

- 用户任务建议最多保留 1 个主任务、2 个次任务、若干弱候选。
- 目标客群建议最多保留 1 个主客群、2 个次客群、若干弱候选。
- 价值战场建议最多保留 1 个主战场、2 个次战场、若干机会候选。

如果一个 SKU 在所有维度上得分接近，必须生成 `dimension_not_distinctive` 复核问题，而不是平均分配。

### 9.3 全局过宽检查

每个维度都要计算：

```text
sku_coverage_ratio
strong_sku_ratio
allocation_eligible_sku_ratio
top_sku_concentration
anchor_overlap_jaccard
```

如果一个维度覆盖几乎全部 SKU，必须解释原因：

- 是真实基础门槛，例如“基础画质体验”。
- 是低区分度维度，应降级为背景指标。
- 是规则过宽，需要拆分或收紧。

## 11. 数据契约补充

### 11.1 新增或扩展规则配置

```text
apps/api-server/app/rules/tv_core3_dimension_boundary_v1.json
```

配置内容：

| 配置段 | 用途 |
| --- | --- |
| `atomic_signal_rules` | 原子线索抽取规则 |
| `task_assembly_rules` | 用户任务组合规则 |
| `target_group_assembly_rules` | 目标客群组合规则 |
| `battlefield_anchor_specs` | 价值战场参数/卖点锚点 |
| `service_exclusion_rules` | 服务履约剥离 |
| `low_value_exclusion_rules` | 低价值评论剥离 |
| `dimension_publish_thresholds` | M08.5 发布阈值 |

### 11.2 结果表兼容策略

短期可以复用现有表：

- `core3_comment_native_signal`
- `core3_native_dimension_candidate`
- `core3_native_dimension_sku_support`
- `core3_native_dimension_alignment_proposal`
- `core3_native_dimension_review_issue`

但字段语义必须调整：

1. `core3_comment_native_signal` 存原子线索，不存已命名的业务维度。
2. `core3_native_dimension_candidate` 存由原子线索组合出来的任务、客群、战场候选。
3. `core3_native_dimension_sku_support` 必须包含 task/group/battlefield 各自不同的支撑拆解。
4. `product_anchor_score` 对价值战场必须来自参数/卖点锚点，不再只用英文关键词粗匹配。

## 12. 开发任务建议

后续开发建议按一个完整模块任务完成，不再拆成微任务。

### 12.1 M08.4 优化任务

目标：

1. 将现有 `NATIVE_SIGNAL_RULES` 改成 `atomic_signal_rules`。
2. 删除同一个 signal 直接配置多个 `candidate_types` 的模式。
3. 新增任务、客群、战场三个 assembly service。
4. 新增产品价值战场参数锚点评分。
5. 服务履约、低价值评论、泛化好评进入 exclusion/review。
6. 205 或本地样例运行后，输出原子线索、候选维度、SKU 支撑和复核问题。

验收：

- 任务、客群、战场名称不再同名。
- 价值战场必须有参数/卖点锚点拆解。
- 服务履约不进入产品战场。
- 游戏、体育、儿童、长辈等窄维度不再覆盖几乎全部 SKU。

### 12.2 M08.5 优化任务

目标：

1. 按本文定义发布 task、target_group、battlefield 三套清晰本体。
2. 输出每个维度的成立条件、排除条件、锚点要求。
3. 为 M09/M10/M11 提供不同消费入口。
4. 标记 `profile_eligible` 和 `allocation_eligible`。

验收：

- M08.5 active 维度中，任务是动作型，客群是人群型，战场是产品价值型。
- 每个价值战场都有参数/卖点锚点规格。
- 下游不能把服务语境和泛化好评纳入销量分配。

### 12.3 M09/M10/M11 适配任务

目标：

1. M09 使用 task 本体和任务组合规则。
2. M10 使用 target_group 本体和客群组合规则。
3. M11 使用 battlefield 本体和参数锚点规格。
4. M11.6 只消费 `allocation_eligible=true` 的结果。

验收：

- 每个 SKU 可输出主/次/弱任务、主/次/弱客群、主/次/机会战场。
- 不再出现所有 SKU 命中所有正式战场和正式客群。
- M11.7 对账时，维度销量分配既能守恒，也能解释每个维度下的 SKU 构成。

## 13. 验收标准

本补充设计落地后，205 真实数据运行应满足：

1. M08.4 能输出原子线索分布，而不是只输出同名前缀维度。
2. M08.5 能发布清晰区分的任务、客群、战场定义。
3. 每个正式价值战场都有参数/卖点锚点说明。
4. 服务安装、配送、售后不再进入产品价值战场。
5. 游戏、体育、儿童、长辈等窄维度不会覆盖大多数 SKU，除非有充分参数、评论和市场证据。
6. M09/M10/M11 的正式画像结果必须稀疏，候选可全量，但可分配维度不能全量铺满。
7. M11.6/M11.7 的销量对账不只看数学守恒，还要能解释某个战场、客群、任务由哪些 SKU 构成、各自占多少、为什么占这些。
