# CatForge 彩电核心三竞品 SOP 详细设计分步执行总纲

## 1. 文档目标

本文定义 M00-M16 SOP 需求进入详细设计阶段的执行方式。详细设计阶段的目标不是继续扩写需求，也不是直接写代码，而是把已评审的模块需求转换成可以拆开发任务、写迁移、写服务、写 API、写测试的工程设计。

后续详细设计必须逐一生成，每次只处理一个文档，不允许一次批量生成多个模块详细设计。

## 2. 输入依据

详细设计必须以以下材料为准：

1. `docs/core3_mvp/real_data_v2/sop_requirements/` 下的 M00-M16 模块需求文档。
2. `docs/core3_mvp/real_data_v2/sop_requirements/00_real_data_baseline.md`。
3. `cankao/CatForge_竞品生成SOP_详细指导_v1.md`。
4. `cankao/CatForge_核心竞品展示页_UI设计规范_v1.md`。
5. `cankao/catforge_sop_md/modules/` 下的模块参考文档。
6. 当前 205 PostgreSQL 真实样例数据基线。
7. 已确认的数据分层：原始表、原始登记层、清洗规范层、证据原子层、抽取特征层、画像推导层、竞品结果层、治理验收层。

如果详细设计与需求文档冲突，以需求文档为准；如果发现需求文档内部存在冲突，应在详细设计文档中记录“待评审问题”，不能私自改口径。

## 3. 详细设计总原则

### 3.1 不直接开发

详细设计阶段只生成文档，不改代码、不写迁移、不部署。

每个模块详细设计完成后，才进入开发任务拆分。开发任务拆分至少要等总体架构和 M00-M02 详细设计完成后再开始。

### 3.2 逐模块生成

后续生成顺序固定：

```text
00 总体架构与数据字典
-> M00 原始数据批次与行登记
-> M01 清洗规范化与质量诊断
-> M02 Evidence 原子层
-> M03A 品类参数语义资产生成
-> M03B 参数字段画像与 SKU 参数事实画像
-> M04a 基础卖点激活
-> M05 评论基础证据层
-> M06 评论下游信号抽取层
-> M04b 评论验证增强
-> M07 市场画像与可比池基线
-> M08 SKU 综合信号画像
-> M08.4 评论原生业务维度发现
-> M08.5 业务维度本体融合发布
-> M09 用户任务模块
-> M10 目标客群模块
-> M11 价值战场模块
-> M11.5 战场内卖点价值分层
-> M12 候选池召回模块
-> M13 竞品组件评分模块
-> M14 三槽位核心竞品选择
-> M15 证据卡与高层报告
-> M16 增量任务编排、复核和验收
```

每次只能新增或强化当前顺序中的一个详细设计文档。

### 3.3 保持模块边界

详细设计必须保持 M00-M16 的职责边界：

- M00 只做原始登记和增量边界。
- M01 只做清洗和质量诊断。
- M02 只做 evidence 原子。
- M03-M06/M04b 只做参数、卖点、评论相关抽取。
- M07-M11.5 只做市场、SKU、任务、客群、战场和卖点价值画像。
- M12-M14 只做候选、评分和三槽位选择。
- M15 只做证据卡和高层报告 payload。
- M16 只做任务编排、复核、验收和发布门禁。

下游不能为了实现方便直接读取原始表做业务判断。

### 3.4 真实数据约束必须落地

详细设计必须明确处理当前真实样例数据约束：

- 当前原始表为 `week_sales_data`、`attribute_data`、`selling_points_data`、`comment_data`。
- 当前样例共 35 个量价型号，品牌均为海信，同品牌 SKU 可以互为竞品。
- 当前只有线上渠道，平台为专业电商和平台电商，不能输出线下结论。
- `attribute_data` 有较多 unknown、空值和 `-`，缺失不能当 false。
- `selling_points_data` 只覆盖 5 个型号，85E7Q 没有结构化卖点，不能伪造宣传证据。
- `comment_data` 行数大但重复和拆行明显，必须以去重评论和句级 evidence 为基础。

## 4. 详细设计文档统一结构

每个模块详细设计文档必须包含以下章节。

### 4.1 模块职责

说明本模块解决什么工程问题、不解决什么问题、对应需求文档是哪一份。

必须写清：

- 模块输入边界。
- 模块输出边界。
- 哪些事情不能在本模块做。
- 是否允许复用历史结果。

### 4.2 数据模型设计

必须落到字段级。

每张表至少写：

- 表名。
- 表用途。
- 字段名。
- 字段类型建议。
- 是否必填。
- 主键。
- 唯一约束。
- 外键或逻辑引用。
- 索引。
- JSON 字段结构。
- 状态枚举。
- hash 字段和版本字段。
- 与上下游表的关系。

如果某张表只是 MVP 可选表，也必须说明是否首版落地、为什么可选、后续如何补。

### 4.3 处理流程设计

必须写成可实现的流程，而不是概念说明。

至少包含：

- 输入读取步骤。
- 数据过滤和校验。
- 核心转换逻辑。
- 置信度或质量状态计算。
- 复核问题生成。
- 输出写入。
- 幂等和重跑策略。
- 失败处理。

复杂流程必须用 mermaid 或编号步骤表达。

### 4.4 增量策略

必须说明：

- 哪些输入变化触发本模块重算。
- 如何识别新增、变化、失效和复用。
- 输出 hash 如何计算。
- 历史版本如何保留。
- 哪些下游模块会被触发。

### 4.5 服务和任务边界

详细设计要为后续开发拆分留接口。

至少说明：

- 后端 service 名称建议。
- repository 或 DAO 边界。
- batch runner 或 task runner 边界。
- 是否需要 API。
- API 是业务展示用、运营查看用，还是内部调度用。
- 不允许前端重新拼业务结论的地方。

### 4.6 质量和复核规则

必须落到可执行条件。

包括：

- warning 条件。
- review_required 条件。
- blocked 条件。
- 质量指标。
- 复核队列字段。
- 人工复核后是否触发重算。

### 4.7 测试设计

每个模块必须写测试点。

至少包含：

- 正常输入。
- 缺失输入。
- 重复输入。
- unknown/null/空字符串/`-`。
- 低置信或样本不足。
- 增量重跑。
- 证据追溯。
- 不允许越界读取原始表。
- 85E7Q 样例约束。

测试不依赖外部 LLM 调用。需要 LLM 的地方必须设计 mock、规则替身或离线 fixture。

### 4.8 验收标准

验收标准必须能支持后续开发任务验收。

至少包括：

- 数据表是否能落地。
- 输出是否可被下游消费。
- evidence 是否可追溯。
- 置信度和复核状态是否正确。
- 增量是否可重跑。
- 当前 205 样例数据是否能跑通。

## 6. 00 总体架构与数据字典设计要求

第一个详细设计文档必须先做 `00_architecture_data_dictionary_design.md`。它不是某个业务模块，而是全链路工程地基。

必须包含：

1. 总体架构图。
2. 数据层级图。
3. M00-M16 模块 DAG。
4. 最终表清单。
5. 统一字段规范。
6. 通用枚举。
7. 通用主键和唯一键规则。
8. 通用 evidence 引用方式。
9. 通用 hash 和版本规则。
10. 通用运行状态和复核状态。
11. API 分层原则。
12. 前端页面边界。
13. 测试分层策略。

00 文档完成后，后续模块详细设计必须复用其中的命名和通用契约。

## 7. 分步执行规则

每次生成详细设计时必须按以下步骤执行：

1. 查看 `sop_detailed_design/` 已有文档。
2. 按清单找出下一个未生成或需要强化的文档。
3. 读取对应需求文档。
4. 必要时读取 cankao 参考文档和真实数据基线。
5. 只编辑当前详细设计文档。
6. 不改代码、不部署。
7. 完成后说明本次处理的文档、主要内容、下次应处理的文档。

如果发现前序详细设计存在命名冲突或接口冲突，不应直接大规模修改多个文件。应在当前文档中记录冲突，并在下一轮单独处理需要修订的文档。

## 8. 质量门槛

一个模块详细设计达到可进入开发任务拆分，需要满足：

- 表结构足够写 Alembic migration。
- 字段契约足够写 Pydantic schema。
- 处理流程足够写 service 和 runner。
- 增量规则足够写幂等测试。
- 复核规则足够写 review queue。
- API 边界足够写前后端联调任务。
- 测试点足够拆单元测试和集成测试。
- 与上下游模块关系清楚，不需要开发时再猜接口。

未达到这些标准的详细设计只能继续强化，不能进入开发任务拆分。

## 9. 阶段完成判定

详细设计阶段完成的判定：

1. `00_architecture_data_dictionary_design.md` 完成并统一命名。
2. M00-M16 每个模块都有详细设计文档。
3. 所有模块详细设计都引用需求文档中的真实数据约束。
4. 全链路表、服务、任务、API、前端展示边界清楚。
5. 可以据此生成开发任务拆分文档。

完成后下一阶段不是直接部署，而是生成开发任务拆分和实施顺序。
