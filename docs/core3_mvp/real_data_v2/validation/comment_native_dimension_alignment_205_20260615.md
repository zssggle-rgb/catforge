# 205 真实评论原生线索与预设维度对照分析

生成时间：2026-06-15

数据范围：205 测试环境 PostgreSQL，批次 `m00_20260613004311_d548f6dc`。

分析目的：不再从预设价值战场、目标客群、用户任务直接套标签，而是先看当前真实评论数据天然在表达什么，再反向校准 M09 用户任务、M10 目标客群、M11 价值战场和 M11.6 销量分配逻辑。

## 1. 结论摘要

当前问题不是 M11.7 数学平账问题，而是 M06-M11 的业务维度推导过宽：

- M06 评论信号已经把大量通用评论映射到几乎所有预设任务、客群和战场。
- M09-M11 虽然大多只是 `weak` 或 `insufficient`，但 M11.6 把所有正分维度都归一化参与销量分配。
- 结果导致 84 个 SKU 都进入 10 个价值战场、9 个目标客群和 10 个用户任务的分配矩阵。
- 这种矩阵可以做到销量守恒，但不能说明业务判断正确。

正确方向应是：

1. 先抽取评论原生语义：谁、什么场景、买来干什么、为什么买、体验什么、服务如何。
2. 再把原生语义对齐到预设任务、客群、战场。
3. 只有满足证据门槛的维度才进入 SKU 画像和销量分配。
4. 服务履约从产品价值战场中剥离，只作为服务体验、履约风险或购买障碍，不参与产品战场销量分配。

## 5. 从问题本身重新定义推导逻辑

### 5.1 用户任务应该如何分析

用户任务回答：买这台电视主要为了完成什么使用任务。

推荐输入：

- 评论原生场景：客厅、卧室、新家、追剧、看电影、看球、游戏、儿童看动画、给父母看。
- 评论原生动机：性价比、换新、国补、尺寸合适、画质升级、操作简单。
- 参数/卖点：尺寸、刷新率、HDMI、护眼、语音、音响、亮度、背光。
- 市场数据：尺寸段、价位段、销量、渠道。

任务成立规则：

1. 任务候选可以从 seed、卖点、参数、评论、市场共同召回。
2. 任务成立必须有至少一个明确任务锚点：
   - 场景词，如卧室、客厅、看球、游戏、新家；
   - 或动作词，如换新、追剧、看电影、打游戏；
   - 或强参数卖点，如高刷+HDMI 支撑游戏，护眼+儿童支撑儿童观看。
3. 单纯“画质清晰、性价比高、安装好”不能推导所有任务。
4. 弱任务可以保留为候选，但不能进入销量分配。

### 5.2 目标客群应该如何分析

目标客群回答：谁会买，为什么是这类人买。

推荐输入：

- 人群词：老人、父母、孩子、家人、自己、单位采购。
- 场景词：客厅、卧室、新家、公司、宿舍。
- 任务结果：SKU 主/次任务。
- 价格渠道：价位、促销、平台、销量。
- 产品证据：是否有对应参数/卖点能支撑该客群。

客群成立规则：

1. 客群不是任务的自动映射。
2. 必须有“人群线索 + 任务/场景 + 价格或产品证据”三者中的至少两类。
3. “家庭泛化”不能直接等同家庭换新用户。
4. “老人”评论可以支持长辈家庭用户，但只有语音、简洁系统、适老、少广告等产品证据时，才能加强为长辈易用战场。
5. 单位/公司采购应单独成为 B 端/工程采购线索，不能混入家庭客群。

### 5.3 价值战场应该如何分析

价值战场回答：SKU 靠什么产品价值参与竞争。

推荐输入：

- 主任务和主客群。
- 卖点价值层：高刷、HDMI、Mini LED、亮度、控光、音效、护眼、语音等。
- 参数强弱：同尺寸、同价位、同战场候选池内的参数相对优势。
- 评论产品体验：画质、音质、游戏、体育、护眼、智能易用。
- 市场验证：销量、销额、价格带、同池表现。

战场成立规则：

1. 必须有产品价值锚点，不能只有服务履约或市场背景。
2. 必须能解释“为什么这个 SKU 在这个战场竞争”，不能只是评论里出现过关键词。
3. 战场应从任务/客群/卖点/参数/市场共同收敛，而不是从 seed 全量遍历后直接分配。
4. 弱战场用于提示和复核，不用于核心竞品召回和销量分配。

## 6. 预设本体优化建议

### 6.1 保留但收紧的战场

| 战场 | 当前问题 | 优化建议 |
| --- | --- | --- |
| 大屏性价比 | 覆盖全部 SKU | 必须同时满足尺寸/价格/价值评论或市场价位优势 |
| 家庭观影升级 | 与画质、音质、客厅场景混用 | 必须有客厅/观影/家庭使用场景，或大屏+音画体验组合 |
| 高端画质 | 被“画质清晰”泛化 | 必须有高端参数或高端卖点，如 Mini LED、HDR、控光、高亮、OLED、量子点 |
| 影院音效 | 被普通“声音不错”泛化 | 必须有音响参数、杜比、沉浸音效，或强音质评论 |
| 游戏体育 | 游戏和体育都是窄场景 | 可保留一个战场，但任务层应拆游戏、体育；战场成立必须有高刷/HDMI/MEMC/低延迟等证据 |
| 智能系统体验 | 容易和长辈易用混淆 | 只保留系统流畅、语音、投屏、应用生态等产品体验 |
| 家庭护眼 | 评论覆盖较弱 | 必须有护眼参数/卖点或儿童/长时间观看证据 |
| 家居美学 | 被安装、挂墙污染 | 只保留外观、超薄、边框、家居搭配；安装服务剥离 |
| 长辈易用 | 当前不应作为多数产品战场 | 只有语音、简洁系统、少广告、适老模式等产品证据充分时成立 |

## 7. 推荐的新处理链路

### 7.1 新增“评论原生语义层”

建议在 M06 或 M06 后新增一个轻量中间层，用于保存评论原生候选，不直接产出最终任务/客群/战场。

候选字段：

- `native_signal_type`：person、scene、motive、product_experience、service_experience、risk。
- `native_signal_code`：如 senior_family、bedroom_second_tv、value_price、picture_quality、installation_service。
- `sentence_count`、`sku_count`、`strong_sentence_count`。
- `representative_phrases`。
- `source_topic_codes`。
- `service_guardrail_flag`。
- `eligible_for_product_dimension`。

### 7.2 M09 用户任务改造

M09 不再把所有 seed 任务全量作为可分配任务。

流程：

1. 从评论原生场景/动机召回任务候选。
2. 用参数和卖点补强任务证据。
3. 用市场价位、尺寸、渠道验证任务合理性。
4. 只输出候选分数；另输出 `eligible_for_profile=true/false`。
5. 只有主/次任务进入 M11.6 销量分配。

### 7.3 M10 目标客群改造

M10 不再从 M09 默认客群映射直接形成客群。

流程：

1. 从评论原生人群和场景抽客群候选。
2. 用任务结果判断购买动机。
3. 用价格渠道和市场表现判断“这类人为什么会买”。
4. 对家庭泛化、服务泛化、低价值泛化好评降权。
5. 只有主/次客群进入 M11.6 销量分配。

### 7.4 M11 价值战场改造

M11 不再让每个 SKU 全量打入所有战场后参与分配。

流程：

1. 从主/次任务、主/次客群、卖点价值、参数优势、产品评论体验召回战场候选。
2. 服务履约只进入服务维度，不进入产品战场。
3. 战场成立必须有产品价值锚点。
4. 弱战场只保留为候选提示，不进入竞品召回和销量分配。

### 7.5 M11.6 销量分配改造

M11.6 只对 `eligible_for_profile=true` 的维度归一化。

规则：

- `insufficient` 不分配。
- `weak` 默认不分配，只作为候选提示。
- `opportunity` 是否分配需要看是否有产品价值锚点。
- 服务履约不进入产品战场分配。
- 如果 SKU 缺少足够证据，不强制把销量分满到所有战场，可以进入 `unassigned_or_generic` 待复核桶。

## 8. 后续开发建议

优先级建议：

1. 新增评论原生语义分析报告/API，不改现有结果表，先作为诊断工具。
2. 修改 M06 的服务守门策略：服务信号不能映射到产品战场。
3. 修改 M09/M10/M11：新增 `eligible_for_profile`、`allocation_eligible_reason`。
4. 修改 M11.6：只消费可分配维度，不再把所有正分候选归一化。
5. 修改 M11.7：在销量守恒外增加业务稀疏性检查。

验收口径：

- 不是所有 SKU 都命中所有战场/客群/任务。
- 每个 SKU 有清晰的主任务、主客群、主战场，允许少量次级维度。
- 服务履约维度能单独展示，但不进入产品价值战场。
- 战场、客群、任务的销量估算纵横能对上，同时具备业务稀疏性。
- 对于证据不足的 SKU，系统应输出“证据不足/待复核”，而不是强行分配到所有预设维度。
