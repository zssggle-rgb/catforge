# 小奥家电市场分析专家能力建设工作清单

本文档用于固化本轮多轮讨论形成的分析口径、典型问题、SOP、程序结果层、CLI/Skill/Agent 设计和实施顺序，避免后续开发中遗失上下文。

## 1. 目标

把 205 上的 OpenClaw + DeepSeek 包装成“小奥家电市场分析专家”。用户用业务语言提问，不需要知道 M00/M01/Mxx，也不需要直接查数据库。智能体通过 CatForge 程序、CLI 和 Skill 读取事实数据、执行必要的生成任务，并给出可追溯、口径一致的市场分析结论。

智能体必须能回答：

- 某个 SKU 的参数、卖点、评论、市场、用户任务、目标客群、价值战场是什么。
- 某个 SKU 的竞品是谁，为什么是这些竞品。
- 为什么 A 比 B 卖得好。
- 某个 SKU 的哪些卖点支撑用户选择，哪些是溢价卖点，哪些是拖后腿卖点。
- 某个 SKU 能否通过价格、参数、卖点调整进入更多价值战场。
- 每个价值战场、用户任务、目标客群覆盖哪些 SKU，支撑多少销量/销额。
- 一个 SKU 同时落入多个价值战场、用户任务、目标客群时，销量应如何估算切分。
- 某个战场空间多大，进入后可抢多少空间，保守/中性/激进估算依据是什么。
- 新数据进来后，如何用自然语言执行数据准备和事实画像生成。

## 2. 已确认的分析口径

### 2.1 竞品判断优先级

竞品判断必须按以下优先级：

1. 同尺寸价格。
2. 同价值战场。
3. 同用户任务 / 目标客群。
4. 同参数 / 卖点。
5. 销量接近。

销量接近只能作为验证或候选扩展，不得作为竞品定义的第一依据。用户购物首先在“多大、多少钱”的货架中比较，再进入价值战场和语义判断。

### 2.2 销量判断口径

- 判断“谁卖得好”必须使用重叠在售周的周均销量和周均销额。
- 累计销量只能作为展示字段，不参与输赢和市场接受度判断。
- 新品、退市、某平台特供、单平台销售是正常商业现象，应在解释中说明。
- 平台结构可以用于解释渠道触达，但不能替代销量因果。

### 2.3 证据优先级

- 评论是用户真实需求和体验表达，优先级最高。
- 卖点是厂家意图和宣传表达。
- 参数是产品能力支撑。
- 市场量价是验证，不是单独的语义原因。
- 负向评论不是排除需求，而是“用户有需求但产品没做好”，应标记为拖后腿。

### 2.4 卖点类型

对某个 SKU 的卖点必须区分：

| 类型 | 判定口径 |
| --- | --- |
| 用户选择卖点 | 主/辅用户任务、主/辅目标客群、主/辅价值战场中被评论、参数或市场共同支撑的卖点。 |
| 销量支撑卖点 | 与主价值战场、主目标客群、主用户任务强相关，并且有评论或市场验证的卖点。 |
| 溢价卖点 | 在同尺寸价格池中，SKU 价格偏高或处于高价带，但仍能支撑销量/销额表现的卖点。 |
| 基础卖点 | 支撑购买但不支撑溢价，例如 4K、基础智能、基础音效。 |
| 厂家声明卖点 | 卖点存在，但评论或参数支撑不足。 |
| 拖后腿卖点 | 用户有需求或厂家有声明，但评论负向集中或参数明显不足。 |

溢价卖点不能只从卖点文本判断，必须同时看价格位置、主/辅战场、主任务、主客群、评论和参数支撑。

### 2.5 价值战场预设补齐

M11C `m11c_tv_value_battlefield_taxonomy_v0.2` 已补齐“次级巨幕 / 巨幕下探性价比”战场，用于避免低价/中价 98 寸以上巨幕被强行解释为旗舰巨幕：

`BF_GIANT_SCREEN_VALUE_DOWNTRADE`：巨幕下探性价比战场。

口径：

- 尺寸门槛：`giant_98_plus`。
- 价格门槛：`low` / `mid` / `mid_high`。
- 典型任务：大屏换新升级、影院沉浸观影、预算内高性价比购买。
- 典型客群：巨幕家庭影院用户、大屏换新升级用户、性价比理性用户。
- 价值锚点：98/100 寸、价格/英寸、家庭影院入门、巨幕换新、非旗舰但大屏。
- 排除：不能把低档巨幕强行解释为旗舰巨幕。

### 2.6 图谱和销量分配口径

当前新版 M09C/M10C/M11C 的实现状态：

| 维度 | 当前能力 | 缺口 |
| --- | --- | --- |
| 用户任务 | 有 SKU 关系和 coverage | 缺任务市场图谱、销量/销额切分 |
| 目标客群 | 有 SKU 关系和 coverage | 缺客群市场图谱、销量/销额切分 |
| 价值战场 | 有 SKU 关系和 graph snapshot | 缺按新版战场口径的 SKU 销量/销额切分 |

205 上旧 M11.6 `core3_sku_business_profile_sales_allocation` 已做过销量分配，但它是旧任务、旧客群、旧战场 code，且当前数据只有 84 个 SKU，不能直接用于新版 M09C/M10C/M11C。

新版应实现：

- 对同一 SKU、同一维度类型，allocation weight 合计为 1。
- 任务、客群、战场是三套不同解释视角，不能互相相加。
- 一个 SKU 可同时分配到多个任务、多个客群、多个战场。
- 分配结果是“估算销量解释”，不能表达成真实用户购买归因。

实现方式必须分两层：

1. 先写批处理程序，把新版用户任务、目标客群、价值战场图谱和销量/销额分配结果跑出来，形成稳定的结果表或快照文件，供复用。
2. 再写 CLI 和 Skill 查询这些结果，并做业务语言解释。

原因：

- 图谱和销量分配是跨 SKU 的全局计算，不适合每次自然语言问答时临时拼接计算。
- allocation weight 需要在同 SKU、同维度类型内归一，必须一次性看全量关系后再落结果。
- Skill 应该负责“调用和解释”，不应该承担长时间计算、全量扫描和归一化逻辑。
- CLI 可以提供执行入口和查询入口，但计算结果应先被程序固化，避免 OpenClaw 每次问答重复跑重任务。

## 3. 典型问题集

### 3.1 SKU 事实画像

示例：

- “这款海信 65 寸电视的参数画像是什么？”
- “这款电视的卖点事实是什么？”
- “它的评论里用户主要说什么？”
- “它的主用户任务、主客群、主战场是什么？”

SOP：

1. 解析 SKU / 型号 / 品牌。
2. 查询参数画像、卖点画像、评论画像、市场画像、用户任务、目标客群、价值战场。
3. 按事实层输出：参数、卖点、评论、量价、任务、客群、战场。
4. 明确缺失和不确定项。

### 3.2 竞品是谁

示例：

- “这款海信 65 寸电视的竞争对手是谁？”
- “这款电视和谁竞争？”

SOP：

1. 确定目标 SKU 的尺寸档、尺寸值、价格带、价格。
2. 先找同尺寸同价格带，必要时加入邻近价格带。
3. 在候选池内按价值战场、用户任务、目标客群重合排序。
4. 再看参数/卖点重合。
5. 最后用重叠周销量表现验证候选强弱。
6. 输出直接竞品、价格压力竞品、上探/下探替代竞品。

### 3.3 为什么 A 比 B 卖得好

示例：

- “这款海信 65 寸电视为什么比某个竞品卖得好？”

SOP：

1. 找到两款 SKU 的重叠在售周。
2. 计算重叠周周均销量、周均销额、周胜负。
3. 对比价格、尺寸、价格带、平台结构。
4. 对比参数优势和短板。
5. 对比卖点事实和评论支持。
6. 对比主用户任务、主目标客群、价值战场。
7. 输出：可能原因、不是原因的部分、证据不足的风险。

### 3.4 哪些卖点支撑用户选择 / 哪些是溢价卖点

示例：

- “这款海信 65 寸电视具体哪些卖点支撑用户选择？”
- “某个 SKU 哪些卖点是溢价卖点？”

SOP：

1. 查 M04C 事实卖点。
2. 查 M05C 评论支持/反驳的卖点。
3. 查 M09C 主/辅用户任务命中的 claim codes。
4. 查 M10C 主/辅目标客群命中的 claim codes。
5. 查 M11C 主/辅价值战场命中的 claim codes。
6. 查价格位置和同尺寸价格池表现。
7. 输出用户选择卖点、销量支撑卖点、溢价卖点、基础卖点、拖后腿卖点。

### 3.5 如何扩大销量

示例：

- “如果这款海信 65 寸电视要扩大销量，怎么做？”

SOP：

1. 先判断当前主/辅/机会战场。
2. 找到卡点：价格门槛、参数缺失、卖点缺失、评论负向、市场弱。
3. 分别提出价格、参数、卖点、渠道动作。
4. 每个动作必须说明会进入哪个战场、抢谁、风险是什么。

### 3.6 能否进入更多战场

示例：

- “这款海信 65 寸电视有没有机会通过价格、参数、卖点调整进入更多战场？”

SOP：

1. 查询当前所有战场 score 和 relation status。
2. 找 `opportunity_battlefield`、`user_observed_battlefield`、`drag_factor_battlefield`。
3. 对每个候选战场拆解 market gate、评论、任务/客群、卖点、参数。
4. 模拟调整：
   - 改价格：能否进入目标价格带。
   - 改参数：能否补足战场关键参数。
   - 改卖点：能否把已有能力表达出来。
5. 输出短期/中期/高成本方案。

### 3.7 战场空间和可抢空间

示例：

- “这些战场空间有多大？”
- “进入这些战场预计能抢多大市场空间？”

SOP：

1. 计算战场当前 SKU 数、周均销量、周均销额、ASP。
2. 拆尺寸档，不把不同尺寸混成一个机会。
3. 计算品牌当前份额。
4. 输出头部 SKU 和周均销量。
5. 按保守/中性/激进输出情景测算，并列出假设：
   - 保守：价格或页面卖点优化。
   - 中性：价格 + 体验修复 + 卖点强化。
   - 激进：价格 + 参数补齐 + 渠道资源。
6. 明确这不是预测模型，而是基于当前数据的可抢空间估算。

### 3.8 图谱问题

示例：

- “每个价值战场有哪些 SKU，支撑多少销量？”
- “目标客群图谱是什么？”
- “一个 SKU 落多个战场，销量怎么分？”

SOP：

1. 先查询维度图谱。
2. 输出维度包含 SKU、主/辅/机会/拖后腿分布。
3. 输出周均销量、周均销额、ASP、头部 SKU。
4. 如果有 allocation，则输出估算切分。
5. 如果没有 allocation，则必须说明“当前只能做覆盖汇总，不能做 SKU 销量切分”。

## 4. 现有能力和缺口

### 4.1 已有 CLI / Skill

`catforge_pipeline`：

- 可执行数据准备和事实画像生成。
- 已支持参数画像、卖点画像、市场画像、评论画像、用户任务、目标客群、价值战场。

`catforge_insight`：

- 可查询 SKU 参数、卖点、评论、市场、用户任务、目标客群、价值战场。
- 可查询标准参数、标准卖点、评论维度、任务预设、客群预设、战场预设。
- 可查询覆盖 SKU 和价值战场图谱。

现有 Skill：

- `catforge-pipeline`：执行型。
- `catforge-insight`：查询型。

## 6. 小奥 Agent 初始提示词草案

```text
你是小奥家电市场分析专家，面向业务用户回答彩电、空调等家电市场问题。

用户不需要知道 M00/M01/Mxx。你必须用业务语言回答，但所有结论必须来自 CatForge CLI 返回的事实。

工具选择：
- 执行数据准备、生成画像：使用 catforge-pipeline。
- 查询单点事实：使用 catforge-insight。
- 解释竞品、销量差异、卖点驱动、战场空间、增长建议：使用 catforge-analyst。

分析原则：
1. 竞品判断优先级是同尺寸价格 > 同价值战场 > 同用户任务/目标客群 > 同参数/卖点 > 销量接近。
2. 销量判断必须使用重叠在售周周均销量/销额，累计销量只展示不判断。
3. 评论是用户声音，卖点是厂家意图，参数是能力支撑，市场是验证。
4. 负向评论代表用户需求没有被满足，不是排除需求。
5. 溢价卖点必须同时看价格位置、主/辅战场、主任务、主客群、评论和参数支撑。
6. 战场空间测算是情景估算，不是财务预测。
7. 没有促销、广告、流量、库存数据时，不得声称确定因果。
8. 服务履约、物流安装、售后不进入产品价值战场，只作为服务风险说明。
9. 不输出 API key、数据库密码、环境变量。
```
